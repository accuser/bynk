# v0.48 — First-party sources as testable files behind `include_str!`

- **Status:** Finalized for sign-off — the five review decisions are settled (below).
  **Merging is approval to build.** *(Behaviour-preserving: emitted output is byte-identical —
  every emitter golden fixture, `tsc_verify`, and `runtime_helpers` stays green and unedited. No
  language surface. Independent of any feature track; its only prerequisite is the existing golden +
  `tsc` fixtures, which are the gate.)*
- **Unblocks v0.49** (`design/proposals/v0.49-security-ci-hardening.md`): relocating `runtime.ts`
  into a real file is what lets v0.49's behavioral auth-bypass tests import the verifier and lets
  CodeQL see it. The security-tooling increment is build-gated on this one.

## Context — the first-party stdlib and runtime live as Rust string literals

The `karn` conformance surface, the platform adapters, the Karn-written collection/string commons,
the per-platform TypeScript bindings, and the emitted runtime are all real Karn and TypeScript
programs. Today every one of them is a Rust string literal:

- **`karnc/src/firstparty.rs`** carries five Karn sources (`KARN_ADAPTER_SRC`,
  `CLOUDFLARE_ADAPTER_SRC`, `KARN_LIST_SRC`, `KARN_MAP_SRC`, `KARN_STRING_SRC`) and three
  TypeScript bindings (`KARN_CLOUDFLARE_BINDING`, `KARN_NODE_BINDING`, `CLOUDFLARE_BINDING`) as
  `r#"…"#` blobs — roughly 18 KB of payload around ~90 lines of actual Rust (the `Platform` enum,
  `provider_takes_env`, `platform_of`, the unit-name consts).
- **`karnc/src/emitter.rs`** carries the emitted runtime as `RUNTIME_TS` (a ~478-line `r#"…"#`
  literal at `:61`) — which **now includes the security-critical Bearer JWT verifier**
  (`verifyBearerJwtHs256`, v0.47). Security-bearing code authored as a Rust string literal is the
  sharpest case for this change: it is the one piece most needing standalone `tsc`, editor tooling,
  and (post-relocation) SAST and behavioral tests — exactly what v0.49 layers on.

Because they are literals, they sit **outside every tool that could check them**. The `.karn`
sources never reach the lexer/parser/checker, `karn-fmt`, or `karn-lsp` directly — they are
exercised only transitively, when a fixture happens to `uses` them (201/202/203/211). The `.ts`
bindings and runtime escape `tsc --strict` and the `runtime_helpers` harness the same way:
`tsc_verify` only catches a binding bug if some positive fixture emits it. No editor gives syntax
highlighting, go-to-definition, or the project's own formatter inside an `r#"…"#`.

**The cost is already visible as drift.** `karnc/runtime/runtime.ts` is a 39-line stub; the live
runtime is the ~411-line `RUNTIME_TS` const in `emitter.rs`. They have diverged hard — the const
wins precisely *because* it is bound to the compiler — which is exactly the failure mode this
change heads off.

This is a pure relocation of bytes: keep them embedded at compile time via `include_str!`, but
author them in real files with the right extension so the existing test machinery and editor
tooling can see them.

## Builds on (verified)

- **The public API is a set of `&'static str` consts + small fns.** `Platform::karn_binding_source`,
  `cloudflare_binding_source()`, and the `KARN_*_SRC` consts all hand out `&'static str`.
  `include_str!` yields exactly that, so **no signature changes** and **no call-site changes** —
  `project.rs` and `karn-lsp/src/completion.rs` are untouched.
- **The Karn sources already round-trip through the real pipeline.** `project.rs` does
  `lexer::tokenize(firstparty::KARN_LIST_SRC)` → `parser::parse_unit(…)` today (`:502`–`:540`), so a
  standalone test asserting each source tokenises, parses, and checks is the same two calls over the
  same `&str` — no new harness.
- **A `tsc --strict` backstop already exists.** `tsc_verify.rs` assembles a temp dir and runs
  `tsc --strict --noEmit`; `runtime_helpers.rs` imports the emitted runtime and exercises it. Both
  already "skip loudly" without a toolchain (`KARN_REQUIRE_TSC=1` makes the skip a hard failure).
  Pointing them at the binding/runtime files standalone is an extension, not new machinery.
- **The delivery model is vendoring, not packages.** karnc depends on no `@karn/*` package; it emits
  `runtime.ts`, the surface binding, and the platform binding into the output tree, and the generated
  `package.json` carries only the user's third-party deps. Nothing in this change touches that — it
  changes where the source is *authored*, not how it is *delivered*.

## Settled

- **▸ Byte-identical embedding.** `include_str!` embeds the same bytes at compile time. Still
  `&'static str`, still no runtime I/O, still one self-contained binary. Any fixture diff is a defect,
  not an update.
- **▸ Real files with real extensions.** Each blob moves to a `.karn` / `.ts` file so the compiler's
  own pipeline, `karn-fmt`, `karn-lsp`, `tsc`, and editors can all see it.
- **▸ One source of truth for the runtime.** `RUNTIME_TS` and the stale `karnc/runtime/runtime.ts`
  collapse into a single file behind `include_str!`; the divergence is removed by construction.

## Decisions (settled at sign-off review)

**[SETTLED] Embedding mechanism — `include_str!`.** The idiomatic, zero-dependency choice:
compile-time embedding, no `build.rs`, no generated artefacts in the tree, identical `&'static str`
contract (so no call-site changes). (Rejected: a `build.rs` that generates the consts — strictly more
machinery for no added guarantee, since `include_str!` already fails the build on a missing path.)

**[SETTLED] File layout — a package-shaped `firstparty/` tree**, so the in-repo layout maps cleanly
onto a future `@karn/runtime` / `@karn/cloudflare` / `@karn/node` even though we are not publishing
now:

```
karnc/src/firstparty/
  karn.karn                       (KARN_ADAPTER_SRC — the karn surface adapter)
  karn.cloudflare.karn            (CLOUDFLARE_ADAPTER_SRC — the platform adapter)
  karn.list.karn  karn.map.karn  karn.string.karn   (the commons)
  bindings/
    karn-cloudflare.ts  karn-node.ts                (surface bindings, per platform)
    cloudflare.binding.ts                           (platform-adapter binding)
    runtime.ts                                      (the emitted runtime — was RUNTIME_TS)
```

`firstparty.rs` keeps the Rust (the `Platform` enum and the `provider_takes_env` / `platform_of`
metadata) and just swaps each payload for `include_str!("firstparty/…")`. (Rejected: a flat
`firstparty/` with no `bindings/` split — fine, but the package-shaped layout makes the later publish
decision a lift-and-shift rather than a redesign, at no cost today.)

**[SETTLED] Scope: fold in `RUNTIME_TS`.** Move `RUNTIME_TS` (`emitter.rs:61`) to
`firstparty/bindings/runtime.ts` in the same increment and **delete the stale
`karnc/runtime/runtime.ts`**. Verified safe to delete: no Rust/build/docs path references it (a repo
grep finds only `.claude/settings.local.json` allowlist entries, which are inert). The drift between
the two — now that the live runtime carries the Bearer verifier — is the strongest concrete
motivation for the change; fixing the consts while leaving the runtime literal in place would be a
half-measure. (Rejected: defer the runtime to its own PR — cleaner blast radius, but leaves the
most-cited example unaddressed and keeps the security-critical verifier a literal.)

**[SETTLED] New test scope — land a focused, self-contained set** (this is the scaffold v0.49's
auth-bypass tests extend):

  1. **Each `.karn` source tokenises, parses, and type-checks standalone** — a table test reusing the
     `lexer::tokenize` + `parser::parse_unit` path `project.rs` already uses, so the stdlib can't ship
     un-parseable.
  2. **`karn-fmt --check` over the `.karn` sources** in CI, so the first-party stdlib obeys the
     project's own formatting rules.
  3. **`tsc --strict --noEmit` over each `.ts` binding + `runtime.ts` standalone**, via a small
     scaffold that lays the binding beside `runtime.ts` and an emitted `karn` surface `.ts`
     (reusing `tsc_verify`'s discover-or-skip-loudly logic).
All three land with the move. (Rejected: relying solely on the existing fixtures — 201/202/203/211 for
the commons, 177 for the cloudflare binding — that emit and `tsc` these *transitively*: cheaper, but
it keeps the "only checked when something happens to use it" gap this increment exists to close.)

**[SETTLED] npm packaging — vendor now, publish later.** Whether
`karn-node` / `karn-cloudflare` (and the runtime) should be **published npm packages** rather than
vendored into each emitted project is a real and separate decision about *delivery*, and this refactor
deliberately does **not** settle it: **keep vendoring; treat publishing as a future ADR, not part of
v0.48.** Rationale: the bindings and runtime are currently part of the compiler's **emit ABI**
— they are coupled to emit shapes (the `tag` layout of `Result`/`Option`, `JsonError`, `Uuid.of`,
`FetchError`). Vendoring makes skew *impossible by construction*: the compiler that emits your code
emits the runtime it talks to. Splitting them into independently-versioned packages introduces a
version-skew axis the compiler would then have to police (pinning an exact `@karn/runtime` into every
`package.json`), inverts the import graph (emitted code importing *from* a package rather than the
package importing project-local siblings), adds a registry/network dependency to builds that are
currently hermetic, and stands up a publish pipeline — for a pre-1.0 language. Publishing becomes
attractive once the runtime interface is stable enough to version on its own (≈1.0). The
package-shaped layout above is the cheap hedge: it keeps that door open without paying for it now.
*This proposal's only commitment is to author the sources package-ready, not to publish them.*

## Spec delta

**None.** No language surface, no grammar, no emission contract — an internal relocation of
already-shipped source from Rust literals into files. The `// Generated by karnc — do not edit by
hand.` header semantics are unchanged (the bindings are still emitted verbatim into output projects).

## Tests

- **No new golden fixtures, and no edits to existing ones.** The full emitter golden suite,
  `tsc_verify`, and `runtime_helpers` are the acceptance gate for byte-identical output; all stay
  green unedited.
- **New first-party self-tests** per the test-scope decision: standalone parse/check of each `.karn`
  source, `karn-fmt --check` over them, and standalone `tsc --strict` over each `.ts` binding +
  `runtime.ts` (discover-or-skip-loudly, honouring `KARN_REQUIRE_TSC`).

## Diagnostics / Risks

- **Trailing-byte / newline skew.** `r#"…"#` literals can carry a leading or trailing newline that a
  file does not (or vice versa). The bindings begin with a `//` comment line and end with a newline;
  any one-byte difference would surface as a golden diff. *Mitigation:* the golden + `tsc` fixtures
  catch it immediately; diff each moved blob against its file at conversion time.
- **`include_str!` path resolution.** Paths are relative to the **including file**
  (`karnc/src/firstparty.rs` and `karnc/src/emitter.rs`), not the crate root. *Mitigation:* a missing
  or mistyped path is a hard compile error, so this cannot ship silently; verify with a clean
  `cargo build`.
- **Stale-runtime cleanup.** Anything that reads `karnc/runtime/runtime.ts` by path (packaging
  scripts, docs `{{#include}}`, examples) must be repointed before it is deleted. *Mitigation:* grep
  for the path across the repo (scripts, `docs/src`, `Cargo.toml` `include`s) before removal.
- **`.karn` files picked up as project inputs.** Real `.karn` files under `karnc/src/` must not be
  mistaken for compilable project sources by any glob (tests, fixtures discovery, `karn-fmt` CI
  targets). *Mitigation:* confirm the discovery globs are scoped to `tests/fixtures` / project roots,
  and add the new dir deliberately where (and only where) the new self-tests want it.

## New decision records this increment adds

- **One small ADR** recording the first-party-source-of-truth posture: *sources authored as files,
  embedded via `include_str!`, vendored (not published) while they remain part of the emit ABI;
  publishing deferred to a future increment.* This is the durable home for the packaging decision so
  the proposal can be deleted on merge without losing the reasoning. (Sits alongside the platform
  decisions 0017/0026.)

## Out of scope (→ later / independent)

- **Publishing `@karn/runtime` and the platform bindings to npm.** A separate delivery-model change,
  gated on runtime-ABI stability — its own future ADR (see the packaging decision above).
- **Refactoring `firstparty.rs`'s Rust logic** (`provider_takes_env`, `platform_of`, the env-takes
  metadata). Unchanged here beyond losing the payload consts.
- **Non-first-party string literals** elsewhere in the emitter (per-construct TS fragments). This
  increment is the first-party stdlib/runtime/bindings only.

## Documentation / book delta

**No book impact**, because this is an internal refactor with no user-facing surface: nothing a Karn
author writes, compiles, or reads in `docs/src/` changes. The one contributor-facing note worth
adding is a short pointer (architecture/CONTRIBUTING) recording that first-party Karn/TS sources live
as files under `karnc/src/firstparty/` and are embedded via `include_str!` — so the next person edits
the file, not a literal. No currency-banner or changelog-feature entry beyond an internal-refactor
changelog line.

## Definition of done

The five Karn sources, three TypeScript bindings, and the emitted runtime move from Rust `r#"…"#`
literals into real `.karn` / `.ts` files under a package-shaped `karnc/src/firstparty/` tree, each
embedded via `include_str!`; `firstparty.rs` keeps only its Rust logic; `RUNTIME_TS` is folded in and
the stale `karnc/runtime/runtime.ts` removed; the new first-party self-tests (standalone parse/check,
`karn-fmt --check`, standalone `tsc --strict`) pass; every emitter golden fixture, `tsc_verify`, and
`runtime_helpers` passes **unedited**; the packaging-posture ADR lands; `cargo build`/`test`/clippy and
CI stay green; the changelog notes the internal refactor. The implementing PR **deletes this file**.
