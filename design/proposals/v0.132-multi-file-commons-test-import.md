# v0.132 — `bynkc test` resolves a multi-file commons: emit an aggregating barrel

- **Scope:** a single-increment **emitter** fix — no grammar, no checker, no
  language surface. The test emitter (`bynk-emit/src/project/tests_emit.rs`) stops
  assuming every commons emits to a single `out/<name>.ts` file; the **test path**
  writes an aggregating **barrel** (`out/<name>.ts` → `export *` from each file of the
  commons) for each multi-file commons a test or integration module imports. The
  existing single namespace import then resolves for both layouts, and production output
  is untouched.
- **Addresses:** [#451](https://github.com/…/issues/451) — a `commons` split across a
  folder (`src/thing/*.bynk`, each `commons thing`) is a **supported** source layout
  (`unit_path_matches`, `paths.rs:199`), passes `bynkc check`, and emits correctly in
  production — but `bynkc test` fails at `tsc` with `TS2307 Cannot find module
  './../thing.js'`, because the generated test module imports one aggregated file that
  a multi-file commons never produces. Discovered building the `ilr` test project
  against v0.118.0.
- **Realises:** parity between `bynkc test` and the two paths that already accept the
  layout (`bynkc check`, production emit) — the test suite compiles and runs
  regardless of whether the target commons is one file or a directory of files.

## What exists today (grounded)

A commons may live as **one file** (`src/thing.bynk`) or as a **directory**
(`src/thing/a.bynk`, `src/thing/b.bynk`, each declaring `commons thing`). Both are
accepted:

- `unit_path_matches` (`bynk-emit/src/project/paths.rs:214`, `:219`) returns true for a
  `single_file_match` (stem parts == name parts) **or** a `multi_file_match` (the
  file's *parent-directory* parts == name parts). The multi-file layout is deliberate,
  not incidental.
- Production honours it. The context emitter groups cross-unit imports **by target
  file path** (`bynk-emit/src/emitter.rs:1567`), resolving each imported symbol to the
  file that defines it via `ctx.imported_decl_paths` (built in `project.rs:2122` from
  each unit's `FileDeclIndex { types, fns, … }`, `symbols.rs:526`). So `out/ilr.ts`
  emits `import { ULN as __CommonsULN } from "./learner/unique-learner-number.js"`,
  one import per defining file — never a whole-module import of `out/learner.ts`.

The **test** emitter does not. `emit_test_module`
(`bynk-emit/src/project/tests_emit.rs:2659`) derives the target import from the module
*name* alone and emits a single whole-module **namespace** import:

```rust
// tests_emit.rs:2681–2707
let target_ns    = target_name.replace('.', "_");          // "learner"
let target_dir   = commons_dir_for(target_name);           // path "learner"
let import_target = relative_import_for_test(&target_dir);  // "../learner"
out.push_str(&format!("import * as {target_ns} from \"./{import_target}.{ext}\";\n"));
// ⇒ import * as learner from "./../learner.js"
```

`commons_dir_for("learner")` returns the directory path `learner` (`paths.rs:161`). For
a **single-file** commons that coincides with the emitted `out/learner.ts` and the
import resolves; for a **multi-file** commons the symbols live in `out/learner/*.ts`,
no `out/learner.ts` is produced, and `tsc` reports `TS2307`.

This is **not** one dangling import — it is a *convention*. The same
`commons_dir_for` → `relative_import_for_test` → `import * as ns from "./<name>.js"`
shape recurs at **four** sites, and every one breaks for a multi-file commons:

| site | `tests_emit.rs` | what it imports |
|---|---|---|
| target commons | `:2705`–`2707` | the unit under test |
| consumed contexts | `:2715`–`2722` | each `consumes` target |
| `uses` commons | `:2731`–`2738` | each `uses` target (e.g. `Money`) |
| integration harness | `:735`–`741` (`emit_integration_module`) | each participant commons |

The test path relies on **namespace-qualified references** everywhere: lowered case
bodies name symbols as `learner.Foo`, `Money.of`, etc. That single fact is what picks
the fix.

## The fix (and why the barrel, not per-symbol imports)

The issue offers two options: **(1)** per-symbol imports (mirror production’s
`out/ilr.ts`) or **(2)** an aggregating barrel. They are not equivalent for the test
path, because the test path is namespace-based and production is not:

- Production imports **individual, rebranded** symbols (`X as __CommonsX`,
  `emitter.rs:1597`) precisely so it can re-brand each type — it *wants* per-symbol
  granularity and never uses `import * as`.
- The test module references those symbols through a **namespace object**
  (`learner.Foo`). Per-symbol imports would force either synthesising that namespace
  object by hand (`const learner = { Foo, Bar }` after N individual imports — reversing
  the very grouping production does) or rewriting every `ns.Symbol` reference in every
  lowered case body. Both are large, and both must be repeated at all four sites above.

The **barrel** preserves the one-line namespace import unchanged and fixes all four
sites at once: emit `out/<name>.ts` that re-exports the commons’ directory, and
`import * as learner from "./../learner.js"` resolves exactly as it does for a
single-file commons.

### Decisions

**[DECISION A] Fix via an aggregating barrel (Option 2), not per-symbol imports
(Option 1). (Recommended, load-bearing.)** The test emitter’s contract is
*namespace imports + namespace-qualified references*; the barrel keeps that contract
and repairs every import site uniformly. Per-symbol imports fit *production* (which
rebrands and never uses `import * as`) but fight the test path, and would have to be
threaded through the target, consumed, `uses`, and integration sites separately. The
issue’s note that Option 1 “keeps the emit surface consistent with production”
misreads the test path — consistency there means *namespace imports*, which only the
barrel preserves. *Recommend: the barrel.*

**[DECISION B] Emit the barrel from the test path only — not the shared project emit.
(Recommended.)** All four broken sites live in the test path: the suite emitter
(`process_tests`, `tests_emit.rs:71`) and the integration emitter
(`process_integration_tests` → `emit_integration_module`), whose outputs flow into
`compiled` via `compiled.extend(test_outputs)` (`project.rs:2897`) and
`compiled.extend(integration_outputs)` (`:2998`). Because integration emit is itself a
test-path site, a barrel emitted from the test path fixes it identically to a global one
— so the "fixes integration for free" advantage a global barrel might claim **does not
differentiate** the two options. What remains for *global* is only (a) a speculative
future non-test consumer of the `import * as ns` contract — none exists today — and (b)
`out/` aesthetic self-consistency. Neither is load-bearing, and both are bought at a
**permanent** cost: global ships an inert `out/<name>.ts` into *every* build for *every*
multi-file commons — including commons with no tests and no test consumer — a new
importable production entrypoint that nothing in production reads.

The subtler reason is altitude. Global splits the requirement from its guarantee: the
code writing `import * as ns` (`tests_emit.rs`) and the code guaranteeing `ns` resolves
(a barrel in the `project.rs` emit loop) sit ~900 lines apart, so someone later
tightening the barrel’s trigger has no local signal a test import depends on it.
Test-only co-locates them — the emitter that emits the namespace import also emits the
barrel it needs.

*Honest counterweight:* global gets dedup for free — the per-unit emit loop visits each
commons exactly once, so it cannot emit a duplicate `CompiledFile` at the same path.
Test-only must guard against it, because several test/integration modules can
`import * as learner` the same commons: a `HashSet<PathBuf>` of already-emitted barrel
paths, ~5 lines. That is the entire "needs its own emission hook" cost. The trade is
~5 lines of dedup (test-only) against a permanent production-surface addition + a latent
drift risk (global); pay the 5 lines. *Recommend: test-only.* Named follow-on — if a
non-test consumer of the namespace contract ever materialises, promoting the barrel from
the test path to the shared emit loop is a mechanical one-time move; defer it until real.

**[DECISION C] The barrel is `export *` from each distinct file of the commons — no
symbol enumeration.** A commons is one merged namespace, so **every** symbol it
re-exports is already unique across its files — types, functions, and methods each have
their own resolve diagnostic (`bynk.resolve.duplicate_type`, `symbols.rs:256`;
`bynk.resolve.duplicate_fn`, `:453`; `bynk.resolve.duplicate_method`, `:504`), so a flat
`export *` merge cannot collide regardless of symbol kind. The file set is the distinct
`PathBuf`s across the commons’
`FileDeclIndex { types, fns, methods }` (`symbols.rs:526`) — the same index production
already builds — so no new discovery machinery is needed. For a commons `learner`
spanning `unique-learner-number.bynk` and `learner-reference-number.bynk`:

```ts
// out/learner.ts — Generated by bynkc — do not edit by hand.
export * from "./learner/unique-learner-number.js";
export * from "./learner/learner-reference-number.js";
```

**[DECISION D] Emit a commons’ barrel once, when a test or integration module imports a
commons whose layout is multi-file.** The condition is unchanged from a global barrel —
the `multi_file_match` branch (the same parent-directory test `unit_path_matches` already
computes, `paths.rs:219`), where no `out/<name>.ts` is otherwise produced; a single-file
commons already owns `out/<name>.ts` and must not get a barrel that would collide. Only
the *host* differs (DECISION B): the barrel is written from the test path, so the trigger
fires per imported commons and is deduped through a `HashSet<PathBuf>` of already-emitted
barrel paths — several test/integration modules may import the same commons. Dotted
commons names compose: `commons a.b` split across `src/a/b/*.bynk` emits the barrel at
`out/a/b.ts` (`commons_dir_for("a.b")` → `a/b`), exactly the slot the single-file form
would occupy.

## The deltas (concretely)

**Emitter (`bynk-emit`) — the whole change, in `tests_emit.rs`.**
- **New barrel emission** from the test path (`process_tests`, `tests_emit.rs:71`, and
  the sibling `process_integration_tests`). For each commons a test or integration module
  imports whose layout is multi-file, push a `CompiledFile` at
  `out/<commons_dir_for(name)>.ts` whose body is `export * from "./<name>/<file>.js"` for
  each distinct file in the unit’s `FileDeclIndex`, sorted for determinism. These files
  join `compiled` alongside the existing `test_outputs` / `integration_outputs`
  (`project.rs:2897`/`:2998`). Reuse `commons_path` /
  `cross_commons_import_specifier_for_path` so the relative specifiers match the rest of
  the emitter.
- **Dedup guard.** A `HashSet<PathBuf>` of already-emitted barrel paths, since several
  test/integration modules can import the same commons; the per-unit project emit loop
  gets this for free but the test path does not (DECISION B’s one real cost).
- **No change to `emit_test_module` / `emit_integration_module` themselves** beyond what
  the barrel enables — the existing `import * as ns from "./<name>.<ext>"` lines now
  resolve for both layouts. **Production emit (`project.rs`) is untouched** — no barrel
  ships in a non-test build.
- **`unit_path_matches`** gains no behaviour but its two branches
  (`single_file_match` / `multi_file_match`, `paths.rs:214`/`:219`) become the single
  source of truth the barrel trigger reads; extract the “is this commons multi-file?”
  predicate so emit and validation cannot disagree.

**Checker (`bynk-check`), grammar/AST (`bynk-syntax`).** Untouched — the layout already
type-checks and parses; this is an emit-only defect.

**Fixtures.** A `bynkc test` fixture with a **multi-file commons** (`src/thing/a.bynk`,
`src/thing/b.bynk`) plus a suite (`tests/thing.bynk`) that asserts: (a) the test build
emits `out/thing.ts` and it `export *`s both files; (b) the generated
`out/tests/thing.test.ts`’s namespace import resolves; (c) `bynkc test` compiles under
the emitted `out/tsconfig.json` and the suite runs. A twin single-file fixture asserts
**no** `out/thing.ts` barrel-vs-file collision (the single file *is* `out/thing.ts`).
Two more assertions pin DECISION B: a **production** (non-test) emit of the same
multi-file commons emits **no** `out/thing.ts` barrel; and a project with two suites over
the same multi-file commons emits the barrel **once** (dedup). An integration fixture
covers the `emit_integration_module` site. The reproduction in #451 is the acceptance
fixture.

## Risks & mitigations

- **Barrel/file collision for a single-file commons** → guarded by DECISION D: the
  barrel is emitted *only* on the `multi_file_match` branch, where `out/<name>.ts`
  does not otherwise exist.
- **Ambiguous re-export across the commons’ files** → impossible: intra-commons symbol
  names of every kind are already unique (`bynk.resolve.duplicate_type`/`_fn`/`_method`),
  so `export *` merges without conflict (DECISION C).
- **Duplicate barrel emission** → several test/integration modules can import the same
  commons, so the barrel is deduped through a `HashSet<PathBuf>` (DECISION B/D); the
  fixture asserts single emission.
- **Production output surface** → unchanged (DECISION B): the barrel is emitted only from
  the test path, so a non-test build ships no `out/<name>.ts` barrel. A fixture pins
  this. (This is the cost a *global* barrel would have carried and the reason it was
  rejected.)
- **Windows path separators in the generated specifiers** → route every specifier
  through the emitter’s existing `ts_specifier` (forward-slash normalisation,
  `emitter.rs`), not `Path::display()`.
- **Dotted / nested commons names** → covered by DECISION D and a fixture; the barrel
  lands at the `commons_dir_for` slot, matching the single-file emit path.

delta is deliberately small — stated here rather than left silent (per the proposals
README).

- **Changelog:** a `reference/changelog.md` entry under v0.132 — “`bynkc test` now

  mirrors it. Advance the
  currency banner and `spec/appendix-version-history.md` to v0.132.
- **Guide (light):** wherever the book documents the **single-file vs directory
  commons layout**, add one sentence that the directory layout is fully supported by
  `check`, `test`, and build alike — the behaviour this fix restores. If no such
  passage exists, a short reference note under the project-layout page; no new concept,
  so no new “Understand” on-ramp.
- **Spec:** no normative change — the emitted barrel is an *implementation* artefact of
  the output tree, not language semantics. If the emit chapter documents the `out/`
  file layout, note the barrel there; otherwise untouched, and this line says why.
- **Tooling delta (ADR 0156 — silence is an oversight):** **hover**, **completion**,
  **semantic tokens**, and **signature help** are all **unchanged** — this increment
  adds no construct and changes no surface an editor sees; it only alters the shape of
  emitted `out/*.ts`. Stated explicitly, per the rule.
