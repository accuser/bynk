# v0.51 — Actors, slice 3 (Signature): HMAC request-signature verification for webhooks, with a replay window

- **Status:** Draft (proposal). **Merging is approval to build.** *(Feature-track slice — the third of
  the actors track, `design/tracks/actors.md`. Adds the second authenticated scheme: `Signature`, for
  verifying inbound webhooks by an HMAC over the request body.)*
- **Sequences after v0.47 (BearerToken) and v0.50 (optional `by` binder).** It parallels the Bearer
  seam (secret sourcing, runtime verifier, fail-closed 401) and leans on the optional binder — a
  webhook handler is naturally `by Webhook ()` (verify authenticity, capture nothing). Provisional
  version (`v0.51` assumes nothing jumps the queue — renumber if sequencing shifts).
- **Verification-bearing slice** → carries the `/security-review` + `/code-review` gate (ADR 0076) and
  a threat model below.

## Context — webhooks are authenticated by a signature over the body, not a token

Bearer (v0.47) handles a caller presenting a JWT in `Authorization`. The other common external
boundary is the **inbound webhook**: a provider (Stripe, GitHub, a sibling system) POSTs an event and
proves authenticity by signing the **raw request body** with a shared secret — an HMAC the receiver
recomputes and compares. There is usually **no per-user identity** — the signature attests *this came
from the trusted sender*; the event itself is the (normal, typed) request body. Webhooks also
**retry**, so a verified request can be *replayed*; the standard guard is a signed **timestamp** within
a tolerance window.

This slice makes the compiler emit that verification — at the boundary, before the body handler runs,
fail-closed — so an app never hand-rolls HMAC comparison (the place webhook integrations most often go
wrong: timing-unsafe compares, verifying a re-serialized body, or skipping the timestamp).

## Builds on (verified)

- **The v0.47 Bearer seam** — `Signature` is a new entry against the same sealed scheme descriptor:
  secret sourced from the env the `Secrets` capability reads, a runtime verifier
  (`verifySignatureHmacSha256`, parallel to `verifyBearerJwtHs256`, using WebCrypto constant-time
  `crypto.subtle.verify`), failure → `HttpResult.Unauthorized`. `Scheme::Signature` is already a
  reserved enum variant; this flips it to admitted.
- **The v0.50 optional binder** — a `Signature` actor yields no captured identity, so `by Webhook ()`
  (binder-less) is the idiom. This slice needs the optional binder to read cleanly; good synergy.
- **The HTTP dispatch body read** (`workers_entry.rs`, `await request.json()`) — Signature must read
  the **raw** body once (`request.text()`), HMAC it, then parse the body param from the same bytes (a
  request body reads once). This slice changes that read for signature-bearing handlers (see the
  ordering decision).
- **`HttpResult.Unauthorized`** (401) and the `karn.actor.bearer_*` diagnostic precedent for the new
  `karn.actor.signature_*` family.

## Settled (from the track / foundational ADRs)

- **▸ `Signature` is a closed-set scheme** (ADR 0080) — new surface against the descriptor, not a new
  kind. **▸ Two-phase, fail-closed** (ADR 0082) — verify before the body. **▸ Identity is sealed**
  (ADR 0081) — though for Signature the typical identity is `()` (authenticity, not a principal).
  **▸ HTTP-only** — a request signature is an HTTP-body concept; admissible only on `from http`.

## Decisions (open — settle at review)

**[DECISION] Verification surface: canonical HMAC + configurable header now; provider presets later.**
Real providers differ (GitHub `X-Hub-Signature-256: sha256=<hex>`; Stripe `Stripe-Signature:
t=…,v1=<hex>` over `timestamp.body`). *Rec:* ship a **configurable HMAC-SHA256 verifier** —
`auth = Signature(secret = "<ENV>", header = "<Header-Name>")` — that recomputes HMAC-SHA256 over the
raw body and compares (constant-time) against the header value (accepting a bare hex digest or a
`sha256=<hex>` prefix, the GitHub shape). Defer **provider presets** (`Signature(provider = "stripe")`
baking in the compound `t=,v1=` format + the `timestamp.body` signing string) to a thin follow-up
layer over this verifier. *Alternatives:* (a) a Karn-canonical fixed header only (`X-Karn-Signature`)
— simplest, but only works Karn-to-Karn, useless for real providers; (b) full provider presets now —
most useful, but the per-provider format variance is a lot of surface for one slice. The configurable
verifier covers the common (GitHub-style) shape and is the substrate presets sit on.

**[DECISION] Identity: `Signature` actors carry no identity by default.** *Rec:* a `Signature` actor's
identity is `()` — the signature attests authenticity; the event is the body param. So **no
`identity =` is required** (unlike Bearer), and `by Webhook ()` (binder-less, v0.50) is the canonical
form. *Alternative:* allow an optional `identity = T` minted from a header/claim — deferred; the
declaration shape doesn't foreclose it.

**[DECISION] Replay posture: a timestamp-tolerance window; dedup deferred.** *Rec:* when a timestamp
header is configured (`Signature(secret=…, header=…, timestamp = "<Header>", tolerance = <seconds>)`),
the seam verifies the timestamp is within `tolerance` of now (reject stale → fail-closed) **and** binds
the timestamp into the signed string where the provider requires it. Full **replay dedup** (by event
id) is the `Idempotency` capability's job (Q8) — **out of scope**, but the declaration shape leaves
room. *Alternative:* no replay guard at all — rejected: a signature scheme that ignores replay is a
known webhook footgun.

**[DECISION] Raw-body ordering.** *Rec:* the seam reads the body **once** as text (`await
request.text()`), verifies the HMAC over those exact bytes, then the existing body-param
deserialisation parses from the **same** text (not a re-read, not a re-serialisation — re-serialising a
parsed body would not match the signed bytes). The emitter coordinates this for signature-bearing
handlers. *Alternative:* verify over a re-serialised body — rejected: byte-fragile, breaks on
whitespace/key-order.

**[DECISION] Failure response: 401, fail-closed.** *Rec:* a bad/absent signature or stale timestamp →
`HttpResult.Unauthorized` (401), consistent with Bearer; the body never runs. *Alternative:* 400 — some
webhook docs use it, but 401 is the authenticity-failure channel and matches the slice's model. (The
403 authorisation-invariant split is the invariants slice, Q3.)

**[DECISION] Require a body.** *Rec:* a `Signature` handler MUST take a `body` param (the signature is
over the body) — a bodyless signed request is meaningless (`karn.actor.signature_requires_body`).

## Spec delta

- **`lexical-grammar.md` / `syntactic-grammar.md`.** `Signature` becomes admitted; the scheme config
  grows `Signature(secret = "…", header = "…", (timestamp = "…", tolerance = <int>)?)` (the `scheme`
  production gains the keyed args — generalises v0.47's `Bearer(secret=…)`).
- **`static-semantics.md`.** Signature admissibility is **HTTP-only** and **requires a body param**;
  the `secret` and `header` configs are required (`karn.actor.signature_missing_*`); a `Signature`
  actor takes no `identity` (or it is `()`); `tolerance` requires `timestamp`.
- **`emission.md`.** The Signature scheme-descriptor entry: read the raw body once, HMAC-SHA256 verify
  (constant-time) against the configured header, optional timestamp-tolerance check, then hand the same
  bytes to body deserialisation; fail-closed → 401 before the handler.
- **`runtime-library.md`.** `verifySignatureHmacSha256` (+ a hex/`sha256=` parser, a constant-time
  compare via `crypto.subtle.verify`, an optional timestamp check). Sourced via `Secrets`.
- **`diagnostics.md`.** `karn.actor.signature_*` family (`signature_missing_secret`,
  `signature_missing_header`, `signature_requires_body`, `signature_tolerance_without_timestamp`).

## Tests

- **Behavioral (the standing guard, parallel to `bearer_auth.rs`):** a Node driver signs bodies with
  WebCrypto and asserts the emitted verifier's verdict for every bypass class — valid signature
  accepted; tampered body, wrong secret, absent/malformed signature header, stale timestamp (when
  configured), and (where applicable) a `sha256=` vs bare-hex mismatch → all rejected.
- **Positive golden:** a `from http` webhook `on POST("/hooks/x") by Webhook (body: Event)` — the seam
  reads-raw-once, verifies, then deserialises; `tsc --strict` green.
- **Negative:** `Signature` on a non-HTTP handler; a `Signature` handler with no body; missing
  `secret`/`header`; `tolerance` without `timestamp`.

## Diagnostics / Risks

- **Constant-time compare.** The HMAC compare MUST be constant-time (`crypto.subtle.verify`), never a
  string `===`. *Mitigation:* the compiler emits the verifier; golden + the behavioral test pin it.
- **Raw-body single-read.** Reading the body twice (text then json) would throw. *Mitigation:* the
  emitter reads once and parses from the held text; golden + `tsc` cover it.
- **Provider-format variance.** The configurable verifier covers GitHub-style; Stripe's compound
  format needs the deferred preset. *Mitigation:* documented scope; presets are additive.

## Security / threat model

*Gates the slice (ADR 0076).*

- **Guarantees.** The body handler runs only on a request whose body HMAC verifies against the sourced
  secret (constant-time) and, when configured, whose timestamp is within tolerance — fail-closed → 401.
  Verification is over the exact received bytes, not a re-serialisation.
- **Secret material** sourced via `Secrets` (ADR 0021), only in the seam, never logged.
- **Explicitly out:** replay **dedup** (by event id — the `Idempotency` capability, Q8), key
  rotation, asymmetric/Ed25519 webhook signatures, and Stripe's compound-format preset.

## New decision records this increment adds

- **ADR — `Signature` is compiler-generated HMAC-SHA256 over the raw body; configurable header; a
  timestamp-tolerance replay window; HTTP-only, body-required, identity `()`.** Parallels ADR 0085
  (Bearer); presets and dedup are later.

## Out of scope (→ later slices / the track)

- **Provider presets** (`Signature(provider = "stripe"|"github")`), **replay dedup** (Idempotency, Q8),
  **asymmetric signatures**, **multi-actor sums** (Q4), **authorisation invariants + 403** (Q3).

## Definition of done

`by Webhook (body: T)` HTTP handler parse, resolve, and check (HTTP-only, body required, configs

timestamp window, 401 on failure, then deserialise the same bytes — with golden emission and
