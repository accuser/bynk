# v0.130 ‚Äî Literal patterns in `match` (Int / String / Bool)

- **Scope:** a single-increment **language** proposal (no track). Grammar +
  parser + checker + emitter + formatter + tree-sitter, plus the editor-surface
  delta (hover / completion / semantic tokens / signature help ‚Äî ADR 0156).
  Version ‚Üí **0.130.0** via `scripts/bump-version.sh`.
- **Closes:** #441 ‚Äî `match` rejects literal patterns, contradicting
  `design/bynk-type-system.md` ¬ß2.3.4 (which lists `p ::= ‚Ä¶ | c` as a pattern
  form). Filed against 0.114.0; still reproduces on 0.129.0.

## The gap, precisely

¬ß2.3.4 lists a literal pattern (`c`) alongside variable, wildcard, and tag
patterns. The implementation has never had it, and the gap is deeper than the
issue's suggested fix implies ‚Äî it is not "add a token branch to `parse_pattern`":

1. **AST.** `Pattern` (`bynk-syntax/src/ast.rs:1906`) has exactly two arms ‚Äî
   `Wildcard` and `Variant`. There is no node a literal pattern can become.
2. **Parser.** `parse_pattern` (`bynk-syntax/src/parser/expressions.rs:993`)
   handles `_`, the built-in `Ok`/`Err`/`Some`/`None`, then falls through to
   `expect_ident("as a match-arm pattern")`. An `IntLit`/`StrLit`/`True`/`False`
   token hits that ident expectation and produces the reported error.
3. **Checker.** `check_match` (`bynk-check/src/checker/expressions.rs:2160`)
   gates the *whole construct* on `variants_of` (`checker.rs:2732`): a
   discriminant that is not a sum / `Result` / `Option` is rejected outright with
   `bynk.types.match_non_sum_discriminant`. **So the issue's assumption that
   literal exhaustiveness works "same as today's non-sum matches" is wrong ‚Äî there
   are no non-sum matches today.** Matching on a primitive is itself new surface.
4. **Emitter.** Match lowering (`bynk-emit/src/emitter/lower.rs:3249` +
   `emit_match_case:3292`) emits `switch (disc.tag) { case "Tag": ‚Ä¶ }`. A raw
   `Int`/`String`/`Bool` value has no `.tag`; literal patterns need a different
   lowering.

Four crates move, not one. That is the honest scope.

## What lands

- **`Pattern::Literal { value, span }`** ‚Äî a third AST arm carrying a restricted
  literal (see DECISION A). `Pattern::span()`, the formatter, and every
  exhaustive `match` on `Pattern` across the workspace get the new arm.
- **`parse_pattern`**: before the `expect_ident` fall-through, branch on
  `IntLit` / `StrLit` / `True` / `False` (and `Minus` + `IntLit` ‚Äî negative int,
  per ADR 0001) into a `Pattern::Literal`.
- **`check_match`**: admit a primitive (`Int` / `String` / `Bool`, and refined
  types over those bases ‚Äî DECISION D) discriminant; type-check each literal arm
  against the scrutinee's base type; exhaustiveness + duplicate-arm rules per
  DECISION B; keep the existing unreachable-after-wildcard check.
- **Emitter**: a literal-kind match lowers to `switch (<value>) { case 31: ‚Ä¶ }`
  with wildcard ‚Üí `default` (DECISION C).
- **Fixtures, formatter, grammar, book, changelog, ADR.**

Explicitly **out of scope** (DECISION E/F): `p where refinement-predicate`
patterns, or-patterns, and literal patterns on the RHS of `is`.

## Decisions

**[DECISION A] The literal set = ADR 0001's closed admission set, minus `()`.**
Admit **integer, string, and boolean** literals, plus **unary minus applied
directly to an integer literal** (`-1 => ‚Ä¶`). This is exactly the closed set
[ADR 0001](../decisions/0001-literal-admission-set.md) already defined for
literals-against-refined-types, reused verbatim so the language has *one* notion
of "a literal". Deliberately excluded:
- **Float** ‚Äî equality on `Float` is a footgun the language already treats
  carefully (ADR 0040/0044); `switch`/`===` on floating point is exactly the
  comparison we don't want to bless in pattern position. A `Float` scrutinee
  keeps the `match_non_sum_discriminant` rejection.
- **Unit `()`** ‚Äî matching a single-inhabitant type is vacuous; no arm carries
  information. Not worth a grammar rule.
*Recommend as stated.* The set can widen later (ADR 0001's own consequence note)
without breaking programs.

**[DECISION B] Exhaustiveness: `Int`/`String` require a wildcard; `Bool` is
exhausted by `true` + `false` *or* a wildcard.** `Int` and `String` are
unbounded, so a literal-only match can never be complete ‚Äî require a trailing
`_`, reusing `bynk.types.non_exhaustive_match`. `Bool` is genuinely two-valued;
forcing `_` on a type a reader knows is finite is a papercut, and "true/false is
exhaustive" is what every comparable language does. So `Bool` completes when both
literals appear *or* a wildcard does. Duplicate literals (`31 => ‚Ä¶` twice) reuse
the `duplicate_variant_arm`-style check under a new
`bynk.types.duplicate_literal_arm`. *Recommend the split rule.* **Fork:** the
cheaper uniform rule ‚Äî *always* require `_`, even for `Bool` ‚Äî is a legitimate
smaller cut (one code path, one message) if the reviewer prefers uniformity over
`Bool` ergonomics. Flagged for sign-off.

**[DECISION C] Emit a value-`switch`, not an if/else chain.** A literal-kind
match lowers to `switch (<disc>) { case 31: ‚Ä¶ default: ‚Ä¶ }`, mirroring the
existing tag-`switch` shape (`emit_match_tail`) so the source-map arm-to-arm
stepping contract (ADR 0103) is preserved for free. JS `switch` compares with
`===`, which is precisely the value equality we want for `Int`/`String`/`Bool` ‚Äî
no coercion surprises. The emitter picks the strategy from the pattern kind of
the arms (all-literal ‚Üí value-switch; all-variant ‚Üí tag-switch), which the
checker has already proven homogeneous. *Recommend as stated.*

**[DECISION D] A match is *either* variant-kind *or* literal-kind ‚Äî no mixing ‚Äî
and refined primitives are matchable against their base.** `variants_of` gains a
sibling that classifies the discriminant: sum/`Result`/`Option` ‚Üí variant-kind
(unchanged); `Int`/`String`/`Bool`, **or a refined type whose base is one of
those** (e.g. `type ShortCode = Int where InRange(‚Ä¶)`), ‚Üí literal-kind. A refined
scrutinee is a raw primitive at runtime, so literal arms are checked against the
*base* type; whether a literal arm's value actually inhabits the refinement is
**not** enforced (a dead arm is harmless, and requiring it would drag the static
evaluator of ADR 0001 into pattern position). Mixing literal and variant patterns
in one match falls out as a type error naturally (a literal can't match a sum
scrutinee, and vice-versa) ‚Äî surfaced as `bynk.types.pattern_type_mismatch`
rather than a bespoke code. *Recommend as stated; the "don't enforce inhabitance"
call is the one to eyeball.*

**[DECISION E] `p where refinement-predicate` patterns ‚Üí separate issue.** ¬ß2.3.4
lists them and #441 offers to fold them in. Decline: refined patterns interact
with refinement propagation (¬ß2.5) and only make sense once narrowing-in-patterns
is designed; bundling them balloons this increment and co: they share nothing with
literal parsing but the grammar nonterminal. *Recommend split* ‚Äî file a follow-on
"refined patterns (`p where ‚Ä¶`)" issue on merge.

**[DECISION F] Reject literal patterns in `is` position (for now).** The grammar
shares one `_pattern` nonterminal between match arms and `value is pattern`
(`tree-sitter-bynk/grammar.js:1081`). `is` is a type/refinement test; `x is 31`
reading as value equality would be a trap when `x == 31` is the intent. Parse it
(the grammar is shared) but have `check_is` reject a `Pattern::Literal` with a
diagnostic steering to `==`. *Recommend as stated.* Cheap to relax later if a use
appears.

**[DECISION G] Version = minor 0.130.0.** New language surface, backward
compatible (every program that parsed still parses). Language+tooling increment ‚Üí
tagged `v0.130.0`; run `scripts/bump-version.sh`.

## Spec delta

¬ß2.3.4 already lists `c` in the grammar, so the normative *grammar* is unchanged.
What it lacks is prose on **primitive-scrutinee matches**: that a `match` may
range over `Int`/`String`/`Bool` (and refinements thereof); the exhaustiveness
rule from DECISION B; that literal and variant arms don't mix. Add a short
"Literal patterns and primitive scrutinees" paragraph under ¬ß2.3.4, and drop the
section's "Settled in shape" framing to "Settled" for this form. One new ADR ‚Äî
**0158, literal patterns: scope + exhaustiveness** ‚Äî records A/B/D/E/F.

## Editor-surface delta (ADR 0156 ‚Äî all four named)

- **Semantic tokens:** a literal pattern tokenises as its literal kind
  (`number` / `string` / boolean keyword) ‚Äî the same classes the lexer already
  emits for those tokens in expression position. No new legend entry.
- **Completion:** match-arm completion (#465) currently suggests variant names
  from the sum discriminant. For a **literal-kind** scrutinee there are no names
  to complete ‚Äî it must **not** offer variant suggestions (there's no sum), and
  should still offer `_`. This is a real change to guard against, not "unchanged":
  the completion provider must consult the new discriminant classification.
- **Hover:** unchanged ‚Äî a literal in pattern position hovers as the literal /
  the scrutinee's type, via existing expr-type hover; no pattern-specific hover
  is added.
- **Signature help:** unchanged (N/A ‚Äî patterns take no argument list).

## Risks

- **`switch` fall-through / coercion.** JS `switch` uses `===` (no coercion) ‚Äî
  correct for our three types ‚Äî but empty-body fall-through is a footgun; the
  emitter already gives each arm a braced block with a terminal `return`
  (`emit_match_body`), so no arm falls through. Regression-covered by the
  snapshot.
- **The four-crate blast radius.** Adding a `Pattern` arm breaks every exhaustive
  `match Pattern` at compile time ‚Äî which is the safety we want (the Rust compiler
  enumerates the sites). Grep before starting: `Pattern::Variant` /
  `Pattern::Wildcard` across `bynk-check`, `bynk-emit`, `bynk-fmt`, `bynk-lsp`.
- **Refined-scrutinee inhabitance (DECISION D).** Not enforcing that a literal arm
  inhabits the refinement means `case 0:` under `Int where InRange(1,9)` is dead
  code, silently. Acceptable (harmless, and enforcing pulls in the static
  evaluator) ‚Äî but call it out in review; a future warning-channel lint
  (ADR 0117) could flag it without failing.
