# Proposal: `bynk dev` — build + serve a project locally

**Increment:** v0.57 (tooling — the `bynk` driver)
**Status:** draft for sign-off
**Depends on:** ADR 0083 (driver is a thin orchestrator over `bynkc` + Node),
ADR 0084 (doctor output/exit contract). Third step of the driver arc
**`doctor` → `new` → `dev`** (`bynk-tooling-roadmap.md` §5.1).

---

## 1. Why

Running a project locally today is a six-line ritual the user must perform by
hand, repeated in every example README:

```sh
bynkc check src
bynkc compile src --output out --target workers
cd out/workers/links               # you must know the dir name
# KV needs a namespace; create one and paste its id into wrangler.toml:
#   npx wrangler kv namespace create KV
npx wrangler dev
```

Four distinct frictions: a manual two-step compile with flags to remember; a
`cd` into a generated directory whose name you have to look up; an apparent
KV-provisioning step; and the bare `wrangler` invocation. The driver already
owns the surrounding concerns (`bynkc` resolution, tool probing, project
rooting) — this increment collapses the ritual to:

```sh
bynk dev          # from anywhere inside the project
```

### The unlock: local dev needs no provisioning

The scariest-looking step — "create a KV namespace and paste its id" — **is not
required for local development.** `wrangler dev` defaults to local mode
(Miniflare), which simulates KV, Durable Objects, and queues keyed by *binding
name*; the generated `id = "<KV_NAMESPACE_ID>"` placeholder is only read under
`--remote`. So `bynk dev` can serve every example in the gallery — including the
KV- and Durable-Object-backed ones — against the generated `wrangler.toml`
untouched. Provisioning is a `deploy`-time concern (the next slice), not a
`dev`-time one. This is what makes `dev` tractable independently of `deploy`.

The generated stanza is literally `id = "<KV_NAMESPACE_ID>"` (verified in the
emitted `wrangler.toml`); local mode *ignores* that field rather than validating
its shape. **Honesty caveat:** this premise rests on wrangler's documented
local-mode behaviour plus the literal placeholder — it has **not** been
exercised against a live `wrangler dev` in this environment (no `wrangler` is
installed or npx-cached here). A working `bynk dev` serving a KV-backed example
is therefore the **first implementation validation gate**, ahead of any other
work; if local mode turns out to validate the id, D4 grows a "rewrite the
placeholder to a throwaway local id before serving" step.

## 2. What ships

A new driver subcommand. Surface (final flag spelling is an implementation
detail; the shape is what's being signed off):

```
bynk dev [PATH] [--context NAME] [-- <wrangler args>]
```

Behaviour, in order:

1. **Locate the project** — two reused functions, not one: `find_project_root`
   (`bynk/src/main.rs`) walks up for `bynk.toml`, and `read_project_paths`
   (`bynkc/src/project/paths.rs`) reads `[paths] src`. `PATH` defaults to `.`.
2. **Pre-flight** — reuse `compiler::resolve` (override → PATH → sibling) and the
   doctor `Deploy` capability probe (Node + `wrangler`). A major `bynkc` skew or
   a missing required tool fails *before* compiling, with doctor's existing
   remedy text — not a confusing error from a half-built tree.
3. **Compile** — *clear `<build-dir>/workers/` first* (see D1: `bynkc compile`
   is **additive** — it `create_dir_all`s and writes per file but never prunes,
   verified in `bynkc/src/main.rs`), then shell `bynkc compile <src> --output
   <build-dir> --target workers`. (`bynkc` checks as part of compiling, so no
   separate `check`.)
4. **Select the worker** — enumerate `<build-dir>/workers/*/` (each carries a
   `wrangler.toml`). Exactly one → use it; `--context` names one explicitly;
   more than one with no `--context` → fail listing the available contexts.
   Because step 3 cleared the build dir, this enumeration only ever sees *this*
   build's contexts — a renamed or deleted context cannot linger and spuriously
   trip the ambiguity check (see D1).
5. **Serve** — spawn `wrangler dev` with **cwd set to the selected worker dir**
   (the emitted `index.ts` imports `../../runtime.js`, so the tree must stay
   intact and wrangler must run from the worker dir — exactly the manual
   recipe's `cd`). Inherit stdio (long-running, interactive). The driver and
   wrangler share the terminal's foreground process group, so a Ctrl-C `SIGINT`
   is delivered to *both* by the terminal — there is nothing to "forward". The
   requirement is the inverse: the driver must **not** bail on `SIGINT` before
   reaping the child, and must exit with wrangler's own exit code. No custom
   signal handling — just wait-and-propagate.

Resolution reuses `probe.rs` provenance ordering: project-local
`node_modules/.bin/wrangler` → `PATH` → `npx --yes wrangler`. As in doctor, an
`npx` resolution is surfaced as a notice (it *downloads* on first use), never a
silent green path.

### Implementation surface

Small, and almost entirely reuse:

- `cli.rs` — a `Command::Dev { … }` variant.
- `main.rs` — a `run_dev` arm.
- `dev.rs` (new) — the orchestrator: pre-flight → compile → select → spawn.
- Reused as-is: `compiler::resolve`/`Skew`, `probe.rs` (`Toolbox`,
  `Provenance`, `detect_npm_tool`), `find_project_root`, the doctor `Deploy`
  capability.
- New dependencies: **none** for the MVP (`std::process` only). `notify` enters
  only with the watch follow-up (Decision D2).

## 3. Decisions

### [DECISION D1] Build-output location — recommend a managed `.bynk/`

Where does `dev` compile to? Options: (a) the user's `out/` (collides with hand
builds they keep for inspection, and forces a `--target`/`--output` opinion onto
their tree); (b) the workspace `target/` (Rust-owned, confusing); (c) a
driver-managed **`.bynk/dev/`** under the project root.

**Recommendation: (c).** `bynk` is to `bynkc` as `cargo` is to `rustc`, and an
owned, clearly-ephemeral build dir is the cargo-shaped answer. The driver writes
`.bynk/.gitignore` containing `*` on first build (cargo's `target/.gitignore`
precedent), so a `dev` run never dirties `git status` and users needn't edit
their own ignore file. `out/` stays exclusively the user's explicit-compile
output.

Owning the dir also means owning its **staleness**. `.bynk/dev/` persists across
runs, and `bynkc compile` is additive (it never prunes — verified in
`bynkc/src/main.rs`), so a renamed or deleted context would leave a phantom
`workers/<old>/` behind and the §2 step-4 selection would read it as a live
context — turning a single-context project into a spurious "ambiguous, pass
`--context`" failure. So `dev` **clears `<build-dir>/workers/` before each
compile**. This is the concrete coupling between D1 and D3: persistent build dir
*requires* a clean step for the selection rule to stay honest.

### [DECISION D2] Watch & incremental recompile — recommend MVP-first

`wrangler dev` watches and reloads the *TypeScript* it serves, but not the
`.bynk` sources behind it. Full DX is: watch `src/**.bynk`, re-run `bynkc
compile` on change, let wrangler's own reload pick up the regenerated TS. The
roadmap (§5.1) explicitly assigns `dev` "the real design weight (incremental
recompile…)".

Fork: **(a)** ship the watch loop in this slice, or **(b)** ship compile-once +
hand-off now and add watch as a fast-follow.

**Recommendation: (b).** The compile-once path already removes *all four*
frictions above and is dependency-free and low-risk; watch adds a file-watcher
dependency (`notify`), debounce/error-surfacing design, and a recompile-failure
UX (keep serving the last good build vs. surface the error inline) that deserves
its own sign-off. Land the felt 80% now; design watch as a named v0.x follow-up.
*If the panel wants the whole arc in one increment, (a) is viable — the watch
loop is ~a day — but it widens the blast radius.*

### [DECISION D3] Multi-context projects — recommend select-or-default

A project may hold several contexts → several workers; `wrangler dev` serves
one. Options: run the sole context and require `--context` otherwise; or attempt
multi-worker dev (service-binding wiring between sibling workers).

**Recommendation: select-or-default.** One context → serve it; `--context NAME`
to choose; ambiguous → fail and list the contexts. Multi-worker local dev with
live cross-context Service Bindings is a real feature with its own design
(matches the cross-context-RPC boundary) and is explicitly **out of scope** —
noted as a limitation, not silently half-done.

### [DECISION D4] Provisioning — recommend none at `dev` time

**Recommendation:** `dev` provisions nothing and never edits `wrangler.toml`.
Local mode needs no namespace id (§1). `--remote` *does* need real ids — and a
flag whose only behaviour is to error is a worse surface than no flag — so
**`--remote` is omitted from the MVP entirely** (committing the stance left
open across the earlier draft's §2/D4/D5). Anyone who wants to try remote dev can
still reach it through the `--` passthrough (`bynk dev -- --remote`); real,
provisioned remote support arrives with `deploy`, whose defining problem *is*
provisioning.

### [DECISION D5] Forward-compatibility with wrangler — recommend `--` passthrough

Rather than mirror wrangler's flag surface (which drifts), the driver curates
**only `--context`** — its own concept (which worker to serve), not a wrangler
flag — and forwards everything after `--` to `wrangler dev` verbatim. In
particular there is **no curated `--port`**: it rides the passthrough (`bynk dev
-- --port 8788`). Curating it would create two paths to the same wrangler flag
and an undefined precedence when both are set — the exact "don't mirror
wrangler's surface" failure this decision exists to avoid. Keeps `bynk dev`
stable as wrangler evolves; pairs with doctor's existing wrangler version probe.

## 4. Risks

- **v1 `workerd` dev-server overlap.** `bynk-status-and-roadmap.md` (§"Held
  resources") anticipates a first-party `workerd` dev server that could one day
  supersede the wrangler hand-off. *Mitigation:* keep the wrangler invocation
  fully encapsulated in `dev.rs` behind the selection/pre-flight logic, so the
  serve step can be swapped without touching the rest.
- **wrangler version/flag drift.** *Mitigation:* `--` passthrough (D5) + doctor's
  version floor; the driver depends on `wrangler dev <dir>` and local mode being
  the default, both long-stable.
- **`npx` surprise download.** A wrangler resolved only via `npx --yes` downloads
  on first run. *Mitigation:* surface it as a notice exactly as doctor does;
  never present it as an installed-green path.
- **Multi-worker projects.** Service-binding-linked contexts won't be wired by a
  single `wrangler dev`. *Mitigation:* D3 scopes this out explicitly and the
  error names the contexts so the user understands the boundary.
- **Local state surprise.** Miniflare persists KV/DO state under `.wrangler/`
  between runs. *Mitigation:* already gitignored in the examples; document it and
  point at wrangler's persistence flags via the `--` passthrough.

## 5. Spec / contract delta

No language-surface change — this is driver tooling, so there is no normative
*language* spec delta. The contract that changes is the **driver CLI**:
`bynk dev` joins `bynk doctor` with its own argument/exit contract (success =
wrangler's exit code; pre-flight failures map onto doctor's existing exit
semantics). It is documented in the driver's CLI reference and pinned by output
goldens in the same style as doctor (ADR 0084), not duplicated here. Those
goldens can only cover the **deterministic** surface — pre-flight messages and
the context-selection output. The `wrangler dev` stream itself (ports,
timestamps, reload chatter) is non-deterministic and is *not* goldened; the
hand-off is covered by asserting the spawn (cwd, argv, propagated exit code),
not the child's stdout.

ADRs to record on implementation (one per defining call): **build-to-`.bynk/`**
(D1), **no-provisioning-at-dev** + local-mode rationale (D4), **context
selection rule** (D3). D2/D5 fold into the dev ADR as posture unless the watch
follow-up earns its own.
