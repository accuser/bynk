# v0.23 — Extend the Cloudflare adapter (`Kv.list`, structured values)

- **Status:** Draft (proposal). **Merging is approval to build.**
- **Builds on (verified):** the **v0.19 `cloudflare` adapter** (`karn.cloudflare`,
  `firstparty.rs:420`) — `exports capability { Kv }`, `Kv` = `get`/`put`/`delete` over
  `String`, `provides Kv = WorkersKv`, binding reads `env.KV` (one namespace, 0024/0025), no
  `binding` clause (toolchain supplies it); **adapter type declarations** — the `karn`
  surface declares `exports transparent { … }` records/enums/refined types
  (`KARN_ADAPTER_SRC`), so an exported result type is feasible; **`List` (v0.20b)** — what
  v0.19's "minimal, collection-free `Kv`" was waiting on, and what makes a *drain* `list`
  expressible; **platform-lock + wrangler `[[kv_namespaces]]` + env threading**
  (0017/0024/0025) — a new op on the already-platform-native `Kv` adds **no** new lock
  machinery; **v0.22 `Json.encode`/`Json.decode[T]` + `JsonError`** — which makes structured
  `Kv` values composable *today* (see Settled).
- **Spec sections touched:** §5 (static semantics — the new ops), §7 (emission — `WorkersKv`
  methods **and** the runtime `KVNamespace` interface, §7.4), §8 (compilation model —
  wrangler unchanged), §9 (diagnostics — none expected), Appendix B.
- **Scope discipline (0023, single-purpose):** an **adapter/surface** increment, no
  language-core change. `Queue` is **not** here (its consumer side is a new entry topology).
  One **language-core limitation** surfaced during design (the `given`-on-free-functions gap,
  below) is *recorded, not fixed* here.

## Settled (recorded as ADRs ≥0050)

- **▸ Structured values are *composition*, not new core surface.** v0.22's typed codec
  already makes typed `Kv` values expressible: `kv.put(k, Json.encode(order))` on write,
  `Json.decode[Order](v)` on the `Some(v)` read. v0.23 ships an **example** (a *behavioural*
  fixture — below), not new operations. First-class typed ops (`getJson[T]`/`putJson`) would
  need generic type-application on *capability* methods — deferred.
- **▸ `Kv.list` is a binding-side *drain*, not a caller-driven cursor loop.** *Forced by a
  language limitation:* the obvious cursor model ("caller loops until `complete`") is
  **inexpressible** — `FnDecl` has no `given` clause (`ast.rs:825`; `given` lives only on
  providers `:447` and handlers `:509`), so no recursive/iterative helper can hold `Kv`, and
  handlers can't self-recurse. So `list` **loops the cursor inside the TS binding** and
  returns the full key set as a `List[String]`. Host iteration inside the host-boundary
  adapter is exactly the adapter posture (0016).

## Discovered limitation (recorded, not fixed here → future increment)

**Free functions cannot take `given`** (`FnDecl`, `ast.rs:825`), capabilities are not
first-class values, and lambdas/handlers cannot self-recurse — so **no capability can be
driven from a recursive or helper-factored routine.** Plain recursion compiles, but nothing
recursive can reach a capability. This forced `list` into a drain (above) and **will recur**
the moment any capability needs driving from a helper (paged iteration, retry loops,
fan-out). The fix — `given` on free functions (or first-class capability values) — is
language-core and out of scope for an adapter increment; flagged here so it is not
re-discovered mid-implementation. *(This is why cursor-paging is deferred, not just unbuilt.)*

## Decisions (open — settle at review)

**[DECISION] Headline scope — `Kv.list` (drain) + structured-values example; `Queue`
deferred.** *Rec:* ship the **drain `list`** and the **structured-values example**; defer
**`Queue`** to its own increment. *Rationale:* `Queue`'s consumer side is a new handler
topology + `[[queues]]` config + batch semantics — a language-surface change that violates
single-purpose (0023) if folded in.

**[DECISION] `Kv.list` shape — drain now, cursor-paging deferred.** *Rec:*
`fn list(prefix: Option[String]) -> Effect[List[String]]` — the binding accumulates keys
across `env.KV.list({ prefix, cursor })` pages internally and returns them. **No
`KvListPage`/cursor type ships now.** *The fork (per the verdict, made here not mid-build):*
- **(a) drain only** *(rec)* — covers the common case, writable today, no new exported type.
- **(b) drain + a single-page `listPage(prefix, cursor) -> Effect[KvListPage]`** — a *single*
  page call is expressible (no loop), useful for bounded/incremental consumers; adds the
  transparent record `KvListPage = { keys: List[String], cursor: Option[String], complete:
  Bool }`. Defer unless an incremental consumer is a concrete need — a *streaming* consumer
  wants the `given`-gap fixed first anyway.
*Sub-decisions (if any list ships):* keys are **names only** (defer per-key
metadata/expiration); `prefix` optional via `Option`.

**[DECISION] `Kv` write options (TTL) — include as `putTtl`.** *Rec:* add
`fn putTtl(key: String, value: String, ttlSeconds: Int) -> Effect[()]` — TTL is the single
most-used KV write option, and a distinct method beats an options record given no optional
params. **`putTtl`, not `put_ttl`** — the language surface is camelCase throughout (`toFloat`,
`getOrElse`, `startsWith`, the records' `contentType`); a snake_case name would be the first.
Defer absolute `expiration` and `metadata`. *Alternative:* defer TTL entirely to keep v0.23
to the roadmap's letter — your call.

- **§9 Diagnostics** — none new (existing `vendor_required`/`vendor_conflict` cover the new
  ops for free).
- **Appendix B** — the **v0.23** entry.

## Diagnostics

None new. `list`/`putTtl` are ordinary ops on a platform-native capability, so
`karn.target.vendor_required` (consuming `Kv` under the wrong `--platform`) already applies.

## Fixtures (shape — not normative emission)

- **Positive (snapshot)** — a context `consumes cloudflare { Kv }` under `--platform
  cloudflare` calling **`list`** (prefix scan; the returned `List[String]` consumed with the
  existing `karn.list` combinators) and **`putTtl`**; `workers` and `bundle` variants
  (env-threading, 0025). Passes `tsc --strict`.
- **Behavioural (executed) — first executed adapter-op test** — a Node driver with a ~10-line
  in-memory fake `env.KV` round-trips `put`/`get`/`delete`, the **`list` drain**, and a
  **structured round-trip** (`Json.encode(order)` → `put` → `get` → `Json.decode[Order]`).
  Extends the `json_codec_behaviour.rs` pattern (snapshots are compile-proof only; this is
  behaviour-proof).
- **Negative** — `list` under `--platform node` → `karn.target.vendor_required`.
- **Re-bless (named exception, 0049 precedent — *not* a byte-stability claim)** — `WorkersKv`
  is a fixed string (`CLOUDFLARE_BINDING`) and the adapter's emitted `karn/cloudflare.ts`
  interface both gain methods, so fixtures **190/191/192** (each snapshots
  `karn/cloudflare.binding.ts`) are **re-blessed**; the diff is small, mechanical, reviewed.
  All *other* prior fixtures stay byte-stable.

## Risks
- **The drain is eager and unbounded** — a namespace with very many keys loads them all into
  one `List`. The honest minimal posture for v0.23; the deferred single-page op (or a future
  capability-driving construct) is the answer for huge namespaces. Note it normatively.
- **`given`-gap recurrence** — see Discovered limitation; the drain is a workaround, not a
  resolution.
- **`list` result mapping** — Cloudflare returns `{ keys: [{ name, … }], list_complete,
  cursor }`; the binding drains until `list_complete`, projecting `keys[].name`.
- **Re-bless across 190/191/192** — keep `get`/`put`/`delete` emission byte-stable; only the
  added methods (and the runtime interface) move.

## Out of scope (→ later)

- **`Queue`** (producer `send` + the consumer handler topology + `[[queues]]`) → its own
  increment.
- **Cursor-paging `listPage` + `KvListPage`** → deferred (and a *streaming* consumer wants
  the `given`-gap fixed first).
- **`given` on free functions / first-class capabilities** → a language-core increment (the
  Discovered limitation).
- **First-class typed `Kv` ops** (`getJson[T]`/`putJson`); **per-key metadata/expiration** on
  `list`; **absolute `expiration` / `metadata`** on write → later.
- **Fetch-headers retirement** (`Map[String,String]` on `Response`, 0022) → a separate
  `karn`-surface increment (0023).
