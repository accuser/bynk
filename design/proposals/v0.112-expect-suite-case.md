# v0.112 — `expect` + `suite`/`case`, and structural failure reporting (testing track, slice 1a)

- **Scope:** a single-increment **language** change — the first slice of the
  testing track. It replaces the `assert` keyword with `expect` and the braced
  `test` block with `suite`/`case`, and upgrades the failure report from a bare
  location to **expected-vs-actual** structure. No new *capability* beyond the
  report — the surface is a rename plus a richer diagnostic — which is exactly why
  it is the smallest standalone win. Version → **v0.112**.
- **Realises:** the testing track's spine (`design/tracks/testing.md`) at its first
  rung, and **ADR 0144** (one predicate surface) — `expect` *is* the invariant
  predicate, so production code and tests share one grammar. Closes track
  **DECISION B** (`assert` → `expect`) and the `suite`/`case` half of **DECISION Q**
  (`property` arrives in slice 2).
- **Relates:** ADR 0144 (one predicate surface — the organising commitment this
  slice first embodies); ADR 0098 (`bynkc test --format json` — the NDJSON runner
  protocol and the `path:line:col` assert location this slice enriches, not
  reshapes).
- **ADRs at implementation:** two, numbered at authoring (next free after 0144; do
  not pre-allocate): **B** — `expect` replaces `assert` (the structural-report
  rationale); **Q** — the `suite`/`case`(/`property`) vocabulary replacing the
  overloaded `test`. Both cite 0144.

## The surface, before and after

The change is a near-mechanical keyword substitution — the target of a suite is an
identifier and a case is a string, exactly as `test` had them, so nothing about
naming or nesting moves:

```bynk
-- before (today)                         -- after (v0.112)
test money {                              suite money {
  test "a tenth off a tenner" {            case "a tenth off a tenner" {
    assert net(100, 10) == 90                expect net(100, 10) == 90
  }                                          }
}                                          }
```

`expect` keeps `assert`'s **two forms** verbatim — a statement inside a case body
and the v0.9.1 expression form (`assert` in a `match` arm etc.) — so both parse
paths and both check sites move together. Inside a predicate it is the same pure
`Bool` grammar as `invariant` (`is` / `implies` / operators / pure methods); nothing
new is admitted.

The integration block is renamed for uniformity but **not** otherwise reworked —
its `wires`/`integration` machinery survives untouched until the tier dial (slice 6)
absorbs it:

```bynk
-- before                                          -- after
test integration "checkout" {                      suite integration "checkout" {
  wires Orders, Payments                             wires Orders, Payments
  test "authorises" { assert … }                     case "authorises" { expect … }
}                                                  }
```

After this slice the keyword `test` and the keyword `assert` are **gone** from the
language.

## What is genuinely new: structural failure reporting

`assert` today reduces a claim to one bit and throws
`AssertionError("assertion failed at <path:line:col>")` — the location, nothing
more (`bynk-emit/src/emitter/lower.rs:262`, the `__bynkAssertionFailure` helper in
`project/tests_emit.rs:1763`). `expect` retains the predicate's **operator/operand
structure** and reports expected-vs-actual for free:

```text
commerce.money › a tenth off a tenner
  expect net(100, 10) == 90
    expected:  net(100, 10) == 90
    actual:    90 == 91          (false)
  at tests/commerce/money.bynk:6:5
```

**Mechanism (emit-time decomposition, bounded).** When the top level of an `expect`
predicate is a binary comparison (`==`, `!=`, `<`, `<=`, `>`, `>=`) or an `is`
narrowing, the emitter lowers it so that on failure it evaluates **both operands**,
renders them, and includes the predicate's **source text** plus the substituted
values. For any other predicate (a bare `Bool`, a pure method call, a boolean
connective) it falls back to source-text + location — strictly no worse than
`assert` today. The `path:line:col` computation (ADR 0098 D0) is reused verbatim;
the enrichment is additive to the runner's existing `message` field, so the pinned
`--format json` document (ADR 0098) is unchanged in **shape** — only the `message`
string grows. **[DECISION — depth]** *Recommend* top-level comparison + `is` only
for 1a; deeper rendering (operands of a nested connective, per-conjunct attribution)
is a named follow-on, not this slice.

## What changes, against the current code

The rename touches every layer but is shallow at each; the one place with real new
logic is the emitter's structural report.

1. **Lexer / keywords** — `expect` is already a reserved-but-unused token
   (`bynk-syntax/src/lexer.rs:111`, `keywords.rs:91` "Reserved keyword"), so there
   is **no lexer conflict**: give it its description, retire the `Assert` token and
   the `Test` token (the keyword-group sync test at `lexer.rs:1036` moves in step).
2. **AST** (`bynk-syntax/src/ast.rs`) — rename `SourceUnit::Test`→`Suite`,
   `TestDecl`→`SuiteDecl`, `TestCase`→`Case`, `AssertStmt`→`ExpectStmt`, and
   `ExprKind::Assert`→`ExprKind::Expect`. `MockDecl` and `IntegrationDecl` keep
   their shape (their cases become `Case`); `SuiteDecl` retains `target` / `uses` /
   `mocks` / `cases` / `form` unchanged.
3. **Parser** — `parse_test_brace`/`parse_test_fragment`/`parse_test_case`
   (`parser/declarations.rs:555,667,1052`) accept `suite`/`case`; the two `assert`
   paths (`parser/statements.rs:184`, `parser/expressions.rs:13`) accept `expect`.
   The implicit-unit-tail special case for an assert-ending block
   (`parser/statements.rs:150`) transfers to `expect`.
4. **Checker** — `check_assert` (two sites: `checker/expressions.rs:322`,
   `checker.rs:1425`) becomes `check_expect`, still gating on `in_test_body` and
   still requiring `Bool`. Crucially, `predicate_impure_construct`
   (`checker.rs:682`) must reject `ExprKind::Expect` exactly as it rejected
   `Assert` — an `invariant` predicate still may not contain `expect`.
5. **Diagnostics** (`bynk-syntax/src/diagnostics.rs`) — normalise per ADR 0144 D4:
   `bynk.assert.non_bool` → **`bynk.expect.not_bool`** (note `non_bool`→`not_bool`,
   joining the `bynk.<position>.not_bool` family); `bynk.assert.outside_test` →
   `bynk.expect.outside_case`; `bynk.test.duplicate_case_name` and the other
   `bynk.test.*` → `bynk.suite.*`. Category titles (`diagnostics.rs:1575`):
   `"expect"`→"Expectations", `"suite"`→"Suites and cases". `Mock[T]`/`mocks` codes
   are untouched.
6. **Emitter / runner** — `lower_test_case_body`/`assert_location`
   (`emitter/lower.rs:36,326`) and the emitted helpers (`tests_emit.rs:1763`) gain
   the structural report above; the NDJSON events (`test_json.rs`) keep their schema,
   with a richer `message`.
7. **tree-sitter grammar** (`tree-sitter-bynk/grammar.js`) — `test_decl`→`suite`,
   `test_case`→`case`, `assert_expr`→`expect_expr` (field `cond` kept),
   `integration_decl`→`suite integration`, `PREC.assert`→`PREC.expect` (still
   lowest); then `BYNK_BLESS=1 cargo test -p bynk-grammar --test
   generated_grammar_json` regenerates `site/src/generated/grammar.json`. Update the
   TextMate keyword lists (`vscode-bynk/syntaxes/bynk.tmLanguage.json:42,51`).

## Migration (clean-slate; codemod-assisted)

`assert`/`test` are removed, not deprecated (the track's clean-slate posture). A
codemod handles the four mechanical rewrites: `assert ` → `expect `; `test <id> {`
→ `suite <id> {`; inner `test "` → `case "`; `test integration "` → `suite
integration "`. The ~35 fixtures under `bynkc/tests/fixtures/{positive,negative}`
named `*test*`/`*assert*` are rewritten and their expected `.test.ts` outputs
regenerated (the failure-message shape changes, so positive expected outputs move);
negative `expected_error.txt` files change only where the diagnostic *code* is
renamed (`bynk.assert.*`→`bynk.expect.*`, `bynk.test.*`→`bynk.suite.*`). The file
model is **unchanged** here (`.test.bynk`, `[paths] src`/`tests`) — DECISION S is
slice 1b.

## Risks

- **Two assert forms, one rename.** The statement and expression forms must move
  together or a half-migrated tree parses inconsistently; the fixture sweep plus the
  `predicate_impure_construct` gate (which must now reject `Expect`) are the guards.
- **Structural report is the only real logic.** Bounding it to top-level
  comparison + `is` keeps the emitter change small; over-reaching into nested
  rendering is where 1a could bloat — hence the depth decision above.
- **Grammar/spec drift.** Renaming keywords in `grammar.js` without re-blessing
  `grammar.json` leaves the spec `{{#grammar}}` embeds and llms-full stale — the
  bless + llms-full steps are part of the increment, not follow-up.
- **Fixture churn is large but low-risk.** ~35 fixtures is real work, but the codegen
  is syntax-agnostic apart from the failure message, so most expected-output diffs
  are the keyword rename plus the enriched message.
