# v0.108.4 — The REPL shell / playground (in-browser track, slice 4)

- **Scope:** the closing slice of the [in-browser track](../tracks/in-browser.md) —
  the playground web app. **Outside the Rust workspace** (a static client-side app
  under `playground/`); no language/compiler change. **Security-bearing** (it runs
  untrusted Bynk in the browser), so `/security-review`-gated per the track posture.
- **Realises:** the track's §3.4 (REPL + sandbox), §3.5 (hosting/origins), §4
  (security), and Q4–Q7. Lands the **REPL execution & sandbox ADR**
  ([ADR 0140](../decisions/0140-repl-execution-and-sandbox.md)), which also ratifies
  the Q7 deep-link read side (shared with the documentation track).

## What this adds

- **`playground/`** — `esbuild` + vanilla TS + CodeMirror 6, two bundles (the app +
  the sandbox execution document). Build scripts: `build-wasm.sh` (cargo wasm32 +
  wasm-bindgen → glue), `build-grammar.sh` (tree-sitter → web-tree-sitter wasm),
  `build.mjs` (esbuild), `serve.mjs` (two-port local server for cross-origin verify).
- **Run flow** — `bynk_compile(source)` (slice-3 wasm, lazy-loaded) → diagnostics
  panel; on success, the JS graph is linked as blob-URL ES modules and run in the
  sandbox; `Logger` output + the returned value render in the output panel.
- **Sandbox** — a separate-origin `<iframe sandbox="allow-scripts">` + Web Worker,
  hard wall-clock timeout, origin/window-validated postMessage (ADR 0140 D2–D4).
- **Deep-link** — `#base64url(deflate-raw(utf8))` via CompressionStream, decoded on
  load; a Share button. **Unsupported path** — platform-lock diagnostics shown as
  *not runnable in-browser* (D6). Starter template (whole-unit, Q5).

## Decisions (all confirmed with the maintainer)

- **[A] Stack:** esbuild + vanilla TS + CodeMirror 6.
- **[B] Sandbox:** cross-origin iframe + Worker on a **second hostname**
  (defence-in-depth).
- **[C] Scope:** build + verify locally; **Cloudflare Pages deployment is the
  maintainer's ops step** (two projects + DNS).
- **[D] Deep-link:** `base64url(deflate-raw(utf8))` via CompressionStream.
- **[E] Version — patch v0.108.4.**

## Verification

The Chrome browser-automation harness was **unavailable in this environment (the
extension was not connected)**, so the in-browser DOM/iframe/Worker flow was not
driven here. Everything else was verified in Node:

- `bynk_compile` (the wasm artefact) loads and runs in a JS host → the full JS graph
  for good input, diagnostics with line/col for bad input.
- The **emitted browser JS graph runs** end-to-end (`composeApp()` → zero-arg entry):
  `Logger.info` output captured + the `Clock.now()` value returned.
- The **blob-URL linker algorithm** (resolve / import-detect / topological order /
  specifier rewrite) links the real graph so it imports and runs.
- The **deep-link** round-trips (`deeplink.ts`), and garbage decodes to `null`.

Remaining (manual / once the Chrome extension is connected): the CodeMirror UI, the
cross-origin postMessage round-trip + timeout in a real browser, and hash-load. Steps
in `playground/README.md`.

## Risks & mitigations

- **In-browser flow unverified here** (no Chrome). *Mitigation:* the load-bearing
  logic is verified in Node; the DOM/sandbox wrapper is small and documented for
  manual verification.
- **wasm payload (~18 MB debug).** *Mitigation:* `--release` + `wasm-opt` + lazy-load
  for deploy (Q3).
- **web-tree-sitter highlighting deferred** (grammar-wasm build needs emcc/docker,
  absent here). *Mitigation:* a faithful stream highlighter ships now;
  `build-grammar.sh` + the integration are the follow-on.
- **No playground CI leg** (the build needs the wasm target + wasm-bindgen-cli +
  npm). *Mitigation:* documented build; a CI leg is a follow-on.
