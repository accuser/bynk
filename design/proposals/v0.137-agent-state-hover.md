# v0.137.0 — Hover for the `key`/`store` contextual keywords and the agent state fields they declare

- **Scope:** an **LSP-only** change — no grammar, no AST, no checker, no emitter.
  A new pure hover path in `bynk-lsp` (`symbols::describe_agent_state_at`), a
  parallel `CONTEXTUAL_KEYWORDS` registry in `bynk-syntax::keywords`, and one new
  public surface renderer (`bynk-fmt::annotation_to_string`). Closes #476.
- **Addresses:** hovering `key` or `store` in an `agent` body produces **no
  hover**. Both are *contextual* keywords — lexed as `Ident`, not reserved
  tokens — so they are absent from `bynk-syntax::keywords::KEYWORDS` and never
  reach `describe_keyword_at`; and the state fields they declare are neither
  top-level declarations (`describe_symbol`) nor `let`/param locals
  (`describe_local_at`), so every hover path falls through. The data to render a
  useful hover already exists — `document_symbols.rs` already classifies these
  fields as `"key"`/`"store"` for the outline.
- **Realises:** a hover for the agent-state surface a reader reaches for first,
  and — via a second mechanical coverage test — a floor that fails when a future
  contextual keyword lands without one, closing the gap in ADR 0156's floor that
  let #476 exist.

## Decisions

See **ADR 0161** (landed with this increment) for the full rationale. In brief:

- **D1 — a parallel `CONTEXTUAL_KEYWORDS` registry, not entries in `KEYWORDS`.**
  Adding `key`/`store` to `KEYWORDS` would fail the `keywords_reference` drift
  guard (asserts `KEYWORDS` == the lexer's reserved tokens) and misrepresent them
  as reserved words in the generated reference page.
- **D2 — the hover renders the field the keyword introduces**, on the keyword
  *or* the field name alike: `key id: String` / `store items: Map[String, Int]
  @indexed(by: id) @bounded(10000)` plus the one-line contextual-keyword doc.
  Annotations render through the formatter's own `annotation_to_string` (no
  drift). Matching is by span, never by name, so an identifier that merely reads
  `id`/`items` elsewhere is never mistaken for the declaration.
- **D3 — the mechanical floor extends to `CONTEXTUAL_KEYWORDS`** — a second
  coverage test beside the reserved-keyword one.

## Tooling deltas (ADR 0156)

- **Hover:** now covers the `key`/`store` contextual keywords and the agent
  key/store field-name declarations (the change). Registered in the
  `describe_symbol`-miss arm, before any project-wide scan (single-file-local).
- **Completion:** unchanged — the agent scaffold already emits `key`/`store`, and
  contextual keywords are not offered as standalone completion items.
- **Semantic tokens:** unchanged — `key`/`store` continue to tokenize as
  identifiers; no new token type.
- **Signature help:** unchanged — neither keyword introduces a callable.

## Docs delta

No **book** page changes: `key`/`store` are already documented in the agents
guide, and the keyword *reference* page (generated from `KEYWORDS`) deliberately
excludes contextual keywords. The durable record is ADR 0161.

## Tests

- `symbols.rs` unit tests: all four cursor targets (`key` kw, key name, `store`
  kw, store name), the rendered annotations, and the negative cases (agent name,
  store kind, an annotation-argument `id`).
- `editor_coverage.rs`: `every_contextual_keyword_has_a_hover_path` — the D3
  floor over `CONTEXTUAL_KEYWORDS`.
