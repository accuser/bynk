## What changes

The track calls codelens "the healthiest — but narrow." This slice widens it in
the two directions slice 6 names:

1. **A per-case test-run filter.** Today the `▷ Run Test` / `Debug Test` lenses
   above a `case` shell the *whole project* — `testCodeLens.ts` says so in a
   comment ("The CLI runs the whole suite… the command is project-wide"), because
   `bynkc test` had no single-test filter. This slice adds one: `bynkc test
   --case <name>` runs only cases whose name matches, and the lens passes the case
   under it, so `▷ Run Test` runs *that* case.
2. **A provider-count lens on capabilities.** A `capability` declaration gains a
   `N providers` lens listing the services that `provides` it — the
   "what provides T" half of [#259](https://github.com/accuser/bynk/issues/259),
   which the binding index already answers (`impls_of` / `implementations`, wired
   to `textDocument/implementation`) but never surfaced as a lens.

Closes [#259](https://github.com/accuser/bynk/issues/259) (the providers half —
see the Decision on the refines half). Migration is nil; both halves are additive
over an existing surface and touch no language semantics.

## Tooling delta

### Part A — `bynkc test --case <name>` and the per-case lens

- **Runner (`bynk-emit`).** The emitted `tests/main.ts` already reads
  `process.env.BYNK_TEST_FORMAT`/`BYNK_TEST_SEED`; it now also reads
  `process.env.BYNK_TEST_CASE` and threads it as an optional `only` argument into
  each module's `run(only?)`. Every emitted suite `run()` (unit and system)
  guards each `results.push(...)` with `only === undefined || only === "<case>"`,
  so a filtered run **executes** only the matching case(s), not merely reports
  them — the NDJSON `suite-begin`/`case` events and the human `✓`/`✗` output both
  reflect the filtered set. An unmatched or empty filter degrades to "run all".
- **CLI (`bynkc`).** `bynkc test` gains `--case <name>`, threaded to the runner
  child's environment as `BYNK_TEST_CASE` in both the `tsc → node` path
  (`finish_runner`) and the `--inspect` debug path (`run_inspect`), mirroring how
  `--seed` becomes `BYNK_TEST_SEED`. `--no-run` discovery is unaffected (the
  filter is a run-time concern; discovery still lists every case).
- **VS Code client.** `testCodeLens.ts`'s `▷ Run Test` / `Debug Test` lenses now
  carry the case name and invoke new `bynk.runTestCase` / `bynk.debugTestCase`
  commands (internal — not palette-contributed). `runTestCase` builds a
  single-item `TestRunRequest` and routes through the existing run handler, which
  derives the case name from `request.include` and passes `--case`; the tree item
  lights up its own pass/fail state. `debugTestCase` threads the name into the
  `bynkc test --inspect --case <name>` launch. The whole-project `bynk.runTests` /
  `bynk.debugTests` commands (palette + editor menus from slice 5) are untouched.

### Part B — provider-count lens on capabilities

- **`bynk-lsp`.** A new `index_queries::capability_provider_lenses` pairs each
  `SymbolKind::Capability` definition in the file with its provider definition
  sites (via the existing `impls_of` edge / `implementations` query — the same
  data behind go-to-implementation). The `code_lens` handler appends, for each,
  an `N provider(s)` lens carrying `editor.action.showReferences` over the
  provider defs — the same client-side hydration the reference-count lens uses
  (`codelens.ts`), so **no VS Code change is needed** for this half. A capability
  with providers therefore shows two stacked lenses (`N references`,
  `N providers`), exactly as a referenced test shows a reference + test lens.

## Decision

- **The "refines T" half of #259 stays parked.** #259 asks for both "what
  *provides* a capability" and "what *refines* a type". The provider relation is a
  first-class index edge (`ImplEdge`), so the providers lens is a pure query. A
  "what refines this type" relation, by contrast, **does not exist** in the index
  — `refined` is only a semantic-token modifier, with no edge linking a refined
  type to its base or to its refiners. Adding that edge is a checker/index change
  disproportionate to this "optional tail"; it is left as a standalone follow-up.
  This slice delivers the lens the track names ("on capabilities"), not a new
  index relation.
- **Filter granularity is by case name, run-time only.** `--case` matches the
  case *name* across suites (case names are unique within a suite; a cross-suite
  name clash runs both, an acceptable convenience for the lens, whose payload is
  the discovered name). It filters execution inside each suite's `run()`, so it is
  honest — a filtered run does not execute the siblings — without a new CLI notion
  of "suite::case" addressing.

## Test delta

- **`bynk-emit`** unit test: the emitted top-level runner reads `BYNK_TEST_CASE`
  and each suite `run()` guards its pushes with the `only` filter.
- **`bynkc`** behavioural test `case_filter.rs` (mirrors `property_behaviour.rs`
  — drives the real CLI, skips loudly without a TS runner, `BYNK_REQUIRE_TSC=1`
  turns the skip into a CI failure): a fixture with two named cases, run with
  `--case "<one>"`, asserts only that case executed (its name present, the
  sibling absent, `1 passed, 0 failed.`).
- **`bynk-lsp`** `index_queries` test: `capability_provider_lenses` returns a
  capability def paired with its provider defs, and is empty for a capability with
  no providers.
- **VS Code** `manifest.test.ts`: the `bynk.runTestCase` / `bynk.debugTestCase`
  commands are registered (the CI-only electron suite).
