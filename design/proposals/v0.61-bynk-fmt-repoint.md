# v0.61 — Re-point `bynk-fmt` onto `bynk-syntax`: the façade becomes a real leaf (crate-decomposition slice 2)

- **Track:** [crate-decomposition](../tracks/crate-decomposition.md), slice 2 —
  the first slice where the decomposition *pays off visibly*: `bynk-fmt` stops
  linking the whole compiler.
- **Realises ADRs:** [0099](../decisions/0099-crate-layering-dependency-direction.md)
  (downward layering — decomposition D4: keep `bynk-fmt`, re-point it onto
  `bynk-syntax`, move the impl down), [0102](../decisions/0102-foundation-types-boundary.md)
  (the syntax foundation `fmt` walks lives in the leaf).
- **Depends on:** slice 1 (`bynk-syntax`, v0.60 — shipped). **Builds on:** the
  re-export pattern slice 1 established.
- **Posture:** a **pure internal refactor** — behaviour-preserving, no language
  surface, no formatter behaviour change. The `bynk-fmt` golden + roundtrip
  suites must pass byte-identical.

## The problem this fixes

`bynk-fmt` today is the track's cautionary example: a one-line **upward façade**
(`pub use bynkc::fmt::{…}`) over a `bynkc` dependency. It narrows the *name* but
not the *link cost* — anything depending on `bynk-fmt` still links the entire
compiler (resolver, checker, emitter, project). Yet formatting is an **AST-walk**:
`fmt.rs` (1894 lines) touches only `crate::{ast, error, lexer, parser}` — all of
which moved to `bynk-syntax` in slice 1. It needs syntax, not the checker.

## What this slice does

Turn the cosmetic façade into a **real leaf**:

1. **Move `bynkc/src/fmt.rs` → `bynk-fmt`** (as `bynk-fmt/src/fmt.rs`, re-exported
   from `lib.rs`). Re-point its four imports `crate::{ast, error, lexer, parser}`
   → `bynk_syntax::{…}`.
2. **`bynk-fmt/Cargo.toml`:** depend on **`bynk-syntax`** instead of `bynkc`.
   `bynk-fmt` now links syntax only — the compiler is gone from its tree.
3. **`bynkc`:** drop the `pub mod fmt;` and instead depend on `bynk-fmt`,
   re-exporting it so **`bynkc::fmt::{FormatError, FormatOptions, IndentStyle,
   format_source}` is preserved** (see DECISION 1). `bynkc` still needs the
   formatter for its own `bynkc fmt` command (`main.rs`).

The dependency direction after:

```
bynk-syntax  [leaf]
   ▲      ▲
   │      └── bynk-fmt   (impl; syntax only — no compiler)
   │              ▲
bynkc ───────────┘  (depends on bynk-fmt for `bynkc fmt`; re-exports as `bynkc::fmt`)
```

## Decisions

**[DECISION 1] Keep `bynkc::fmt` as a re-export — recommend yes.** `bynkc`'s own
`fmt` command and `bynk-lsp/src/project.rs` use `bynkc::fmt::{…}`. Preserve that
path with `pub use bynk_fmt as fmt;` in `bynkc` (the slice-1 pattern: stable
public API, zero consumer churn). *Alternative:* delete `bynkc::fmt` and switch
`main.rs` + the LSP to `bynk_fmt::` directly — a cleaner long-term surface but a
breaking change to `bynkc`'s API for no gain this slice. **Rejected**; if `bynkc`
later wants to shed the re-export it is a trivial follow-up. Note `bynk-lsp` keeps
depending on both `bynkc` and `bynk-fmt`; its formatting *path* is now
compiler-free even though the LSP **binary** still links `bynkc` for diagnostics
until slice 5.

**[DECISION 2] Version — recommend a minor `v0.61`.** No new crate (unlike slice
1), but `bynk-fmt`'s **public dependency footprint changes** (it no longer pulls
`bynkc`), which is observable to any consumer. Follows the per-increment minor
cadence (v0.60 → v0.61). *Alternative:* a patch — rejected; the repo reserves
patches for the refactor track (ADR 0059), and this ships as a normal increment.
No new crate means **no new release surface** — `bynk-fmt` is already published,
so no bootstrap-seed is needed (contrast slice 1).

## What this slice does *not* do

- It does not slim the `bynk-lsp` **binary** — that links `bynkc` for diagnostics
  until slice 5. The win here is that `bynk-fmt` *as a crate* (and any future
  consumer of just-formatting) stops linking the compiler.
- It does not touch formatter behaviour, options, or output. Same code, new home.

## Risks & mitigations

- **A formatter regression hidden in the move.** Mitigated by the existing
  `bynk-fmt` **golden** + **roundtrip** suites (already `bynk_fmt::`-based, so
  they exercise the new home directly) — they must pass byte-identical, and the
  scaffold's "compiles and is `bynk-fmt`-clean" test (`bynk new`) is a second net.
- **A missed `bynkc::fmt` consumer.** Verified: only `bynkc/src/main.rs` and
  `bynk-lsp/src/project.rs` use `bynkc::fmt`; both keep working via the
  DECISION 1 re-export.
