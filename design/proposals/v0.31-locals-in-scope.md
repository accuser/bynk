# v0.31 — Locals in scope: the scope-at-offset foundation

- **Status:** Draft (proposal). **Merging is approval to build.** *(Tooling-queue item 3, the recurring
  deferral. Records **local bindings** — `let`/`let <-`, lambda/handler/fn params, match-arm pattern
  bindings — with their lexical **scope ranges**, so the editor can navigate and complete them. Lifts
  references/rename, semantic tokens, and completion to locals "in one move." Spans `karnc` (the
  recording machinery) and `karn-lsp` (the consumers); multi-slice.)*

## Context — the recurring deferral

Locals have been deferred at every A-tier LSP increment: the v0.25 index (`index.rs:13` — "local
bindings are deferred (no edges are recorded for them)"), v0.27 inlay hints, v0.28 semantic tokens, and
v0.30.2 completion all stop at top-level symbols. The blocker each time is the same: **no scope-at-offset
query exists** — the index tracks only top-level symbols, and the checker's local scopes are transient.

A feasibility scout confirms the path is clear (no fundamental unknown, unlike v0.30.2's mid-edit
question): the checker *already* tracks locals in a `scopes: Vec<HashMap<String, Ty>>` stack
(`checker.rs:408`), entering bindings at known sites with their name **spans** available — it just
**discards** the scopes after use. This increment records them instead.

## Builds on (verified — scout citations)

- **Every binding site is known, with spans:** `let`/`let <-` in `type_of_block` (`checker.rs:828/914`,
  name span at `l.name.span`), lambda params (`expressions.rs:698`), fn/handler params
  (`calls.rs:44-57`, `checker.rs:243-265`), match-arm pattern bindings (`expressions.rs:1765/1794`).
  Scope push/pop is centralised (`checker.rs:474-478`).
- **The sink pattern is proven** — `RefSink` (index), `HintSink` (v0.27), `ExprTypeSink` (v0.30.2) all
  thread `&mut` through `check_record` and surface per-file data on `ProjectAnalysis`
  (`FileHints`/`FileExprTypes`). A `LocalsSink` → `FileLocals` is the same shape.
- **Consumers are mostly ready:** references/rename (`index_queries.rs::sites_for`/`definition_at`)
  explicitly exclude locals (`index_queries.rs:51`) and just need a per-file scope-aware query — locals
  are file-local (no cross-file refs); semantic tokens iterate `index.symbols` and can take a parallel
  loop over locals; inlay hints **already** record local binding identifiers + types
  (`checker.rs:822`). Expression-position completion does **not** exist yet — it's the new context.

## The core model

A `LocalsSink` (mirroring `HintSink`) records, per file, one entry per local binding:

```
LocalBinding { name, def_span, ty_display, scope: Span /* start..end */, refs: Vec<Span> }
```

surfaced as `ProjectAnalysis.locals: FileLocals` (and through to `ProjectDiagnostics`, like
`expr_types`). The query `locals_at(file, offset)` returns the bindings whose `scope` contains the
offset (innermost-shadowing resolved by source order). `ty_display` is pre-rendered (Karn surface
syntax, as hints do) so no `Ty` crosses the surface.

**Scope ranges are captured at push/pop, not reconstructed from the AST.** The sink mirrors the
checker's scope stack: on `push_scope` it records the frame's start offset; each `bind` adds a binding
(start = the *statement after* the binding, per Karn's sequential `let` visibility); on `pop_scope` it
stamps every frame binding's `scope.end` from the block's tail/exit span. This piggybacks on the
checker's existing, correct scoping rather than re-deriving it (the scout's #1 risk — nesting/shadowing
correctness — is thereby the checker's already-tested behaviour, not new logic).

## Decisions (open — settle at review)

**[DECISION] Home: `ProjectAnalysis.locals` (parallel to `hints`/`expr_types`), not the index.** *Rec:*
yes — locals are per-file and never cross-file, so the cross-file `ProjectIndex` (def + global refs) is
the wrong shape; a per-file `FileLocals` with scope ranges is right and reuses the analysis envelope.

**[DECISION] Record refs in the foundation, or defs only?** *Rec:* record **both** — the binding def
(+ scope + type) *and* its in-scope use sites — in the foundation slice, so references/rename/tokens are
cheap to wire after. Local refs are recorded where the resolver already validates a local
(`resolver.rs:646`, `name_in_scope`) — it currently records nothing; the foundation adds the edge to the
binding's def span. (Completion needs only defs + scope; nav needs refs.)

**[DECISION] Slicing.** *Rec:*
- **Slice 1 — foundation (`karnc`):** `LocalsSink` + `FileLocals`, wired through the binding sites,
  surfaced on `ProjectAnalysis`/`ProjectDiagnostics`; a `locals_at` query. Compiler-only, fixture-tested
  (names, def spans, scope ranges, refs under nesting/shadowing). No LSP change — like the v0.30.2
  part-1 infra.
- **Slice 2 — navigation (`karn-lsp`):** references/rename for locals (scope-aware, same-file; rename
  validated within scope) + semantic tokens (a `local`/parameter token, additive over the legend).
- **Slice 3 — completion (`karn-lsp`):** the new **expression-position** context — a bare partial
  identifier in a body that isn't one of the existing specific contexts — offering in-scope locals
  (`locals_at`) + params. The hardest part (detecting expression position lexically); kept conservative,
  fixtured.

**[DECISION] New decision record.** *Rec:* **ADR 0064** — locals recorded via a sink with scope ranges
captured at the checker's push/pop, homed per-file on the analysis (not the cross-file index); the
slicing.

## Tests

- **`karnc`** (slice 1): a fixture asserting `let`/`let <-`/lambda-param/fn-param/handler-param/match-arm
  bindings are recorded with correct names, def spans, **scope ranges**, and ref sites — including a
  **nested/shadowing** case (`let x = { let x = 1; x }`) where the inner `x`'s scope is the inner block
  only, and a match-arm case (per-arm binding scope).
- **`karn-lsp`** (slices 2–3): references/rename on a local stay within its scope; a renamed local
  doesn't capture a shadowing use; semantic tokens colour local def+refs; expression-position completion
  offers in-scope locals and nothing out of scope.

## Diagnostics / Risks

- **Scope-range correctness under nesting/shadowing** — the central risk; mitigated by capturing ranges
  at the checker's own push/pop (reusing tested scoping) rather than re-deriving from the AST, and by the
  explicit nested/shadowing fixtures.
- **Expression-position detection** (slice 3) — the general "bare identifier in a body" context is
  broader than the existing specific triggers; keep it conservative (don't flood), fixtured.
- **Memory/size** — recording every local + refs per file; bounded by file size, same order as hints.

## Out of scope (→ later)

- The other deferred index kinds (instance methods, record fields *as index entries*, capability op
  names, generic type parameters) — separable from local bindings.
- Cross-file anything (locals are file-local by definition); signature help; the v0.30.2 clean-file
  ceiling (orthogonal).

## Definition of done (slice 1 — the foundation)

A `LocalsSink` records per-file `LocalBinding`s (name, def span, rendered type, scope range, ref sites)
at the checker's binding sites, surfaced as `ProjectAnalysis.locals` / `ProjectDiagnostics.locals`, with
a `locals_at(file, offset)` query; fixture tests cover every binding kind plus nested/shadowing scope
ranges; `karnc` build/test/clippy(`--all-targets`)/fmt green; ADR 0064 + spec note; changelog; version
→ 0.31.0. Slices 2–3 (navigation, completion) follow as their own PRs. The implementing PR(s)
**delete this file** with the final slice.
