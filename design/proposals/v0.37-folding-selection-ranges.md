# v0.37 — Folding & selection ranges

- **Status:** Draft (proposal). **Merging is approval to build.** *(Tooling-queue item 7, the structural
  polish increment. `textDocument/foldingRange` (collapse contexts, services, handler bodies, match arms,
  blocks…) + `textDocument/selectionRange` (smart expand-selection). Both are **purely structural**,
  driven by the recovered AST — no binding-index or analysis data, no compiler-side change.)*

## Context

Folding and selection are the two structural editor features the server doesn't yet provide. Both read
the same thing: the per-file **recovered AST** the LSP already parses for document symbols. Every AST node
carries a byte `span`, and `position::span_to_range` already converts spans to LSP ranges. So both
features are a single AST walk plus a thin transport handler — the established document-symbols shape.

## Builds on (verified — scout)

- **The parse path exists.** `document_symbols::outline` calls `karnc::lexer::tokenize` →
  `karnc::parser::parse_unit_with_recovery(&tokens, source) -> (Option<SourceUnit>, _)` (recovery on, so
  it works mid-edit). Folding/selection reuse exactly this (`document_symbols.rs:20-35`).
- **Spans on every node.** `Commons`/`Context`/`Adapter` containers, `TypeDecl`/`FnDecl`/`CapabilityDecl`/
  `ServiceDecl`/`AgentDecl`, `Handler.body: Block`, `Block`, `MatchArm`, `Expr` (If/Match/Block/Record…)
  — all carry `pub span: Span` (`ast.rs:40/81/647/825/501/888/1140/1007`). `span_to_range`
  (`position.rs:92`) → LSP `Range`; `FoldingRange` is line-based (`Position.line`).
- **AST-driven is the house path.** `karn-lsp` has **no tree-sitter dependency** (`Cargo.toml`);
  document-symbols, locals-nav, semantic-tokens all walk the hand-written `karnc` AST. Folding/selection
  follow suit — no second grammar to sync.
- **Neither provider is declared.** `server_capabilities()` has no `folding_range_provider` /
  `selection_range_provider`; the tower-lsp methods are `folding_range(FoldingRangeParams) ->
  Option<Vec<FoldingRange>>` and `selection_range(SelectionRangeParams) -> Option<Vec<SelectionRange>>`.
- **Comment spans need the token stream.** The parser's trivia keeps comment *bodies*, not spans
  (`parser.rs` `split_trivia`) — but the **lexer's `Comment` tokens carry spans**, so comment-run folding
  is a cheap token scan (the provider already tokenizes), not a parser change.

## Decisions (open — settle at review)

**[DECISION] One shared AST visitor.** *Rec:* a single recursive walk — `visit_spans(unit, &mut
|span, kind|)` — yields every node's `(span, NodeKind)`. **Folding** consumes the multi-line block-like
kinds; **selection** filters to spans containing the cursor. One walker, two consumers — no parallel
traversals, and the foldable-kind set lives in one place. Pure modules `folding.rs` / `selection.rs`
(+ the shared visitor), thin handlers in `main.rs` — the index_queries/document_symbols split.

**[DECISION] What folds.** *Rec:* the structural multi-line constructs — the `commons`/`context`/`adapter`
container body, type record/sum bodies, capability op lists, service/agent handler lists, provider/op
and fn/handler **block bodies**, `match` (and multi-line arms), `if`, block expressions, and multi-line
record/list literals. Emit a `FoldingRange` **only when `endLine > startLine`** (LSP folds ≥2 lines).
Structural ranges carry no `kind` (default region).

**[DECISION] Comment folding.** *Rec:* **include** multi-line comment runs as `FoldingRangeKind::Comment`,
from a scan of consecutive `Comment` tokens (their spans are in the lexer output the provider already
produces). Small and genuinely useful; if it proves fiddly at the run-grouping boundary, defer it (the
structural folds stand alone).

**[DECISION] Clause-list folding.** *Rec:* leave `given`/`exports`/`consumes` clause lists **unfolded**
in v0.37 (they're usually short; the AST groups them loosely). Revisit only if asked.

**[DECISION] Selection-range chain.** *Rec:* for each requested position, collect every node span that
**contains** the offset, sort by size (innermost first), dedup equal spans, and link them into the
`SelectionRange { range, parent }` chain (root last). Degrades gracefully on a partial recovered AST
(chain what parsed). No "node at offset" walker exists today — this is the one new traversal, templated
on the document-symbols recursion.

**[DECISION] New decision record.** *Rec:* **ADR 0070** — folding + selection from one recovered-AST span
visitor; AST-driven (no tree-sitter); the foldable-kind set; comment-run folding via the token scan.

## Tests

- **`karn-lsp` pure-function**: `folding_ranges(source)` over a fixture with nested structure (context →
  service → handler → match) yields the expected `(startLine, endLine)` set, omits single-line nodes, and
  tags comment runs `Comment`. `selection_ranges(source, pos)` returns the nested chain for a cursor deep
  in an expression (expr → block → handler → service → context), innermost first.
- **Recovery**: a malformed-tail fixture still folds the items that parsed (no panic, partial result).

## Diagnostics / Risks

- **Partial AST mid-edit** — recovery yields `Some(unit)` with the broken item dropped; both providers
  emit what's present. No crash, graceful degradation (same guarantee document-symbols has).
- **Comment-run grouping** — the only fiddly bit (consecutive `--` lines vs. blank-separated runs);
  bounded by the deferral option above.
- **No analysis dependency** — unlike the index features, these don't need a cached round; they parse the
  live document, so they're correct even when the project doesn't check.

## Out of scope (→ later)

- **`foldingRange` `imports`/clause folding** and per-statement folding within blocks.
- **Folding/selection precision for trivia** beyond comment runs (doc-block attachment, etc.).
- Range-formatting-aware folding hints.

## Definition of done

`textDocument/foldingRange` and `textDocument/selectionRange` are declared and answer from the recovered
AST via one shared span visitor: folding collapses the structural multi-line constructs (+ comment runs)
with the `endLine > startLine` rule; selection returns the innermost-first enclosing-span chain per
position. Both degrade gracefully on a partial parse. Pure `folding.rs`/`selection.rs` modules with thin
handlers; `karn-lsp` tests cover the fold set, the selection chain, and the recovery case;
`design/karn-lsp-spec.md` gains §3.20; ADR 0070 added; CI green; changelog; version → 0.37.0. The
implementing PR **deletes this file**.
