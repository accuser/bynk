## What changes

Slice 3 (v0.124) delivered scrutinee-variant completion at an **`<expr> is
<cursor>`** position and explicitly deferred the **`match`-arm** half as a named
fast-follow. This closes it: at an arm-pattern-start inside a `match`, the editor
offers the scrutinee sum type's variants.

```
match order.status {
  <cursor>          -- offers Pending, Shipped, Delivered, тАж
  Shipped => тАж,
  Del<cursor>       -- offers Delivered
}
```

It reuses slice 3's two pieces unchanged тАФ the scrutinee is typed through
`type_receiver` (the `expr_types` value-member path, so it is subject to the same
clean-file ceiling and goes silent, never wrong, on a broken buffer), and the
candidate set is `completion::sum_type_variants` (already `pub(crate)` and shared,
its doc comment already naming `is`/`match`). Only the **arm-position detector**
is new. `bynk-lsp` only; no grammar, checker, or emitter change. Migration is nil.

## Tooling delta

- **`match_scrutinee_offset(text, cursor)`** тАФ a lexical detector mirroring
  slice 3's `is_scrutinee_offset`. It fires only at an **arm-pattern-start**: the
  cursor sits in the innermost still-open `{`, that brace is headed by a
  standalone `match <scrutinee>` (a scrutinee with no intervening block/arrow),
  and the current arm тАФ from the last top-level `,` (or the brace) to the cursor
  тАФ holds nothing but the partial pattern being typed and **no `=>`** (which would
  put the cursor in the arm body). It returns the scrutinee's last-char offset for
  `type_receiver`. Two small helpers back it: `innermost_open_brace` (a `{`-only
  balance scan) and `arm_start_offset` (the last depth-0 `,` after the brace).
- **`match_arm_completions`** тАФ `match_scrutinee_offset` тЖТ `type_receiver` тЖТ
  `sum_type_variants`, the exact shape of `is_pattern_completions`. The shared
  tail is factored into `scrutinee_variant_completions`, which both call.
- **Dispatch (the one subtlety).** Unlike an `is` position, a fresh-line arm
  (`is_keyword_position`) or an after-comma arm (`is_expression_position`)
  *already* yields keyword/local candidates, so gating on `items.is_empty()` (as
  the `is` path does) would never fire. Instead the handler runs
  `match_arm_completions` unconditionally тАФ but the expensive re-analysis is
  gated behind the **cheap lexical `match_scrutinee_offset` check**, so a
  non-match position pays only a string scan тАФ and, when it yields, **prepends**
  the variants to the keyword/local items (variants are the most relevant
  candidate at an arm). The scrutinee is only re-typed when the cursor is
  lexically at a match arm, so ordinary keyword-position completion is unaffected.

## Decision

- **Arm-pattern-start only; bare variant names.** The detector fires at the start
  of an arm's pattern, not inside a nested constructor pattern (`Ok(<cursor>`) or
  a guard тАФ consistent with slice 3's `is` completion, which also offers the
  top-level variants and inserts the bare name (`Ok`, not `Ok(${1:_})`). Nested
  and payload-binding completion is a separate, later concern; the conservative
  fire keeps the detector honest mid-edit (it never guesses inside an arm body).
- **Named sum types only.** `type_receiver` must return `Ty::Named`; a `match`
  over an `Int`/`String` (literal patterns) or a tuple yields nothing, exactly as
  `is` does. No new candidate kinds.
