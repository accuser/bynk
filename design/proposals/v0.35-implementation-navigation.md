# v0.35 — Implementation navigation (capability → providers)

- **Status:** Draft (proposal). **Merging is approval to build.** *(Tooling-queue item 6, the
  implementation half of A‑3 navigation. `textDocument/implementation` on a capability jumps to the
  provider(s) that implement it — the Karn analogue of "go to implementations" on an interface. Like
  v0.34, the link it needs is **already collected** by the index as a `provides Cap` reference whose
  enclosing owner is the provider; this surfaces it via the v0.34 `owner_keys` resolution.)*

## Context

`given Cap` and `provides Cap = Provider` are Karn's interface/impl seam: a capability is the contract,
a provider is the implementation. The natural navigation is `textDocument/implementation` on a
capability → the providers that implement it. The earlier scout flagged the capability→provider mapping
as living in `UnitTable.providers` (computed then discarded), implying a metadata export into the cached
analysis. **A closer look makes that unnecessary**: every `provides Cap` clause already records a
**`Capability` reference whose `owner` is the provider's name** (`validate.rs:613` sets
`set_owner(provider_name)`, `symbols.rs:621` records the capability ref) — and v0.34 just added the
`owner_keys` map that resolves an owner string to its `SymbolKey`. So a provider→capability link is an
index edge of kind `Capability` whose owner resolves to a `Provider`, exactly parallel to v0.34's
`CallEdge`. No `unit_tables` export, no new analysis.

Navigations (1) "`given Cap` use → provider" and (2) "`capability Cap` def → providers" are the **same
query**: a capability *symbol* (whether the cursor sits on its declaration, a `given Cap` use, or a
`provides Cap` use) → all providers implementing it. The third item under the queue heading — a
consumed‑context reference → the context's source — needs a unit→file map (units aren't index symbols)
and is a different, separate increment; **deferred** (see Out of scope).

## Builds on (verified — scout)

- **The link is collected.** `provides Cap = Provider` records a `Capability` ref at the `Cap` name span
  with `owner = provider_name` (`validate.rs:612-621`, `symbols.rs:616-626`). For cross‑context
  `provides`, it's `record_in_unit(...)` into the **defining** unit (`symbols.rs:621-622`), so the
  capability key qualifies correctly across contexts.
- **Both ends are index symbols.** Capabilities (`SymbolKind::Capability`) and providers
  (`SymbolKind::Provider`, keyed by `provider_name`, def at its name span) are both `add_def`'d
  (`symbols.rs:96-103`, `:123-131`). The provider's def site is already in the index — the edge only has
  to *name* it.
- **The owner resolves to a key.** v0.34's `owner_keys: HashMap<(unit, name), SymbolKey>`
  (`index.rs`, populated in `add_def`) resolves `(namespace, provider_name)` to the provider's
  `SymbolKey` — the identical mechanism the `CallEdge` graph uses.
- **The capability symbol is resolved at the cursor** by the existing `symbol_at` / `definition_at`
  (`index_queries.rs:40`) — `given Cap`, `provides Cap`, and the `capability Cap` def all resolve to the
  same `Capability` key.
- **The transport exists.** `goto_implementation(GotoImplementationParams) -> Option<GotoImplementationResponse>`
  (tower‑lsp 0.20) and `implementation_provider: ImplementationProviderCapability::Simple(true)`; the
  response is a `GotoDefinitionResponse` (the same `Location` shape goto‑def returns). No
  `implementation_provider` is declared yet.

## Decisions (open — settle at review)

**[DECISION] How the graph is stored.** *Rec:* an `impls: Vec<ImplEdge>` on `ProjectIndex`, `ImplEdge {
capability: SymbolKey, provider: SymbolKey, site: SiteRef }`, built in `IndexBuilder::build` alongside
the `CallEdge` push: when an edge of kind `Capability` has an `owner` resolving via `owner_keys` to a
**`Provider`** key, record an `ImplEdge`. Pure, in‑memory, mirrors `calls`. (Filtering the existing edge
stream by `owner.kind == Provider` keeps the two graphs' construction symmetric.)

**[DECISION] What `implementation` resolves.** *Rec:* the symbol under the cursor must be a
**`Capability`**; return the **def sites of every provider** implementing it (via `impls` filtered by
`capability == key`, then the provider's `def` from the index). Covers (1) and (2) uniformly. If the
cursor is on a non‑capability symbol, return `None` (the request falls through; goto‑def still works).

**[DECISION] The reverse (provider → capability).** *Rec:* **not** under `implementation` — a provider
*is* an implementation. Provider → its capability is already **goto‑definition** on the `provides Cap`
name (an existing index ref → the capability def). Documented, not re‑plumbed.

**[DECISION] External / adapter providers.** *Rec:* **included** — an `external` provider still has a
Karn `provides Cap = Name { external }` declaration that is an index symbol; navigation lands on **that
declaration**, never the `.binding.ts` (which isn't in the source tree). The provider's `external`-ness
needs no special handling here.

**[DECISION] Ordering & multiplicity.** *Rec:* a capability may have several providers (across
contexts); return all, **sorted by provider def position** (path, span) for a stable list. Zero
providers → `None`.

**[DECISION] New decision record.** *Rec:* **ADR 0068** — implementation navigation from an `ImplEdge`
table built by the v0.34 owner resolution (a `provides`-clause capability ref whose owner is a provider);
capability → providers; reverse is goto‑def; external providers land on the Karn decl; context‑source
`typeDefinition` deferred.

## Spec delta (sketch)

- **`design/karn-lsp-spec.md`** — a "Implementation navigation" capability section (§3.19): `goto_implementation`
  on a capability symbol → its providers, the `ImplEdge` graph from the `provides`‑clause owner, the
  provider→capability reverse served by goto‑def, external providers landing on the Karn declaration, and
  the deferred context‑source/value‑type `typeDefinition`.

## Tests

- **`karnc` index** (the `reference_index`/`call_hierarchy_index` harness): a fixture with a `capability
  Cap` and a `provides Cap = P` asserts one `ImplEdge { capability: Cap, provider: P }`; a capability with
  **two** providers yields two edges sorted by provider def; a capability with none yields no edge. A
  cross‑context `provides` (capability in context A, provider in B) links the A‑unit capability key to the
  B‑unit provider key.
- **`karn-lsp` pure‑function** (the `index_queries` harness): `implementations(index, capability_key)`
  returns the provider def sites, sorted; an unknown / non‑capability key returns empty.
- LSP‑layer position/`Location` conversion stays thin (covered as elsewhere).

## Diagnostics / Risks

- **Owner/key resolution misses** — a `provides` whose owner doesn't resolve (shouldn't happen; providers
  are always `add_def`'d) simply yields no edge, never a wrong one. Same guarantee as the call graph.
- **Clean‑file dependence** — the graph reflects the last clean analysis round (§3.2), like every
  index‑served feature.
- **`implementation` vs `typeDefinition` confusion** — clients bind distinct keys; the spec section names
  exactly which navigation each Karn relationship uses.

## Out of scope (→ later)

- **`textDocument/typeDefinition`** — both the conventional value→type jump *and* the consumed‑context
  (`uses B` / `B.Cap`) → context‑source jump. Context units aren't index symbols (no `SymbolKind` for
  them), so context‑source nav needs a unit→declaring‑file map — a separate, different data source.
- **Provider/adapter → `.binding.ts`** — the TypeScript binding is outside the Karn source tree;
  navigation stops at the Karn declaration.
- Method/op/field implementation lenses; multi‑level impl trees.

## Definition of done

`textDocument/implementation` is declared and, on a capability symbol (its declaration, a `given Cap`
use, or a `provides Cap` use), returns the def site(s) of every provider implementing it, sorted by
provider position. Built on an `ImplEdge` graph assembled by the v0.34 owner resolution — no analysis
export. `karnc` index tests cover the single/multi/zero/cross‑context cases; a `karn-lsp` query test
covers the provider‑def listing; `design/karn-lsp-spec.md` gains §3.19; ADR 0068 added; CI green;
changelog; version → 0.35.0. The implementing PR **deletes this file**.
