# v0.131 ŌĆö CORS for `from http` services: a declarative per-service `cors { }` policy + a synthesised preflight

- **Scope:** a single-increment **language + emitter** change ŌĆö one new service-body
  section (`cors { }`), its checker validation, and the entry-router lowering that
  synthesises an `OPTIONS` preflight and stamps `Access-Control-*` on every response
  of a CORS-enabled `from http` service. No new `HttpResult` variant; the closed
  status/payload registry (ADR 0126, ADR 0143) is untouched.
- **Addresses:** [#396](https://github.com/ŌĆ”/issues/396) ŌĆö a `from http` Worker emits a
  fixed header set (`content-type`/`location`/`cache-control`) and has no `OPTIONS`
  handler, so it **cannot be called cross-origin from a browser**. Deferred from the
  retired in-browser track (ADRs 0136ŌĆō0140); surfaced building the playground
  snippet-share backend (slice 5c, #392), which sidestepped CORS with a Cloudflare
  route (`playground.bynk-lang.org/api/* ŌåÆ the Worker`) rather than letting Bynk
  express it.
- **Realises:** a Bynk HTTP API a browser can consume cross-origin ŌĆö the preflight
  path and the `Access-Control-*` headers ŌĆö declared once per service, not
  hand-plumbed per handler.

## The framing question (and why CORS belongs in Bynk)

The instinct to reach for "just let handlers set arbitrary response headers" is the
wrong one here, and the issue says so: *keep it ergonomic ŌĆ” rather than hand-rolled
header plumbing.* CORS is not presentation (the frontend tier ŌĆö Cloudflare Pages ŌĆö
which Bynk deliberately does not own; cf. ADR 0143's "Bynk serves bytes + a
content-type; it does not template HTML"). CORS is **wire behaviour of an HTTP
service**: which origins may call it, and the preflight contract that gates the call.
The service tier is exactly what Bynk owns, and it already owns the response's
*status vocabulary* (ADR 0126) and its *body shape* (ADR 0143). Access control at the
browser boundary is the same kind of thing ŌĆö a property of the service, expressible
declaratively and lowered by the compiler, the same way `by Visitor` lowers to an
auth seam. So this is a Bynk concern, and the shape should match the rest of the tier:
**one closed, declarative policy the compiler reads**, not an open header escape hatch.

## What exists today (grounded)

- `httpResultToResponse` (`bynk-emit/src/emitter/runtime.ts:446`) builds each
  `Response` with a *fixed* header per variant ŌĆö `content-type` for the JSON/`Raw`
  bodies, `location` for redirects, `cache-control` for the SSE stream. No
  `Access-Control-*`, ever.
- The entry router (`bynk-emit/src/emitter/workers_entry.rs`) collects `http_routes`,
  matches `method` + `path` per request (`emit_http_route_dispatch`, line 448), and
  **falls through to `404` when nothing matches** (line 264). A browser preflight ŌĆö
  `OPTIONS /links` ŌĆö therefore gets a `404`, and the real cross-origin `POST` response
  carries no `Access-Control-Allow-Origin`, so the browser blocks it.
- A service (`ServiceDecl`, `bynk-syntax/src/ast.rs:554`) carries a `protocol`
  (`ServiceProtocol::Http`) and its handlers ŌĆö and **no config surface**. There is a
  precedent for compile-time metadata attached to a declaration: `store`-field
  annotations (`@ttl(ŌĆ”)`, `@indexed(ŌĆ”)` ŌĆö ADR 0111), a closed registry the checker
  validates. CORS wants a small *structured* record, not a flat annotation, so this
  proposes a section, not an annotation (DECISION B).

## The surface

A `cors { }` section in the `from http` service body, in header position ŌĆö before the
routes, mirroring the agent phase order (`store` ŌåÆ `invariants` ŌåÆ handlers,
`AgentDecl`, `ast.rs:592`). It reads like a record literal, the shape already familiar
from request/response types:

```
service api from http {
  cors {
    origins:     ["https://playground.bynk-lang.org"],
    credentials: false,
    maxAge:      1.hour,
  }

  on POST("/links") by Visitor (body: CreateRequest) -> Effect[HttpResult[CreatedView]] given Random, Kv {
    ŌĆ”
  }

  on GET("/links/:code") by Visitor (code: Slug) -> Effect[HttpResult[Url]] given Kv { ŌĆ” }
}
```

From this the compiler synthesises, for that service:

1. **A preflight branch** ŌĆö `OPTIONS` against any of the service's route paths returns
   `204` with `Access-Control-Allow-Origin` (the matched origin), `-Allow-Methods`
   (**derived from the routes** ŌĆö DECISION C), `-Allow-Headers`, `-Max-Age`, and, when
   `credentials`, `-Allow-Credentials: true`. Answered **before** the `by Actor` auth
   seam (DECISION E).
2. **Response stamping** ŌĆö every real response from a route in this service gets
   `Access-Control-Allow-Origin` (+ `Vary: Origin`, + `-Allow-Credentials` when set)
   appended, uniformly, regardless of variant (`Ok`/`Raw`/redirect/error/stream).

A service with **no** `cors { }` block is byte-for-byte unchanged ŌĆö this is opt-in and
fully additive.

## Decisions

**[DECISION A] A declarative per-service CORS policy ŌĆö *not* a general response-header
surface. (Recommended, load-bearing.)** The alternative ŌĆö a per-handler
`headers { ŌĆ” }` map or a `withHeader` on `HttpResult` ŌĆö is the "hand-rolled plumbing"
the issue rejects, and it would crack open the closed `HttpVariantPayload` registry
(ADR 0143 D1: "one uniform shape ŌĆ” one-variant-one-shape model") that gives completion
and the runtime their single source of truth. A focused `cors` policy stays closed,
keeps the ergonomics ("say which origins; the compiler does the rest"), and covers the
concrete need in #396 and every future Bynk HTTP API. A general header surface remains
a **separate, larger, re-openable** decision; this increment deliberately does not open
it. *Recommend: the policy.*

**[DECISION B] The policy is a `cors { }` section in the service body, not an
annotation and not a `from http(ŌĆ”)` clause argument.** CORS config is a small
structured record (origins list, flags, a duration); a flat `@cors(...)` annotation
(the `@ttl` shape) reads badly for a multi-field record, and stuffing it into the
`from http(ŌĆ”)` header conflates *protocol* with *policy* (the `from` clause carries the
binding ŌĆö `queue("name")`, `WebSocket(in:,out:)` ŌĆö not access config). A named section
in header position matches the agent-phase precedent and leaves room for sibling
service-policy sections later (rate-limit, cache) without another grammar fork.

**[DECISION C] `Access-Control-Allow-Methods` is *derived from the routes*, not
declared.** The service already enumerates its methods (`on POST`, `on GET`, ŌĆ”); making
the author restate them in `cors` is a second source of truth that will drift. The
preflight advertises the union of the methods declared on the matched path (fallback:
the service's method set) plus `OPTIONS`. This is the most Bynk part of the design ŌĆö
the same "the registry is the enumeration" move as ADR 0126. *A route added later is
automatically preflightable; nothing to keep in sync.*

**[DECISION D] Origin model: a static allowlist of exact string literals, plus a `"*"`
wildcard ŌĆö reflect the matched origin with `Vary: Origin`; no dynamic predicate in
v1.** `origins: ["https://a", "https://b"]` compares the request `Origin` header
against the list and **reflects the matched value** (never echoes an unvalidated
origin), adding `Vary: Origin` so a shared cache does not serve one origin's ACAO to
another. `origins: ["*"]` emits a literal `*`. A no-match omits `Access-Control-Allow-
Origin` entirely (the browser blocks ŌĆö fail-closed, the same posture as the auth seam).
A *computed* origin policy (a Bynk predicate over the origin) is a named follow-on; the
static list covers #392 and the overwhelming common case.

**[DECISION E] The synthesised preflight is answered *before* the `by Actor` / Bearer
auth seam, unauthenticated.** This is a correctness fork, not a convenience one: a CORS
preflight is by spec credential-less and method `OPTIONS` ŌĆö the browser sends it before
it will attach `Authorization` or cookies. If the preflight hit the `by Visitor` /
Bearer seam (`emit_http_route_dispatch`'s `401` paths, `workers_entry.rs:512`/`537`) it
would `401` and the real request would never fire. So the preflight is a dedicated
branch **ahead of** the route dispatch, returning `204` + the `Access-Control-*`
headers with no handler invocation. The real (non-`OPTIONS`) request still runs the
full auth seam unchanged ŌĆö preflight approval is not request authorisation.

**[DECISION F] `credentials: true` with `origins: ["*"]` is a compile-time error.** The
Fetch spec forbids `Access-Control-Allow-Credentials: true` alongside a wildcard ACAO
(the browser rejects it at runtime ŌĆö a silent, baffling failure). Bynk catches it at
the boundary: a checker diagnostic (`bynk.http.cors_wildcard_credentials`) rather than a
runtime surprise. This is the fail-at-compile posture the language takes everywhere
else.

**[DECISION G] `Access-Control-Allow-Headers` defaults smartly, overridable.** Default
= `content-type` (the JSON body always needs it) ŌĆö plus `Authorization` **derived**
when the service has any Bearer-seam route (`route.bearer`, already tracked in
`workers_entry.rs:575`), the same single-source-of-truth move as DECISION C. An
explicit `headers: ["content-type", "x-api-key"]` overrides. `maxAge` defaults to
omitted (browser default) and accepts a `Duration` (`1.hour`) lowered to
`Access-Control-Max-Age` seconds. `credentials` defaults `false`.

**[DECISION H] Additive; minor bump v0.131. New ADR 0158.** No variant, shape, or
existing behaviour changes; a service without `cors { }` emits identical output.
`HttpResult` and its registries (`HTTP_VARIANTS`, `HTTP_STATUS`) are **not** touched ŌĆö
this is a service-policy layer *around* the result lowering, not a change to it. The
proposal's decisions land as ADR 0158 on implementation.

## The deltas (concretely)

**Grammar / AST (`bynk-syntax`).**
- `ServiceDecl` (`ast.rs:554`) gains `cors: Option<CorsPolicy>`.
- New `CorsPolicy { origins: Vec<String>, allow_headers: Option<Vec<String>>,
  credentials: bool, max_age: Option<Duration>, span }`. `allow_methods` is **not**
  stored ŌĆö it is derived at emit time (DECISION C).
- Parser: a `cors { ŌĆ” }` section in the service body, fields in record form. The
  grammar accepts any field name (unknown ŌåÆ checker diagnostic, per the `@`-annotation
  precedent).

**Checker (`bynk-check`).**
- `origins` non-empty; each a string literal (a compile-time constant, like a route
  path). `credentials` a bool literal; `maxAge` a `Duration` literal.
- **DECISION F** diagnostic: `credentials: true` + `"*"` in `origins`.
- `cors { }` is only legal on `ServiceProtocol::Http` (a diagnostic on `Call`/`Cron`/
  `Queue`/`WebSocket` ŌĆö CORS is meaningless off the browser-facing HTTP surface).

**Emitter (`bynk-emit`).**
- `runtime.ts`: two helpers ŌĆö `corsPreflightResponse(policy, requestOrigin)` ŌåÆ the
  `204` + `Access-Control-*` `Response`; and `applyCors(response, policy,
  requestOrigin)` ŌåÆ mutates `response.headers` (Response headers are mutable
  post-construction, so this works uniformly for the `Ok`/`Raw`/redirect/stream/error
  responses `httpResultToResponse` returns). `CorsPolicy` gets a runtime-side
  interface. The bundler regenerates the shipped `runtime.ts` (drift-checked), same as
  ADR 0143.
- `workers_entry.rs`: group `http_routes` by service; for each CORS-enabled service,
  (1) emit an `OPTIONS`-and-path-matches preflight branch **before** the route
  dispatch loop (line 260), computing allow-methods from that service's routes; (2)
  wrap the `httpResultToResponse(...)` return (line 594) in `applyCors(...)` for that
  service's routes.
- The **bundle/test entry** (the non-Workers target) gets the parallel preflight +
  stamping so `bynkc test` exercises CORS behaviour without a Worker runtime.

## Risks & mitigations

- **`credentials` + `*` foot-gun** ŌåÆ compile error (DECISION F), not a runtime mystery.
- **Cache poisoning across origins** ŌåÆ `Vary: Origin` whenever the origin is reflected
  (DECISION D); the literal-`*` path needs no `Vary`.
- **Preflight bypassing auth "feels wrong"** ŌåÆ it is spec-correct and intentional
  (DECISION E); documented loudly in the guide. The real request is fully
  authenticated; the preflight authorises nothing.
- **Header stamping on already-built responses** ŌåÆ `Response.headers` is mutable;
  `applyCors` sets, does not reconstruct, so streaming (`ReadableStream`) and `Raw`
  bodies are untouched. Covered by a fixture per variant family.
- **Same path, mixed CORS across services** (two services declaring the same route
  path, one with `cors`) ŌĆö out of scope: route paths are already expected unique across
  a context's HTTP surface; note it, don't build for it.
