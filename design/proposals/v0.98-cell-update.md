# v0.98 — `Cell[T].update(fn)`: the method-shaped read-modify-write

- **Scope:** a single-increment language proposal (storage track). A **checker +
  emitter** change; no grammar change (the surface is an ordinary `MethodCall`).
  Version → v0.98; ADR **0125**.
- **Realises:** the already-specified Cell operation. `design/bynk-type-system.md`
  §2.7.3 lists `Cell[T].update(f) : Effect[Unit]` (`f: (T) -> T`, line 1186), and
  line 1163 frames it as method-shaped — *"Other Cell operations (`cell.update(fn)`,
  `cell.swap(v)` if added) require explicit `<-`"*; the design notes §10 name it core
  "apprentice-level" vocabulary alongside `:=`. **Caveat:** §2.7.3 is headed
  *"Operations and their types — Open"* (line 1179) and is listed under continued
  refinement (line 1761), so the signature is *written down but not yet normative*.
  This proposal therefore does two things: wire the operation through (the bulk of
  the work), **and** settle §2.7.3 from Open to normative for `Cell.update` (a real,
  if small, spec decision — see DECISION F). It is not pure wiring.

## The gap (against the current code)

`Cell.update` is **half-built**: the language already *assumes* it exists, but no
code resolves it.

1. **No dispatch branch.** The `MethodCall` dispatcher in
   `bynk-check/src/checker.rs` (~line 1800) routes a store-field receiver by
   provenance — there are branches for `store_maps`, `store_sets`, `store_caches`,
   and `store_logs`, each delegating to a `check_store_*_op` helper. There is **no
   `store_cells` branch**. So `n.update(fn)` falls through to the generic
   `check_method_call`, which knows nothing about cells and reports a misleading
   "no such method" rather than a storage-op diagnostic.
2. **Dangling guidance.** The `:=` checker (`checker.rs` ~line 1428) deliberately
   rejects a self-referencing write — `n := n + 1` raises
   `bynk.cell.self_reference` with the note *"use `<cell>.update(fn)` for a
   read-modify-write so the dependency on the prior value is explicit"*. That note
   points at a method that was never implemented; today the only way through is the
   `let cur = n; n := cur + 1` workaround the rule was meant to retire.
3. **No lowering.** `lower_method_call` (`bynk-emit/src/emitter/lower.rs` ~line
   706) has the matching map/set/cache/log branches but no cell branch, so even a
   well-typed `update` would not emit.

Net: the diagnostic steers users to `update`, the spec defines `update`, and
nothing implements `update`. This proposal closes that loop.

## What this adds (concretely)

1. **Checker dispatch** (`checker.rs`): a `store_cells` branch in the `MethodCall`
   dispatcher, mirroring the others —

   ```rust
   else if let ExprKind::Ident(id) = &receiver.kind
       && let Some(t) = ctx.store_cells.get(&id.name).cloned()
   {
       check_store_cell_op(method, args, &t, expr.span, ctx)
   }
   ```

2. **`check_store_cell_op`** (`bynk-check/src/checker/calls.rs`): a near-verbatim
   sibling of `check_store_set_op`. One op — `update(f)` — typed `f: (T) -> T`
   returning `Effect[Unit]`; arity and argument-type checks identical to the set/map
   helpers; an unknown op raises `bynk.store.unknown_op` ("a `Cell` store field has
   no operation `<x>` — expected `update`") with a note that a cell is read by its
   bare name and written with `:=`.

3. **Emitter lowering** (`lower.rs`): a cell-receiver branch in `lower_method_call`
   keyed on the cell set already carried in `agent_store_state.1`, lowering
   `n.update(f)` to a **staged read-modify-write** over the working state:

   ```
   (() => { __state.n = (<f>)(__state.n); return undefined; })()
   ```

   This is `Map.update`'s lowering **minus the key-absent guard** — a cell is
   always present, so there is no fault path. The mutation is synchronous against
   the in-memory working state (read-your-writes within the handler); the single
   end-of-handler `commitState` flush runs the invariant gate before the durable
   write, exactly as `:=` already does. No new runtime helper.

The canonical handler becomes:

```
on call increment() -> Effect[Int] {
  <- n.update((c) => c + 1)
  Effect.pure(n)
}
```

— await the write, then the bare `n` reads back the new value and
`Effect.pure(n)` returns it. This works for one specific reason: the bare-name read
desugars to a read of `__state.n`, the *same* staged working-state slot the `update`
lowering just mutated synchronously — the identical mechanism `:=` relies on. That
is why `increment()` returns the incremented value rather than the stale one; there
is no separate read path to fall out of sync.

## Decisions

**[DECISION A] `update` returns `Effect[Unit]`, not `Effect[T]`.** It mutates the
cell; it does **not** thread the new value out. This keeps parity with
`Map.update` / `Cache.update` and preserves the rule that *reading* a cell is
always the bare-name sugar — a value-returning `update` would create a second,
method-shaped read path. Read-modify-write-**and-return** is therefore the
two-line `<- n.update(…)` then `Effect.pure(n)`, and that explicitness is the
point. *(Settled — this was the explicit call: `increment()` returning `n` is the
intended behaviour, not `update` handing the value back.)*

> **Notation:** this proposal writes the spec's type as `Effect[Unit]` to match
> §2.7.3 (line 1186). Where it describes checker output or surface code it writes
> `Effect[()]` — `Ty::Unit` displays as `()` (`checker.rs:138`) and that is the
> spelling users type (`-> Effect[()]`). Same type; the two spellings are
> deliberate — spec-notation versus surface-form — not drift.

**[DECISION B] `update` is the *only* method-shaped Cell op.** §2.7.3 also lists
`read()` and `write(v)` as the underlying operations, but in the surface they are
sugar — the bare name reads, `:=` writes. We do **not** expose `read`/`write` as
callable methods: two ways to do the identical thing invites style drift and
undercuts the "cells are not first-class" framing. They stay illustrative in the
spec, named as the desugaring targets, not as surface vocabulary. The spec already
draws exactly this line — `design/bynk-type-system.md` line 1163 singles out
`cell.update(fn)` (and a hypothetical `cell.swap(v)`) as the operations that *are*
method-shaped and require explicit `<-`, in contrast to the sugared read/write.

**[DECISION C] No `given Clock` requirement.** Unlike `Cache`, a cell update reads
no clock — it is a pure staged mutation. No capability gate.

**[DECISION D] The `:=` self-reference diagnostic stays — its suggested fix now
exists.** `n := n + 1` continues to raise `bynk.cell.self_reference` pointing at
`update`; the only change is that following the advice now compiles. The note
wording is reviewed for accuracy but not relaxed — the rule (make the prior-value
dependency visible and retry-safe) is unchanged.

**[DECISION E] The lambda must be pure (`(T) -> T`, no `<-` inside).** Enforced for
free: the parameter type is a non-effectful `Ty::Fn`, so any effectful body fails
the existing function-type check, identically to `Map.update`'s combiner. This
pre-empts the obvious "what if the lambda reads another cell?" question — a
bare-name cell read is *itself* effectful sugar (`<- other.read()`), so a combiner
that touches another cell is effectful and is rejected at the same gate. A
`(T) -> U` mismatch surfaces as `bynk.types.argument_mismatch`.

**[DECISION F] Version — minor v0.98; ADR 0125.** A language increment (the
surface gains a working operation) *and* a spec settlement (§2.7.3 moves from Open
to normative for `Cell.update`), so a version bump and a decision record per the
proposals lifecycle.

## Risks & mitigations

- **Divergence from the `Map`/`Cache` helpers.** Mitigated by writing
  `check_store_cell_op` as a structural copy of `check_store_set_op` (same arity /
  argument-mismatch machinery) and the lowering as `Map.update` minus the guard —
  no novel control flow to maintain.
- **Atomicity assumptions.** `update` mutates the same staged `__state` that `:=`
  writes, so it inherits read-your-writes and the end-of-handler commit gate
  unchanged. A fault between an `update` and commit persists nothing — same as
  today. Covered by extending the `store_behaviour.rs` runtime test.
- **Misuse diagnostics.** Wrong arity, a non-function argument, and a
  `(T) -> U` lambda must each give a clear error, not a generic fall-through —
  asserted by a negative fixture.

## Docs & tests

- **Design spec:** settle `Cell[T].update` in `design/bynk-type-system.md` §2.7.3
  — move the signature out of the *"Operations and their types — Open"* framing to
  normative (and drop it from the line-1761 continued-refinement list). This is the
  design spec, **not** the published book — the book has no §2.7.3.
- **Book:** the storage guide gains a "modify a cell in place" entry distinguishing
  `:=` (unconditional write) from `update` (read-modify-write), with the *why*
  (visible prior-value dependency); the Cell reference page lists the op; a
  `reference/changelog.md` row; the currency banner advanced to v0.98. The counter
  is the shared worked example (`{{#include}}` from a compiled fixture so it can't rot).
- **Tests:**
  - **Positive e2e fixture** (mirroring `224_store_map_agent`): a `Counter` agent
    whose `increment` uses `<- n.update((c) => c + 1)`; golden blessed via
    `BYNK_BLESS=1`.
  - **Runtime** (`store_behaviour.rs`): an `update` persists across invocations and
    a read after `update` in the same handler sees the new value.
  - **Negative fixture**: `n.foo()` (unknown op → `bynk.store.unknown_op`),
    `n.update()` (arity), and `n.update((c) => "x")` (argument mismatch).
  - **Checker** (`store_checker.rs`): `update` types as `Effect[()]`; `n := n + 1`
    still raises `bynk.cell.self_reference`.

## Done when

  types as `Effect[()]` and an unknown Cell op gives `bynk.store.unknown_op`.
- The emitter lowers `n.update(f)` to the staged read-modify-write over
  `__state.<cell>`; positive golden blessed and green.
- Runtime, negative, and checker tests green; existing suites unaffected.

- On merge: delete this proposal. (Per DECISION B, surfacing `read`/`write` as
  methods is explicitly *not* a follow-on — they remain sugar.)
