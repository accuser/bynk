# v0.73 — Bynk-native debug values: read your data in Bynk's vocabulary (debugging track, slice 5)

- **Track:** [debugging](../tracks/debugging.md), slice 5 — the first increment of
  **Phase 2** (the bynk-native semantic layer). Phase 1 (slices 0–4) made *control*
  read in Bynk: breakpoints, stepping, and the stack land on `.bynk` lines. Phase 2
  makes *values* read in Bynk: `Ok(42)`, not `{tag: "Ok", value: 42}`.
- **Realises ADR:** [0104](../decisions/0104-debug-launch-model.md) D1 — the one place
  the deferred semantic layer earns its keep is "a custom adapter **or a
  variable-formatter contribution**… showing `Result`/`Option`/sum values unwrapped."
  This slice takes the **variable-formatter** path (the cheap half of that disjunction).
- **Posture:** tooling (VS Code extension + possibly a small runtime touch). No language
  change. Builds directly on slice 4's `DebugConfigurationProvider`.

## The gap

A `.bynk` breakpoint now binds and pauses on the exact statement, under Node and
workerd (Phase 1). But when you *inspect* a value, you see its emitted TypeScript shape,
not its Bynk shape. Bynk's algebraic values lower to **tagged objects** (`bynk-emit`'s
runtime):

| Bynk value | Runtime shape (what the debugger shows today) | What we want |
|---|---|---|
| `Ok(42)` | `{tag: "Ok", value: 42}` | `Ok(42)` |
| `Err(e)` | `{tag: "Err", error: {...}}` | `Err(ValidationError {...})` |
| `Some("hi")` / `None` | `{tag: "Some", value: "hi"}` / `{tag: "None"}` | `Some("hi")` / `None` |
| a sum variant | `{tag: "NotFound"}` / `{tag: "BadRequest", message: "…"}` | `NotFound` / `BadRequest("…")` |

The shape is *legible* (the `tag` discriminant is right there) but it isn't Bynk. Every
hover, the Variables pane, and Watch expressions read in the lowering's vocabulary. This
is the "variable inspection… is the known rough edge that Phase 2's semantic layer
addresses" that ADR 0104 D1 explicitly deferred to here.

## The mechanism — and the one genuine uncertainty (spike-first)

`vscode-js-debug` accepts a **`customDescriptionGenerator`** in the launch/attach config:
a function (as a string) evaluated *in the debuggee* for each object, with `this` bound
to the object and `defaultValue` the description js-debug would otherwise show. It returns
the one-line description the debugger renders. (There is a sibling
`customPropertiesGenerator` for the *expanded children*, but it is documented as
**deprecated / a temporary implementation** — see Decisions.)

The fit is almost too neat: slice 4's provider **already builds the `pwa-node` attach
config**, so this is a *config addition*, not new machinery — no custom Debug Adapter
(ADR 0104 D1's expensive alternative stays unbuilt). The generator recognises Bynk's
uniform tagged-value shape (`typeof this.tag === "string"`) and renders Bynk constructor
syntax: `Ok(42)`, `Err(…)`, `Some(x)`, `None`, `BadRequest("…")`, `NotFound`.

What's **unconfirmed** — and what the slice spikes before building anything:

1. **Does `customDescriptionGenerator` render legibly, and how deep does it nest?** A
   single level (`Ok(…)` with the inner value on expand) is the floor; ideally the
   generator renders one or two levels inline (`Ok(Some(42))`). The generator is a
   self-contained string with only `this`/`defaultValue` in scope — recursion (for
   nested ADTs) must fit in that box (a self-referential function expression) or be
   accepted as shallow for v1.
2. **Does it reach the workerd debuggee?** It worked-or-not is a *wrangler/CDP* question,
   exactly as slice 4 spiked the attach. The generator must run under both Node and
   `workerd`; the spike confirms both (Node is the headline; workerd is the dev path).
3. **Structural sniffing vs a runtime brand.** `this.tag` is cheap but could false-positive
   on user records that happen to carry a `tag` field. The alternative — a non-enumerable
   brand on the runtime's ADT objects (`Symbol.for("bynk.adt")`) — makes recognition exact
   at the cost of a small `bynk-emit` runtime touch. The spike weighs whether structural
   sniffing is good enough or the brand is worth it.

No production formatter code until the spike shows `Ok(42)` in a real Variables pane.

## What this slice does

1. **A `customDescriptionGenerator`** for Bynk values: a compact, total, side-effect-free
   function that maps a tagged value to its Bynk constructor rendering and returns
   `defaultValue` for everything else. Authored as a TS source the extension bundles and
   injects as a string (so it's testable and not a magic blob in a config literal).
2. **Provider wiring.** `src/debug.ts` adds the generator to the resolved attach config
   for both `mode: test` and `mode: dev`. A `bynk.debug.semanticValues` setting (default
   **on**) lets a user fall back to the raw shape.
3. **A runtime brand (iff the spike calls for it).** If structural sniffing proves
   fragile, `bynk-emit`'s runtime tags ADT objects with a non-enumerable
   `Symbol.for("bynk.adt")`; the generator keys on it. Decided by the spike, not upfront.
4. **A VS Code integration test** extending slice 4's harness: at a breakpoint, evaluate a
   `Result`/`Option`/sum value and assert the DAP `variables`/`evaluate` description reads
   `Ok(42)` (not `{tag:…}`). Node path harness-provisioned (DECISION D); workerd opt-in.
5. **Docs:** a short section in the "Debug in VS Code" guide (values read in Bynk; the
   toggle); changelog row; the bump.

## Decisions

**[DECISION A — recommend, spike-gated] The cheap path: `customDescriptionGenerator`, no
custom adapter.** ADR 0104 D1 offered "a custom adapter *or* a variable-formatter
contribution." Take the formatter: it's a config addition to the attach slice 4 already
builds, it carries zero session-proxying risk, and it delivers the headline win (values in
Bynk vocabulary). The custom-adapter path — full control over scopes/stack/properties, and
the only way to do some Phase-2 asks (below) — stays the **documented fallback** if the
formatter proves too limited.

**[DECISION B] Description first; properties (hiding `tag`) is a stretch, gated on the
deprecation.** `customDescriptionGenerator` (the one-liner) is stable and is the headline.
`customPropertiesGenerator` (which could drop the `tag` child and show a `Result` as just
its payload) is documented **deprecated/temporary** — adopt it only if the spike shows
it's reliable on the pinned js-debug, else leave the expanded children as the raw shape
(the description already carries the meaning). Don't build v1 on a deprecated API.

**[DECISION C] Scope Phase 2's *value* layer here; name the rest as follow-ons.** Phase 2
as framed in the track also wants **contexts/actors as scopes** and **capability calls
legible in the stack** — both of which likely *need* the custom adapter (they reshape
scopes/stackTrace, which a description generator can't touch). And the **lowered-temp
noise** in the Variables pane (compiler temps from `?`/`match` lowering) is a *different*
mechanism again (emitter naming or scope filtering). This slice ships the value
descriptions — the cheapest, highest-frequency win — and names those three as the next
Phase-2 increments. If they land, Phase 2 likely graduates to **its own track** (as the
debugging track foresaw); slice 5 is its on-ramp.

**[DECISION D — sign-off] Recognition: start structural, brand iff needed.** Prefer the
zero-runtime-change structural `this.tag` check; adopt the `Symbol.for("bynk.adt")` brand
only if the spike shows false positives or fragility. Keeps the runtime untouched unless
the data says otherwise.

**[DECISION E] Version — minor v0.73.** A new extension capability, no language change;
the extension version tracks the workspace bump.

## Risks & mitigations

- **The generator runs on *every* object in the debuggee.** It must be total, fast, and
  side-effect-free, returning `defaultValue` immediately for non-tagged values. Mitigated
  by a tight guard (`this && typeof this.tag === "string"`) and a unit-style assertion on
  the function in isolation before it's wired in.
- **`customPropertiesGenerator` is deprecated.** Don't depend on it (DECISION B); the
  description path is the stable surface. Pin `@vscode/test-electron` so the config shape
  can't drift unobserved (as slice 4 does).
- **Nested ADTs render shallow.** Acceptable for v1 (inner value expands one click away);
  the spike measures how deep the generator can reasonably go.
- **workerd may not honour the generator.** Spiked first, gated like slice 4's attach; the
  Node path is independent and is the headline.
- **False positives on user `tag` fields.** DECISION D's brand is the escape hatch.
