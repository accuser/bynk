- **Status:** Draft (proposal). **Merging is approval to build.** *(Refactor track item 4 — the
  `checker.rs` counterpart to the `project.rs` split. **Sliceable** (pins → banner-and-split →
  sub-struct). Independent of the `project.rs` chain; its prerequisite is **checker-helper
  characterization pins** landed first (the deferred half of item 11). Behaviour-preserving — checker
  diagnostics for every fixture are byte-identical.)*

## Context — 6.7k lines, no signposts, one 20-field god-context

- `checker.rs` is **6,696 lines / 106 functions with 0 section banners** (verified post-`29.11`) —
  navigation is by scroll-and-grep. No structural seams are marked.
- `Ctx<'a>` (`:837`) is a **20-field** mutable context threaded through the checker. **Six** of those
  fields are capability-bookkeeping (verified, `:866`–`:886`): `capabilities`,
  `declared_capabilities`, `given_remaining`, `given_used`, `given_entries`, `given_anchor` — a
  cohesive cluster (the `given`-clause lifecycle + dispatch) diffused into the top-level struct.

## Builds on (verified)

- **`Ctx` field inventory** (`:837`–`:899`): `input`, `expr_types`, `errors`, `refs`, `hints`,
  `scopes`, `return_ty`, `return_ty_span`, `effectful`, `agent_state_ty`, `commit_seen`, **the six
  capability fields**, `in_test_body`, `test_services`, `type_vars`. The six capability fields are
  used together at capability-dispatch and `given`-clause sites — a natural `CapabilityCtx` sub-struct.
- **The split is genuinely mechanical — checker.rs is free functions, not an interwoven `impl`.**
  Verified: of the 106 functions, **105 are free functions** taking `Ctx` by parameter (only 9 `&self`
  methods exist, in small `impl Ty`/`impl ConstLit`/`impl Ctx` blocks). So, exactly like `project.rs`
  (`29.2`), the functions **move to submodules behind a `checker/mod.rs` facade** via the flat
  `use super::*` re-export pattern — no `impl`-block-splitting gymnastics. The thematic seams are also
  **richer than `calls`/`refinements`/`match`**: the names cluster into calls (`check_call`/`check_method_call`/
  `check_cross_context_call`/…), refinements (`check_refinement`/`check_*_refinement_consistency`/
  `admit_refined_literal`/`check_is`), **kernels** (`check_{list,map,numeric,string,option,result}_kernel_method`
  + the statics), expressions/match (`check_match`/`check_if`/`check_record`/`check_ok`/…), and bodies/decls
  (`check_handler_body`/`check_fn`/`check_lambda`/`check_mock`). The "let the banners drive it" decision
  below stands — these are the likely banner groups.
- **`checker.rs` has only ~4 `#[test]`** (the queue's coverage note) — so, like the `project.rs` split,
  the pure helpers it moves must be **pinned first**. Candidate pures: type-compatibility/match helpers
  (`type_refs_match`-style), refinement-narrowing helpers, the `is`-narrowing logic — the analogues of
  the `29.1` targets, on the checker side.

## Settled

- **▸ Behaviour-preserving throughout.** The split is a verbatim move (mechanical `use`/visibility
  edits only); the `CapabilityCtx` change is a field-access rewrite (`cx.given_remaining` →
  `cx.caps.given_remaining`). No checking logic changes; diagnostics are byte-identical.
- **▸ Pins precede the move.** Per the track ADR, checker-helper characterization tests land **ahead
  of** the split — the deferred second half of item 11.

## Decisions (open — settle at review)

**[DECISION] Slice the increment into three PRs.** *Rec:*
- **Slice 0 — checker-helper pins.** Direct `#[cfg(test)]` tests for the pure checker helpers about to
  move (type-match, refinement-narrowing, `is`-narrowing), green on current `main`. Lands first, on the
  `29.1` precede-the-move rule.
- **Slice 1 — banner + split.** Add section banners, then split into `checker/{calls,refinements,
  match}.rs` (+ a `mod.rs` facade), verbatim moves. The big mechanical PR.
- **Slice 2 — `CapabilityCtx` sub-struct.** Group the six capability fields into
  `Ctx.caps: CapabilityCtx` and rewrite accessors. The smallest, most surgical PR.

**[DECISION] Submodule boundaries.** *Rec:* start from the queue's `calls` / `refinements` / `match`
seam, but **let the banners drive it** — add banners in slice 1 first, group the 106 functions under
them, and split along whatever the banners reveal as cohesive (likely also `capabilities`, `types`,
`patterns`). Don't pre-commit to exactly three files; commit to "single-purpose submodules behind a
facade."

**[DECISION] `CapabilityCtx` shape.** *Rec:* a plain sub-struct owning the six fields with the same
types; `Ctx` holds it by value. Methods that mutate the `given` lifecycle (`mark_used`,
`remaining`, …) can move onto `CapabilityCtx` opportunistically, but the **minimum** is the field
grouping — keep the behaviour-preserving core small, add methods only where they remove duplication.

## Spec delta

**None.** No language surface, no `docs/src/spec/` chapter — module reorganisation + a struct regroup.

## Tests

- **Slice 0 adds** the checker-helper pins (the increment's safety net).
- **Slices 1–2 add no fixtures** — existing checker golden/diagnostic fixtures plus the slice-0 pins
  must pass **unedited**; that invariance is the proof for both the split and the sub-struct.

## Diagnostics / Risks

- **Coverage thinness is the real risk** — with ~4 end-to-end tests, an internal change is under-pinned
  until slice 0 lands. **Slice 0 is therefore a hard gate**, not optional: do not split or regroup
  before the pure helpers are pinned.
- **`pub(crate)` cascade from the split** — cross-submodule calls force visibility widening; widen to
  `pub(crate)`, never `pub`, and treat a transitive cascade as a coupling signal worth recording (it
  informs whether the chosen seam is right).
- **`CapabilityCtx` borrow shape** — grouping six fields under one sub-struct can change borrow
  granularity (a function that borrowed `given_used` mutably while reading `capabilities` now borrows
  `cx.caps` for both). Resolve by borrowing the specific fields, or split-borrowing the sub-struct;
  the failure mode is non-compilation, not silent drift.

## Out of scope (→ later / independent)

- **Decomposing individual giant checker functions** (beyond the file split) — the split + banners make
  them navigable; carving specific functions (à la `lower_expr`) is separate, scheduled only if a given
  function warrants it.
- **`Ctx`'s non-capability fields** — left as-is; only the cohesive capability cluster is grouped. Don't
  over-aggregate fields that aren't used together.
