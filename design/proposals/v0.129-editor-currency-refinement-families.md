## What changes

Slice 6 (v0.127) delivered the **providers** half of #259 (`N providers` on a
capability) and parked the **refinements** half, because — unlike providers,
already an index edge — the index had no relation linking a refined type to its
base. This adds that relation and surfaces it as a codelens.

#259 asks for **"refinement families"**: every refined/opaque type over a builtin
base `T` — each `type Email = String where …`, `type UserId = String`,
`type Slug = String where …` — as a flat *"what refines `T`"* query, deliberately
**not** a `typeHierarchy` tree (the base `String` is a builtin, not a navigable
symbol). Delivered here as a codelens:

```
type Email  = String where NonEmpty     -- 3 refinements of String
type UserId = String                    -- 3 refinements of String
type Slug   = String where Matches(…)   -- 3 refinements of String
type Age    = Int where InRange(0, 130) -- (no lens — a family of one)
```

The lens sits on each refined/opaque type declaration, titled **`N refinements of
<Base>`**, and peeks the whole family on click (the same `showReferences` action
the reference and provider lenses use). `bynk-lsp` + a small `bynk-check` index
edge + its one population line in `bynk-emit`; no grammar, checker-semantics, or
emitter-output change. Migration is nil.

## Tooling delta

- **The index edge (`bynk-check`).** A new `index::RefineEdge { base: BaseType,
  ty: SymbolKey }` and `ProjectIndex.refinements: Vec<RefineEdge>`, with two
  accessors — `refined_base(key)` (the builtin a type refines, if any) and
  `refinements_over(base)` (every type over a base) — mirroring `ImplEdge` /
  `impls_of`. Unlike `ImplEdge` (resolved from ref-edges during `build`), a
  `RefineEdge` needs no resolution: the base is a builtin already in hand and the
  `ty` is the def being recorded, so it is captured straight at the type-def walk.
- **Population (`bynk-emit`).** At the type-def site in `project/symbols.rs` —
  where `symbol_modifiers` already destructures `TypeBody::Refined`/`Opaque` and
  *discards* the `base` — a `builder.add_refinement(unit, name, base)` records it.
  A plain alias (`type Age = Int`, i.e. `Refined { refinement: None }`) counts, as
  #259 intends. First-party/synthetic units are excluded (they carry no on-disk
  def site), exactly as they are from `symbols`.
- **The lens (`bynk-lsp`).** `index_queries::refinement_family_lenses(index,
  path)` — for each refined/opaque `Type` def in the file, the family over its
  base (def sites across the project, sorted by position). It emits a lens only
  for a family of **≥ 2** (a lone refinement has nothing to navigate to, so no
  lens — mirroring `capability_provider_lenses`, which is empty without a
  provider). The `code_lens` handler appends an `N refinements of <Base>` lens
  carrying `editor.action.showReferences` over the family — the same client-side
  hydration the reference/provider lenses use, so **no VS Code change**.

## Decision

- **Codelens, not a workspace-symbol filter.** #259 offered either. The codelens
  fits the track's codelens-depth theme and slice 6's pattern, needs no client
  change (the hydration exists), and puts the family where you read the type.
- **The family includes `self`; count ≥ 2 to show.** `N` is the full family size
  (a stable number every member shows), and the peek lists all members. A
  family-of-one gets no lens — there is nothing to navigate. This matches the
  provider lens's "empty unless there is something to point at".
- **Builtin base only — by construction.** A refined/opaque type's base is always
  a `BaseType` (the AST admits nothing else), so a family is grouped by one of the
  seven builtins; there is no user-type refinement graph to model, and #259 is
  explicit that this is the query that fits Bynk's model where `typeHierarchy`
  does not.

## Test delta

- **`bynk-lsp`** `index_queries` test: `refinement_family_lenses` pairs each
  refined/opaque type in the file with its base's family (≥ 2), spanning files;
  a lone refinement and a non-refinement type yield no lens; off-file query empty.
- **`bynk-check`** `index` test: `RefineEdge`s are recorded for refined/opaque/
  alias types and grouped by base via `refinements_over` / `refined_base`.
- The existing e2e index/coverage suites exercise the population path.
