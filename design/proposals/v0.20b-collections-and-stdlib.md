# v0.20b — Built-in collections + the combinator stdlib

- **Status:** Draft (proposal). **Merging this PR is the approval to build.**
- **Builds on (verified shipped in v0.20a):** `TypeRef::Fn`, `LambdaExpr`, `TypeParam`;
  generic functions; value application; effectful function-value calls under `<-`
  confinement (0031); closures + bottom-up lambda effectfulness (0033); function types
  confined to non-boundary positions (0030). `Result` already carries two type params
  (`ast.rs`), so `Map[K,V]` has a TypeRef-level template. Reserved `karn.*` prefix
  (0012/0026) is where the stdlib commons live.
- **Spec sections touched:** §4 (grammar), §5 (static semantics), §6 (type system), §7
  (emission), §8 (compilation model), §9 (diagnostics), Appendix B.
- **Second slice of the functional core.** v0.20a shipped the machinery; this ships its
  **first real consumer** — `karn.list`/`karn.map` written *in* Karn over generic
  functions, lambdas, and effectful traversal.

## Goal

Add built-in **`List[T]`** and **`Map[K, V]`** as compiler-known generic types, a thin
**built-in kernel**, and the **Karn-written combinator stdlib** over it. The increment
after which Fetch's missing-headers compromise (0022) becomes *retirable* — the
retirement itself is a separate `karn`-surface touch-up (0023), not this slice.

## Settled (recorded as ADRs ≥0034)

- **▸ The hybrid: thin built-in kernel, Karn-written combinators.** Only the irreducible
  primitives are compiler/emitter magic; everything derivable is ordinary Karn in
  `karn.list`/`karn.map` (dogfooding v0.20a).
- **▸ `List`/`Map` are built-in generic *types*** (TypeRef-level, like `Option`/`Result`),
  **immutable**: every op returns a new value, none mutates. Emission: `List[T]` →
  `readonly T[]`; `Map[K,V]` → `ReadonlyMap<K, V>`.
- **▸ Collections serialise at boundaries; nested functions still don't.** `List`/`Map`
  get `serialise_`/`deserialise_` cases, so a handler may take/return `List[Order]`. The
  0030 boundary rule **looks through** them — a `List[Int -> Int]` field stays
  `function_at_boundary`.
- **▸ `Map` wire format + order.** A `Map` serialises as an **entries array** `[[k,v],…]`
  (uniform across `String`/`Int` keys, unlike a JSON object), and is **insertion-ordered**
  — `keys` and the wire form are deterministic, normatively.

## Decisions (open — settle at review)

**[DECISION] The kernel (how thin, incl. the builder).** Karn has no loops, so the kernel
supplies iteration *and* construction. The combinators can't build their results without a
list-builder primitive (a `fold` over `length`/`get` alone can read but not write). *Rec:*
**`List` kernel** = literal `[…]`, `List.empty`, `length`, `get(i) -> Option[T]`,
**`prepend(x)` (cons, O(1))** as the builder, and the two folds **`fold`** (pure) +
**`foldEff`** (`(Acc, A) -> Effect[Acc]`, sequential, `<-`-bound per 0031). Order-preserving
combinators build with `fold` + `prepend` + a derived `reverse` (O(n) — *not* `[...acc, x]`
append, which is O(n²)); the emitter may use hidden local mutation inside the fold loop to
keep it linear. **`Map` kernel** = `empty`, `insert(k,v)`, `get(k) -> Option[V]`, `keys ->
List[K]`, `length`; `Map.fromList` derives. The rejected alternative — expose `head`/`tail`
and recurse — invites unbounded user recursion the no-loops stance exists to avoid.

**[DECISION] Call surface (biggest unstated call).** Method resolution is **declared-method
(`has_self`) based, not universal UFCS** — and *declared* generic methods are deferred
(0028) — so a *user-written* generic combinator can't be `xs.map(f)`. (The kernel ops are
exempt: `xs.fold` is also generic in its accumulator, but kernel ops are **compiler-known
special forms typed directly by the checker**, not declared methods, so they're generic
without the declared-generic-methods feature existing. The deferral bites only on declared
methods — state this in ADR 0037.) *Rec:* (a) **kernel receiver-ops are compiler built-in
methods** (`xs.fold`, `xs.length`, `xs.get`, `xs.prepend`; `m.get`, `m.insert`, `m.keys`) —
the compiler registers them on `List`/`Map`; (b) **construction** via the `[…]`
literal + **qualified statics** `List.empty()` / `Map.empty()` / `Map.fromList(…)` (reusing
the `HttpResult.Ok` qualified-call form); (c) **combinators are Karn-written generic *free*
functions** in `karn.list`/`karn.map` — `map(xs, f)`, `filter(xs, p)`, `traverse(xs, f)` —
brought in by `uses karn.list`. **No method chaining (`xs.map`) in v0.20b**; it arrives
additively when generic methods land. (Free names like `map`/`filter` are explicit-import
via `uses`; clash handling is the existing `uses` behaviour.)

**[DECISION] `Map` key constraint.** TS `Map` uses reference equality for object keys.
*Rec:* keys restricted to **value-keyable** types (`String`, `Int`, refined/opaque over
them); record/sum keys → `karn.types.unkeyable_map_key`. Revisit with bounded generics.

**[DECISION] Literals.** *Rec:* `List` literal `[a, b, c]` (leading `[` in expression
position — currently a reserved-syntax error at `parser.rs:3635`, so no collision; type-app
`[` is postfix only, `parser.rs:3521`). **No `Map` brace literal** (`{ }` is records/blocks).
Empty `[]` needs an expected type or annotation → `karn.types.uninferable_element_type`.
`List.empty()`/`Map.empty()` share that rule (their element/key types are equally
uninferable); confirm whether v0.20a's postfix `name[T](…)` type-application extends to a
qualified static (`List.empty[Int]()`) as the explicit escape, or note that these too rely
on an expected type.

**[DECISION] `traverse` only; defer `parTraverse`** (Promise.all emission, fast follow).
**[DECISION] `Set` deferred** (same keyable constraint; adds little the others don't prove).

## Fixtures (shape — not normative emission)

- **Positive** — `[…]` literal + `map`/`filter`/`find` from `karn.list` (over the built-in
  `xs.fold` kernel method beneath them); a generic
  `sumBy` (dogfooding v0.20a generics); **`traverse`** with an effectful lambda calling a
  `given` capability (proves 0031/0033 through the stdlib); `Map` build/get/keys; a handler
  returning `List[Order]` (collection boundary serialisation). Each passes `tsc --strict`.
- **Negative** — `Map[Order, V]` → `unkeyable_map_key`; `let xs = []` no type →
  `uninferable_element_type`; a `List[Int -> Int]` record field → `function_at_boundary`
  (boundary sees through `List`).
- **Additive guard** — prior fixtures byte-stable; v0.20a programs unchanged.

## Risks

- **Injection for a non-adapter unit.** *Verified:* `firstparty.rs`/`project.rs` inject the
  `karn` surface **as a synthetic adapter** — the only injected unit kind. `karn.list`/
  `karn.map` are plain commons of pure generic functions; the injection + `uses`-resolution
  path for a first-party *library* unit is **untested territory** and must be built/verified
  first.
- **Kernel completeness.** The `prepend` builder is the fix for the "fold can read but not
  write" hole; without it the combinators fall back into the compiler.
- **Linearity / immutability.** Build with `prepend` + `reverse` (O(n)), not `[...acc, x]`
  (O(n²)); emit `readonly`/`ReadonlyMap` with no in-place mutation leak (a stray `.push` is
  exactly the bless-without-reading trap).
- **`fold` can't short-circuit.** `find`/`any`/`all` scan the whole list — semantically
  invisible for pure code, but record it so nobody later "fixes" it into a compiler
  primitive without a decision record.
- **Boundary look-through, latent v0.20a check.** Confirm the 0030 rule already recurses
  through `Option`/`Result` payloads (an `Option[Int -> Int]`). If it does, the `List`/`Map`
  extension is mechanical; **if it doesn't, that's a latent v0.20a gap this slice fixes**.
- **List-literal statement boundary.** Confirm a statement beginning with `[` (after an
  expression line) can't be misparsed as type application — depends on statement separation.

## Out of scope (→ later)

- **List patterns** (matching `[]`/`[head, …tail]`) — list *literals* without *patterns* is a
  deliberate v0.20b stance; patterns are deferred, not forgotten.
- Method **chaining** (`xs.map`) — arrives with generic methods, additively.
- `Set`; `parTraverse`. **Fetch headers** — *unblocked* by this slice (0022); the retirement
  (`Map[String,String]` on `Response`) is a separate `karn`-surface touch-up (0023).
- Generic *types*, bounded generics (value-equality keys, `Ord`-bounded sort) → later.
