# v0.126 — Editor currency, slice 5: the UI surface

- **Status:** Draft (sign-off artefact; deleted on merge)

- **Realises:** [ADR 0156](../decisions/0156-editor-surface-tracks-language.md)
  (the editor surface tracks the language) — the manifest is part of that surface.

## What changes

The VS Code extension had the *commands* (`newContext`/`runTests`/`debugTests`/
`openProjectConfig`/…) but none of the **menu, keybinding, or editor-config**
wiring that makes them discoverable — every command sat only in an ungated
Command Palette, there were no run/debug buttons, and `language-configuration.json`
carried only bracket/indentation rules. This slice adds that UI surface. It is
**manifest-only** plus one small activation hook; no grammar, checker, emitter,
or spec change. Migration is nil.

Closes [#305](https://github.com/accuser/bynk/issues/305) (menus + keybindings),
[#306](https://github.com/accuser/bynk/issues/306) (language-configuration
enrichment); pre-Marketplace polish for
[#258](https://github.com/accuser/bynk/issues/258).

## Tooling delta

The four LSP responses ADR 0156 names are **unchanged** — this slice touches no
language request; it is editor-shell wiring around commands that already exist.

- **`contributes.menus`:**
  - `commandPalette` — the Bynk commands (`newContext`, `openProjectConfig`,
    `runTests`, `debugTests`, `restartServer`, `downloadServer`,
    `showServerOutput`) gated `editorLangId == bynk || bynk.hasProject`, so they
    vanish from unrelated workspaces. **`newProject` is deliberately left
    ungated** — it bootstraps a project from an empty folder, exactly the case
    where `bynk.hasProject` is false.
  - `editor/title/run` and `editor/context` — `runTests`/`debugTests`, scoped
    `editorLangId == bynk`.
  - `explorer/context` — `newContext` / `openProjectConfig` on a folder in a
    Bynk project (`explorerResourceIsFolder && bynk.hasProject`).
- **`contributes.keybindings`:** a two-stroke `ctrl+;` / `cmd+;` chord —
  `;`→`t` runs, `;`→`d` debugs — scoped `editorLangId == bynk` so it shadows
  no global default (the chord leader also sidesteps the reopen-closed-editor
  and debug-console defaults a single chord would clobber).
- **`language-configuration.json`:** a `wordPattern` (numeric literal or
  identifier) and `onEnterRules` (brace-aware auto-indent — `indentOutdent`
  between `{`/`}`, `indent` after a trailing `{`).

## Decision

- **`workspaceContains:` is not a `when`-clause key.** The track text sketched the
  palette gate as `editorLangId == bynk || workspaceContains:bynk.toml`, but
  `workspaceContains:` is an *activation* predicate, invalid in a `when` clause.
  *Resolution:* the extension sets a custom **`bynk.hasProject`** context key
  from `vscode.workspace.findFiles("**/bynk.toml", …)` at activation and on
  workspace-folder change, kept live by a `**/bynk.toml` create/delete watcher;
  the menu gates read that key. This is the one non-manifest addition — a small
  activation hook in `extension.ts`.

## Test delta

- **New** `vscode-bynk/test/suite/manifest.test.ts` (the extension integration
  suite, CI-only): asserts the palette gate is present on each Bynk command and
  absent on `newProject`; that `editor/title/run`/`editor/context` carry the
  run/debug commands scoped to bynk editors; that `explorer/context` carries
  New Context / Open Project Config gated on a project; that both keybindings
  exist and are bynk-scoped; and that `language-configuration.json` defines a
  compilable `wordPattern` and non-empty `onEnterRules`. A dropped `when` clause
  — the regression that would flood every workspace's palette — fails the build.
