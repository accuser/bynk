# v0.29.14 — Refactor track: split `parser.rs` into named submodules (impl-method split)

- **Status:** Draft (proposal). **Merging is approval to build.** *(Applies ADR 0060 to the second
  remaining sprawling file — and the **first split of an `impl`-method file**, establishing that recipe
  for the convention. Independent; the gate is the parser unit tests + the tree-sitter corpus.
  Behaviour-preserving — the parse tree for every input is identical.)*

## Context — a 5.3k monolith, and a different shape

- `parser.rs` is **5,295 lines** and **fully monolithic** (no `parser/` directory).
- It is **not free functions.** It is **79 `&self` methods on `struct Parser`** (3 `impl` blocks, 6
  free fns). So — unlike `project.rs`/`checker.rs`/`emitter.rs` — the flat-resolution *free-function*
  move does **not** apply. This is the first impl-method file the convention meets, and the proposal's
  job is to establish how that splits.

## The headline decision

**[DECISION] Split via multiple `impl Parser` blocks across submodules — not a free-function move.**
Rust allows a type's `impl` to be spread across several modules of the same crate. So `struct Parser`
(+ its scanning core) stays in the parent `parser.rs`, and each concern becomes its own
`impl Parser { … }` block in a submodule:
- `parser/declarations.rs` — `parse_context`/`parse_commons`/`parse_capability`/`parse_provider`/
  `parse_service`/`parse_agent`/`parse_fn`/`parse_test`/`parse_mock`/`parse_wires`/`parse_uses`.
- `parser/types.rs` — `parse_type`/`parse_sum`/`parse_variant`/`parse_refinement`/`parse_record`/
  `parse_unit`/`parse_signed`.
- `parser/expressions.rs` — `parse_expr`/`parse_binop`/`parse_unary`/`parse_primary`/`parse_match`/
  `parse_call`/…
- `parser/patterns.rs` — `parse_pattern`/… ; `parser/statements.rs` — `parse_statement`/`parse_let`/…

*Rec: do this.* It gives the same legibility win as the free-function splits, by concern.

**[DECISION] The cost — most methods become `pub(crate)`.** Private methods are *module*-private, so a
`declarations` method calling `self.parse_expr()` (private, now in `expressions.rs`) **won't compile**.
The parser is `parse_X` calling `parse_Y` throughout, so **most of the 79 methods must widen to
`pub(crate)`** for cross-submodule `self.`-dispatch — a broader visibility surface than the
free-function splits' targeted bumps. *This is the accepted ADR-0060 trade (legibility over a tight
surface); call it out, don't fight it.* Drive it compiler-first: build, bump the specific methods the
privacy errors name.

## Builds on (verified)

- **The concern clustering is clean** (verified from the `parse_*` method names): declarations
  (context/commons/capability/provider/service/agent/fn/test/mock/wires/uses), types
  (type/sum/variant/refinement/record/unit/signed), expressions (expr/binop/unary/match/…), patterns,
  statements. The methods are cohesive — all advance the one token stream — so the seams are about
  navigation, not data ownership.
- **The scanning core stays in the parent**: `struct Parser`, the cursor/token state, and the low-level
  primitives (`advance`/`peek`/`expect`/`bump`/error-recovery) + the public entry point. Every
  submodule's `impl Parser` calls these via `self`.

## Settled

- **▸ Behaviour-preserving, parse-tree-identical.** Method bodies move **verbatim**; only `impl`-block
  placement and `pub(crate)` visibility change. The gate is the parser unit tests **plus the
  tree-sitter corpus** (which exercises the grammar end-to-end).
- **▸ The struct + core stay put.** Only the per-concern `parse_*` methods relocate, as `impl Parser`
  blocks in the submodules.

## Decisions (open — settle at review)

**[DECISION] Slice into PRs.** *Rec:* **per-concern PRs** — `declarations` first (the largest cluster),
then `expressions`, then `types`/`patterns`/`statements` (smaller, can pair). Each moves one
`impl Parser` block, golden/corpus-verified.

**[DECISION] Characterization pins?** *Rec:* unlike `checker.rs`, the parser is covered by its unit
tests + the tree-sitter corpus (full grammar coverage), so **no new pins are needed** — the corpus is
the safety net. (If a pure free-function helper among the 6 is moved and is under-covered, pin it, per
the v0.29.1 rule.)

## Spec delta

**None.** No language/grammar surface, no `docs/src/spec/` chapter — module reorganisation only. (The
normative grammar lives in the spec + the tree-sitter corpus, both untouched.)

## Diagnostics / Risks

- **The `pub(crate)` surface is broad** (most of 79 methods) — the accepted ADR-0060 trade. It widens
  the parser's internal API but changes no behaviour; the alternative (keep one 5.3k file) is worse.
- **`impl`-block split is new** — verify the cross-submodule `self.parse_*()` dispatch compiles
  (it does, same crate); the failure mode is a privacy error at build time, not a silent change.
- **Content-preservation check** — every method present exactly once across the new tree, as in the
  prior splits.

## New decision records this increment adds

- **None** beyond what ADR 0060 already establishes — though this increment is the **worked example**
  of 0060's impl-method case (a one-line note in 0060's consequences, or a PR note, suffices). Decide at
  review whether the impl-block recipe warrants its own short ADR.

## Definition of done

`parser.rs` is `struct Parser` + the scanning core + a facade, plus `parser/{declarations,types,
expressions,patterns,statements}.rs` each holding an `impl Parser` block of the relevant `parse_*`
methods (moved verbatim, widened to `pub(crate)` only as cross-submodule dispatch demands); the parent
drops from 5.3k to ~1–1.5k; content-preservation verified; the parser unit tests + tree-sitter corpus
pass **unedited**; `cargo build`/`test`/clippy/fmt and CI stay green; changelog notes the split. The
implementing PR(s) **delete this file**.
