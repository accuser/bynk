# v0.70 — Source maps for spliced bodies: per-statement breakpoints in handlers and tests

- **Track:** [debugging](../tracks/debugging.md) — the deferred handler-body / test-body
  follow-on that slices 1–2 noted, and the **prerequisite for slice 3 being useful**.
- **Realises ADR:** [0103](../decisions/0103-source-map-contract.md) — completes its
  coverage (slice 1 mapped free-function bodies and declarations; this maps the bodies
  that lower through *spliced local buffers*).
- **Posture:** codegen / tooling. Richer source maps; the emitted TypeScript is
  byte-identical (only `.ts.map` contents change, plus test modules gain a map).

## Why now

The slice-3 spike confirmed workerd debugging works end-to-end — `wrangler dev
--inspector-port` exposes an attachable CDP endpoint and esbuild composes slice 1's
`handlers.ts.map` into the worker bundle, so a `.bynk` breakpoint binds and pauses on
a real request. **But it binds at the wrong granularity.** A Worker *is* its service
handlers, and handler bodies lower into a **spliced local buffer**, so every handler
line collapses to the `service` *declaration* line: the spike's six worker-bundle
lines all mapped to `svc.bynk:5`, not to the `Logger.info` call at `svc.bynk:7`. The
same is true of test-case bodies (slice 2 left test modules unmapped entirely).

So the single highest-value piece of debugging work right now is not slice 3's thin
launch flag — it is closing this gap, which makes **both** the Node test path (slice 2)
and the workerd path (slice 3) resolve breakpoints to the *statement* a user actually
clicks. That is why this increment comes before slice 3.

## Current state — the spliced-buffer gap

Slice 1's `SourceMapBuilder` records `(byte offset in the output buffer, source span)`
checkpoints via `cx.record_span(out.len(), span)`, resolved to generated lines against
the final buffer. `emit_free_fn` attaches the builder and lowers **directly into the
main `out`**, so its offsets are correct. The deferred sites lower into a **local
buffer** that is later spliced into the module, so their offsets are wrong (or, today,
simply unrecorded — the body `cx` has no builder, and the lines inherit the enclosing
declaration checkpoint). The exhaustive site list:

| Site | File | Splice shape |
|---|---|---|
| **Service handlers** | `emit.rs` `emit_service` (~677–789) | `out.push_str(&body_out)` — verbatim |
| **Agent handlers** | `emit.rs` `emit_agent` (~1468–1512) | `out.push_str(&body_out)` — verbatim |
| **Unit test bodies** | `tests_emit.rs` `emit_test_case_function` (~2127–2140) | line-by-line, indent prepended |
| **Integration test bodies** | `tests_emit.rs` `emit_integration_module` (~749–760) | line-by-line, indent prepended |
| **Mock op bodies** | `tests_emit.rs` `emit_mock_op_body` (~1783–1796) | line-by-line, indent prepended |

(`emit_provider` lowers **directly into `out`** — no splice — so provider bodies are
already correctly mappable and only need the builder attached, like `emit_free_fn`.)

Two complications beyond "attach a builder":

1. **The test-body splices reconstruct text line-by-line** (prepending indentation),
   so byte offsets do not survive — but the *line structure* does (one body line → one
   output line). A byte-for-byte rebase fails; a **line-anchored** merge works.
2. **Test modules aggregate multiple `.bynk` files** (a `test <target>` group's cases
   can come from several files), so a test module's map needs **multiple `sources`** —
   slice 1's builder serialises a single source. Handler bodies stay single-source (one
   `.bynk` file per `emit_project` call), so this only bites the test path.

## The mechanism — sub-builder per body, merged at the splice line

The shape is uniform across all spliced sites:

1. **Lower each body with its own `SourceMapBuilder`** (a local `RefCell`), attached to
   the body's `LowerCtx` via `with_source_map`. The existing `record_span` calls in
   `emit_statement`/`emit_match_*` then record checkpoints **relative to the local
   buffer** — no new instrumentation, slice 1's already fires.
2. **At the splice, merge into the module builder, anchored by generated line.** A new
   `SourceMapBuilder::merge(sub, sub_text, base_line, source_id)` translates each
   sub-checkpoint's body byte-offset → body line → output line (`base_line + body_line`)
   and records it into the module builder. Because it anchors on *lines*, it is correct
   for both the verbatim `push_str` splices (handlers) and the line-by-line indented
   splices (test bodies) — both preserve line structure. `base_line` is the output
   line at the splice point (newlines in `out` so far).
3. **Multi-source for test modules.** The builder gains a `sources` list; each
   checkpoint carries a source index, and `to_v3` emits `sources` / `sourcesContent`
   arrays. Handler/free-fn emission uses a single source (index 0), unchanged. A test
   module registers each participating `.bynk` file and tags its cases' checkpoints
   with the right index.

No emitted-TS change: the bodies emit exactly as today; only the accompanying map
gains the per-statement checkpoints, and test modules gain a map where they had `None`.

## What this slice does

1. **`SourceMapBuilder` gains `merge` + multi-source.** `merge(sub, sub_text,
   base_line, source_id)` (line-anchored); `sources: Vec<(name, text)>` with a
   `source_id` on each checkpoint; `to_v3` emits the arrays. Single-source callers pass
   `source_id = 0` and one source — byte-identical to slice 1's output for those.
2. **Handler bodies (service + agent).** Thread the module builder into `emit_service`
   / `emit_agent`; lower each handler body with a sub-builder; merge at the
   `out.push_str(&body_out)` splice. (`emit_provider` just attaches the builder — no
   splice.) These are single-source (the context's `.bynk`). **Unblocks slice 3's
   per-statement workerd breakpoints.**
3. **Test-case bodies (unit + integration + mock).** Give `emit_test_module` /
   `emit_integration_module` a builder; lower each case body with a sub-builder; merge
   at the line-by-line splice; register each case's source file; serialise the module
   map (test modules stop returning `source_map: None`). **Unblocks slice 2's
   in-test-body breakpoints.**
4. **Decode goldens** asserting per-statement mapping for: a service handler statement
   (→ its `.bynk` line, not the `service` line), an agent handler statement, and a test
   body `assert` (→ its `.bynk` line), including a multi-file test group to exercise
   multi-source.

## Decisions

**[DECISION A — sign-off] Line-anchored merge, not byte-rebase.** Fork: (a) a
line-anchored `merge` that works for both verbatim and line-by-line splices; or (b)
byte-rebase, which forces refactoring the test-body splices to verbatim `push_str`
(lowering with indent instead of prepending it). **Recommend (a)** — it leaves the
splice code untouched (lower golden risk) and is the model line-level maps actually
need; the body byte-offset is only ever used to derive a line.

**[DECISION B — sign-off] Map test modules now (multi-source), not later.** The user
chose "handler **and** test bodies." Test modules need multi-source support; that is
the one genuinely new builder capability. *Recommend* building it now so slice 2's
test-body breakpoints land with this increment rather than spawning yet another
follow-on. (If multi-source proves disproportionate, the fallback is handler-bodies-only
this increment, test bodies next — but the spike shows both are the same merge.)

**[DECISION C] `emit_provider` just attaches the builder.** No splice, so provider
operation bodies map per-statement simply by giving their `cx` the builder — folded in
for completeness (providers run under workerd too).

**[DECISION D] Version — recommend minor v0.70.** Richer maps, no language or emitted-TS
change → minor, over the 0.69.0 baseline.

## Risks & mitigations

- **Wrong `base_line` drifts a whole body.** A merge off by one line mis-anchors every
  statement in that handler/test. Mitigated by decode goldens that assert exact
  `.bynk` lines for real handler/test statements (not just "a mapping exists").
- **Multi-source index bugs.** A checkpoint tagged with the wrong source maps to the
  wrong file. Mitigated by the multi-file test-group golden.
- **Golden churn.** Handler/test `.ts.map` contents change (richer); the emitted `.ts`
  stays byte-identical, so only `.ts.map` goldens (decode tests) move — no 500-file
  churn. Test modules gain a map on disk (was none).
- **Scope creep into columns.** This stays **line-level** (ADR 0103 D1) — the merge
  anchors on lines; expression/column granularity remains out of scope.
