# 0154 — test-time provider substitution is `provides`, not `given`: a per-seam `Cap.method(<args>) returns <v> | fails` override, capability-only, at case and suite scope; `mocks` retired

- **Status:** Accepted (v0.118; 2026-07-03)
- **Provenance:** the v0.118 increment — the testing track's seventh slice. It is the load-bearing record for how a test substitutes a collaborator, and it retires the `mocks` re-implementation block — after which the word "mock" is gone from the language.
- **Realises:** the testing track's seam triad — `consumes` *declares* a seam, `given` *requires* it, `provides` *supplies* it — with `given` no longer doing double duty. A test double is an alternative *provider* of a capability, and production already spells that `provides`.
- **Relates:** the one predicate surface (ADR 0144 — a `provides` argument pattern reuses the predicate surface, `_` / literal, so there is no matcher vocabulary); the tier dial (ADR 0153 — `as <tier>` sets the default provision of every seam, a `provides` clause overrides one); the auto-recorded seam (ADR 0152 — a `provides` stub sits behind the same recording proxy observation reads, so `expect Cap.op called …` composes with `provides` unchanged); `Val[T]` replacing `Mock[T]` (ADR 0148 — together they remove the word "mock" from the language).

## Context

The old test double was a `mocks Cap = Impl { fn op(…) -> T { … } }` block: a whole alternative *implementation* of a capability, with method bodies. It was a second sublanguage — re-implementing a collaborator inside a test — and it invited exactly the drift and over-fitting a test double should avoid (a `mocks` body could grow arbitrary logic, becoming a parallel program to maintain). Meanwhile Bynk already had a first-class word for "supply an implementation of a capability at a seam": `provides` (`provides Clock = SystemClock { … }` in production). A test double is that same idea, scoped to a test. The questions: what does the substitution *spell*, what may its right-hand side *be*, and how does it express a collaborator whose successive calls differ (an advancing clock, a fail-twice-then-succeed network)?

## Decisions

**D1 — a test double is `provides`, scoped to a test, always an explicit call pattern.** A test substitutes a provider at a seam with `provides Cap.method(<args>) returns <value> | fails`. Naming the method with its call pattern keeps the target unambiguous (no "which method?" inference), makes argument-shape matching uniform, and reuses the predicate surface for the pattern. This completes the triad — `consumes` declares, `given` requires, `provides` supplies — and `given` reverts to its one production meaning (require a capability in scope), which is already what the compiler enforces.

**D2 — the right-hand side is a value or a fault, never a computed body.** `returns <value>` supplies a value; `fails` injects a capability fault (a `Result`/`Err` is an in-band *outcome* asserted directly in the case, but a *fault* is what `fails` models — Principle 3). The RHS is never a block: a double that needs logic is the signal to use the real provider (promote the tier) or to reach for the sequenced form. This is the rule that stops `provides` becoming the `mocks` block reborn. A `returns` value whose type disagrees with the operation's declared return type is `bynk.provides.rhs_type`.

**D3 — argument patterns reuse the predicate surface; first match wins.** Each parameter takes a pattern: `_` (any) or a value the recorded argument must equal (a literal or pure value expression). Multiple `provides` clauses for one method form an ordered match list, tried top-to-bottom, first match wins — so a specific clause precedes a fallback (`provides Rates.lookup("GBP") returns 1.25` before `provides Rates.lookup(_) returns 1.0`). No matcher words; the pattern surface is the one predicate surface.

**D4 — `provides` is capability-only, at case and suite scope, case wins (DECISION M).** An agent's realness is the tier's job (ADR 0153), not a provider's — so `provides` targets a *capability* seam only. It sits at suite scope (applies to every case) and case scope (overrides for one case); precedence is case `provides` > suite `provides` > the tier default. Overriding a capability the unit under test does not consume is `bynk.provides.not_a_seam`; naming an operation the capability does not declare is `bynk.provides.unknown_op`. Bare observation needs no `provides` — calls are recorded at the seam regardless (ADR 0152); a `provides` is written only when the case depends on a collaborator's *return*.

**D5 — sequenced provision is `returns each [<outcome>, …]`, last outcome repeats (DECISION V).** A single value cannot express a collaborator whose successive calls differ — an advancing clock, a `Kv.get` that returns `None` then `Some`, a network that fails twice then succeeds — and "use the real provider" fails for fault sequences (a real network cannot be asked to fail exactly twice). The **return-sequence** supplies one outcome per call, in order: `provides Clock.now() returns each [1000, 2000, 3000]`. Each outcome is a value (a success) or `fails` (a fault). The `each` disambiguates "one outcome per call" from a single call that returns a *list value*. **Exhaustion:** the last outcome repeats (steady state) — `[fails, fails, ok]` is "fails twice, then succeeds forever". Rejected exhaustion rules: erroring on exhaustion (brittle — an unrelated extra call fails the case on the wrong axis) and cycling (surprising). A malformed sequence (e.g. empty) is `bynk.provides.bad_sequence`. A collaborator that must *compute* its next return (a delta-advancing clock, a threaded cursor) exceeds a fixed sequence and is the reserved **virtual fixture** — a named follow-on, not shipped here.

**D6 — `mocks` is retired; "mock" leaves the language.** The `mocks` re-implementation block and its `bynk.mock.*` diagnostics are removed. A `mocks Cap = Impl { fn op(a) -> T { body } }` migrates to one `provides Cap.op(<pattern>) returns <value>` per operation; a `mocks` body with genuine logic has no value form and is the signal to promote to `as integration` or to use `returns each`. Together with `Mock[T] → Val[T]` (ADR 0148), the word "mock" no longer appears in Bynk.

## Consequences

- A test double is written in the same seam word production uses (`provides`), the same predicate surface for its argument patterns (`_`, literals), and a value or `fails` for its result — no re-implementation sublanguage, no matcher library.
- `given` means exactly one thing (require a capability); `consumes`/`given`/`provides` are a clean declare/require/supply triad.
- A `provides` stub sits behind the slice-5 recording proxy, so a case can both stub a return *and* observe the call with `expect Cap.op called …` — one seam, two views.
- The value-or-`fails`-only rule keeps a double honest; a double that wants logic is a signal to change tier, which is a deliberate friction.
- **Re-openable:** a virtual-fixture form for genuinely stateful collaborators (the named follow-on above); `is`-narrowing argument patterns beyond equality; and provider substitution across the `system` wire.
