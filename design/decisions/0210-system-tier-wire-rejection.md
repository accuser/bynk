# 0210 — A `system`-tier case drives the boundary with raw `Wire(…)` input and observes the rejection

- **Status:** Accepted (v0.189)

**Context.** Slice B (#697) let a `case` drive an http route at `system` over a real `fetch`, but only with **typed** arguments — which are valid by construction, so the boundary can never reject them. The testing-the-boundary track's §1 thesis is that *no Bynk test can observe the boundary rejecting attacker input*; `examples/feature-flags/tests/keys.bynk:15` ("an empty flag name is rejected at the boundary") asserts the refinement *predicate*, not the router. To close that, a case must be able to hand the boundary input the type system forbids — which no typed argument can express.

**Decision.**

- **D1 — `Wire(<String>)` is a raw, `system`-only service-address argument.** `api.POST("/cart", Wire("{\"sku\": \"\"}"))` passes the string to the boundary *unvalidated* — the router deserialises and validates it exactly as it would an over-the-wire request, so a refinement violation / malformed JSON is rejected before the handler. `Wire` is legal only as an argument to a service address in a `system`-tier case: there is no wire to be raw about at `unit`, and it is meaningless in any other position (`bynk.test.wire_needs_system`). It takes a `String` (not a JSON value) so a case can reach `MalformedJson` (`Wire("{not json")`). No refined value is ever minted from a `Wire` — the *router* validates it — so ADR 0182 is untouched. A new `wire_expr` grammar rule (dual parser) carries it; the checker keeps `Wire`'s inner typed as `String` but bypasses the param's type (the whole point).

- **D2 — the result follows the arguments: a `Wire` call yields `Rejected | Handled`.** A typed call keeps `HttpResult[T]` (Slice B) and promotes; a `Wire`-carrying call yields an outcome sum — `Rejected(detail)` when the router refused the input before the handler, `Handled(httpResult)` when it ran. A per-route **raw driver** (`__sysdrive_raw_…`, emitted only into `out/tests/`) sends the raw string verbatim (no `serialise`) and decodes the response through a new `responseToHttpOutcome` runtime helper (the sibling of Slice B's `responseToHttpResult`). The outcome is kept **loose** in the checker — as Slice B keeps `HttpResult` loose — so no new type-system machinery is added; the value is decoded at runtime and `expect r is Rejected(_)` lowers to a `.tag` test. Promotion is preserved: a typed case body is byte-for-byte unchanged.

**Consequences.**

- A `system` case proves the boundary claim §1 is about: `Wire("{\"sku\": \"\"}")` → `expect r is Rejected(_)` (the empty `sku` is refused before the handler); a malformed body → `Rejected`; a valid raw body → `Handled`. Fixture `385_system_wire_rejection` drives all three and type-checks under `tsc --strict`; negative fixture `386_wire_needs_system` locks the misplaced-`Wire` diagnostic.
- `responseToHttpOutcome` joins the runtime as the one net-new helper (test-output-only drivers aside). It classifies on the body's **shape, not its status**: a `400` is `Rejected` only when its body carries a recognised `BoundaryError.kind` (`RefinementViolation`/`MalformedJson`/`StructuralMismatch`) — the router's pre-handler rejections. A handler that *ran* and returned `BadRequest(msg)` also yields a `400`, but its body is `{ error: msg }` with no `kind`, so it is `Handled` — because the handler ran, which is the whole meaning of the distinction. Fixture 385 drives a handler-returned `BadRequest` with a valid `Wire` and asserts `Handled(_)` to lock this. The raw driver is emitted only for routes with a `Wire`-eligible slot (a body or path param).
- **Not in scope (noted follow-ons):**
  - **Rejection-kind discrimination.** `is` tests one variant level by language design (`is Ok(_)` ignores the payload), so `expect r is Rejected(RefinementViolation(_))` tests only `Rejected` — it proves the boundary rejected, not *which* kind. Distinguishing `RefinementViolation` from `MalformedJson` needs nested-`is` (a language-wide `is` change) or `match` over the loose outcome (not wired); deferred.
  - **The `401` path** (missing/invalid credential) needs a credential-override surface — the framework always signs a valid credential at `system` (Slice B), so there is no way yet to drive an unauthenticated request.
  - **The `405` fall-through** needs wrong-method addressing — the checker requires an address's `(method, path)` to match a declared route exactly (`service_unknown_route`), so a wrong-method call cannot be spelled today.
  - **Mixed typed+`Wire` arguments in one call** (a raw path with a typed body): the raw driver forwards every slot raw, so a `Wire` call's non-`Wire` args are lowered as-is; a typed body mixed with a `Wire` path is deferred.
  - The unit-tier `wire_needs_system` guard fires for a misplaced `Wire` and for a `Wire` in a non-`system` suite case; a `Wire` in a bare `unit` suite targeting an http route is caught by `tsc` (a raw `string` where the handler's typed param is expected) rather than a clean diagnostic — tightening that to all check paths is a follow-on.
