# 0211 — `is` tests nested variant patterns structurally, not just the outer tag

- **Status:** Accepted (v0.190)

**Context.** ADR 0210 (system-tier-wire-rejection, v0.189) gave a `Wire`-driven `system` case the outcome `Rejected(detail) | Handled(HttpResult[T])`, where `detail` carries the router's `BoundaryError.kind`. But it deferred *kind discrimination* (#705): `expect r is Rejected(RefinementViolation(_))` tested only the outer `Rejected`, because `is` was one-level — a variant pattern lowered to a single `.tag` check and dropped its payload patterns. So a case could prove the boundary rejected, but not *which* rejection it produced — §1's complaint (`expect FlagKey.of("") is Err(_)` asserts the predicate, not the boundary) applied to the rejection kind too.

**Decision.**

- **D1 — a variant pattern's nested refutable payload patterns are part of the `is` test.** `is` now lowers a variant pattern through the same `pattern_match_tests` a `match` arm uses: `r is Rejected(RefinementViolation(_))` emits `r.tag === "Rejected" && r.value.tag === "RefinementViolation"`. A payload bound to a **name** (`is Ok(x)`) or a **wildcard** (`is Ok(_)`, `is Rejected(_)`) is irrefutable and adds no nested test — it only narrows, exactly as before. So every existing one-level `is` is byte-for-byte unchanged; only a nested *variant/literal* payload — previously silently ignored — now tests. This is a strict sharpening: the old behaviour accepted `is Ok(SomethingElse(_))` as matching any `Ok`, which was a latent bug, not a feature.

- **D2 — the `Wire` outcome stays loose; kind names are checked at runtime.** ADR 0210 keeps the outcome untyped (the runner recovers the shape). The nested lowering needs no static type: `pattern_match_tests`' single-field `"value"` fallback matches the outcome's runtime `{ tag, value }` shape, so `r.value.tag === "<kind>"` lands without typing the outcome. The cost is that a mistyped kind (`Rejected(RefinementViolatoin(_))`) is not a compile error — the pattern never matches, so the case *fails loudly at runtime* rather than passing wrongly. That is consistent with 0210's loose-outcome choice (a loose `is <Variant>` was already runtime-only) and avoids the type-system machinery a typed outcome would need.

**Consequences.**

- A `system` `Wire` case discriminates the rejection kind: `Rejected(RefinementViolation(_))`, `Rejected(MalformedJson(_))`, `Rejected(StructuralMismatch(_))`, or the umbrella `Rejected(_)` / `Handled(_)`. Fixture 385 now asserts the specific kinds; the emitted golden carries the two-level test (`&& r.value.tag === "…"`), so a regression to one-level `is` fails the fixture. Behaviourally verified that the wrong kind does **not** match (a `RefinementViolation` response fails `is Rejected(MalformedJson(_))`).
- The lowering change is language-wide (`is` over any sum), realised entirely in `lower_is` (`bynk-emit/src/emitter/lower.rs`) by reusing `pattern_match_tests`; the grammar and runtime are untouched. The full workspace suite passes unchanged — no existing `is` used a nested variant payload as anything but narrowing.
- **`is` now *checks* nested variant patterns for a typed scrutinee, matching `match`.** Because `is` gives nested patterns meaning everywhere it is legal — not just in `expect` — a mistyped nested variant (`if r is Err(Typo(_)) { … }`) would otherwise be a silent wrong-branch in production code. `check_is` now validates each nested refutable payload pattern against its statically-known payload type (variant existence, recursively), reusing the same `unknown_variant_in_pattern` diagnostic `match` emits; it binds nothing (nested `is` payloads only test — depth-1 narrowing names are collected separately). The loose `Wire` outcome has no type, so it is unreachable by this check and stays runtime-only (D2). Positive fixture 387 pins the typed path (`r is Err(Denied(_))` → `r.error.tag === "Denied"`, exercising payload-field resolution); negative 387 pins the typo → diagnostic.
