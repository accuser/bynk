# 0256 — The `Locale` capability, slice 1 — fixed locale, bundle-free `render`

- **Status:** Accepted (v0.221)

**Context.** Slice 1 of the `Locale` capability track ([`design/tracks/locale-capability.md`](../tracks/locale-capability.md), spine #838), per the accepted proposal issue #844. The accepted surface: `LocaleTag = String where Matches(...)`, `MessageArg = Text(value: String) | Whole(value: Int) | Num(value: Float) | Moment(value: Instant)`, `Message = { code: String, params: Map[String, MessageArg] }`, `capability Locale { fn current() -> Effect[LocaleTag] }`, `fn render(tag: LocaleTag, msg: Message) -> String` (slice 1 formats `code` plus a deterministic sorted-by-key rendering of `params` — no bundle, no locale-dependent behaviour; `tag` is accepted but unused). The proposal committed to no grammar change and no new checker rule.

**Decision — placement.** `render`, `LocaleTag`, `MessageArg`, and `Message` live in a new first-party commons, `bynk.locale` (`bynk-check/src/firstparty/bynk.locale.bynk`), alongside `message`/`withText`/`withWhole`/`withNum`/`withMoment` builder functions — not inline in the `bynk` adapter as the track doc's example layout implicitly assumed. Two structural facts force this:

1. A plain `fn` declared inside an `adapter { ... }` cannot be exported to consumers at all — `ExportKind` (`bynk-syntax/src/ast.rs`) has only `Type(Visibility)` and `Capability` variants, and adapter `exports` clauses accept only types declared locally in that same adapter file. A commons is the only unit kind whose functions are visible to consumers without an export mechanism (the existing `bynk.list`/`bynk.map`/`bynk.string` pattern).
2. A commons cannot `uses`/`consumes` an adapter (`Commons` carries no such field), so `LocaleTag` — needed by both `render`'s signature and the capability's — is likewise forced out of the adapter. The `bynk` adapter instead `uses bynk.locale` to reference `LocaleTag` in `capability Locale { fn current() -> Effect[LocaleTag] }` (an adapter `uses`-ing a commons is already precedented by `bynk.map.bynk`'s own `uses bynk.list`).

Application code gets `Locale` via the ordinary `consumes bynk { Locale }` and gets `LocaleTag`/`Message`/`MessageArg`/`render`/the builders via its own `uses bynk.locale` — the same shape as `uses bynk.string` today.

**Decision — a builder API, not raw record/sum construction.** Application code builds a `Message` via `message("code")` then `withText`/`withWhole`/`withNum`/`withMoment`, never via a bare `Text(...)`/`Whole(...)` sum-variant call or an explicit `Map[String, MessageArg]`-typed `let`. This works around (not a checker/grammar change, an API-shape choice) a real emitter gap: `emit_context_rebrands` (`bynk-emit/src/emitter.rs`) rebrands every commons type a consuming *context* references by name into a per-context-branded alias (`T & { __ctxBrand: "..." }`), so two unrelated contexts each `uses`-ing `bynk.locale` don't get structurally-interchangeable nominal types — but it only re-exports a **type** alias for this brand, never a payload-sum-type's value-side variant-constructor object, and record types' own field types stay unbranded inside the rebrand. A `fn` mixed in from a `uses`d commons is a value import and is never rebranded, so `message`/`withText`/etc. sidestep the gap entirely by construction — building the whole `Message` through function calls whose types stay inferred end to end.

The same rebranding also means a `LocaleTag` value from `Locale.current()` (unbranded — capability returns aren't subject to context rebranding) must flow **unannotated** into `render(...)` (`let tag <- Locale.current()`, not `let tag: LocaleTag <- ...`) in a context that also `uses bynk.locale` — an explicit annotation resolves to that context's own branded alias and fails `tsc --strict` (`Property '__ctxBrand' is missing`). Confirmed empirically; not fixed here (would need relaxing the adapter's `exports transparent` local-only-declaration rule, a checker change out of this slice's stated scope) — the shipped API (§ builder, unannotated capability reads) never needs the annotated form, so this is a usage note, not a shipped restriction.

**Five narrow, previously-unexercised compiler bugs fixed along the way** (each is a small, targeted fix, not a new rule — grounded against a from-scratch reproduction with no dependency on the `Locale`/`bynk.locale` design above; no existing fixture had ever combined `consumes bynk` with a test suite, or stubbed a refined-type-returning capability, so none of these had ever fired):

- **`makeTestDeps()` never wired an adapter-flattened capability at all — the critical one, caught in review.** `emit_test_deps` (`bynk-emit/src/project/tests_emit.rs`) built the test harness's `deps` object from `UnitTable.capabilities`, which holds only capabilities a unit declares *locally* — never ones flattened in via `consumes U { Cap }`. A stub class was generated but never instantiated, and an un-stubbed call was simply `undefined`, so any test calling a `Locale`-style capability threw `Cannot read properties of undefined` at runtime instead of exercising the stub — invisible to golden-diffing and `tsc_verify` alike (the test body casts the whole target module to `any`, defeating structural checking of `deps`). Fixed by threading `unit_flattened` (the same capability-name → owning-unit map production `compose()` wiring already uses, via `cross_context_caps_used`) into `emit_test_deps`, and folding its entries in alongside locally-declared capabilities. A new `bynkc/tests/adapter_flattened_capability_stub_behaviour.rs` drives the real `bynkc test` CLI against fixtures 804/807 end to end (compiles, runs under `tsc`+`node`, asserts the actual pass count) so a regression of this class fails CI, not just a manual re-run.
- **Test-scaffold cross-context "surface" wrongly generated for an adapter-consumed capability.** `emit_test_deps`/`emit_test_scope_setup` treated every `consumes` target uniformly as a peer context needing a `deps.surface.<name>` entry (`ReturnType<typeof <ns>.makeSurface>`) — but an **adapter** target (e.g. `consumes bynk { Locale }`) has no `makeSurface` at all; its capabilities are already flattened onto `deps` directly (the bug above). Fixed by skipping adapter-kind targets (via `UnitTable.kind`) in both call sites.
- **`admit_refined_literal` didn't see through one `Effect[_]` layer.** A `stub Cap.op() returns <literal>` rhs is checked as a bare tail value against the op's full `Effect[T]` return type; when `T` is a refined type, `admit_refined_literal` (`bynk-check/src/checker/refinements.rs`) requires `expected` to be the refined type *directly*, so it never saw past `Effect[T]` and the literal fell back to its unrefined base type, later rejected by the return-type compatibility check. Fixed by unwrapping one `Effect[_]` layer before the pattern match.
- **A synthetic single-expression block's tail-type entry was clobbered by its own auto-lift.** `type_of_block` (`bynk-check/src/checker.rs`) records the auto-lifted (`Effect`-wrapped) tail type at `block.span` — harmless for a real `{ ... }` block, whose span always strictly contains its tail's, but a `stub` rhs's synthetic `value_block` reuses the tail's own span verbatim, so the lifted entry overwrote the literal's own more specific refined-type entry the emitter's brand-cast lookup depends on. Fixed by skipping that insert when `block.span == block.tail.span`.
- **Stub method return-type qualification assumed every `uses`-reachable type is exported under the target's own namespace.** `emit_stub_class` qualified `LocaleTag` as `<target_ns>.LocaleTag`, but `emit_context_rebrands` only re-exports a `uses`-sourced type under that namespace when the target's *own* lowered body references it by name, which a stub class implementing an adapter-flattened capability generally doesn't. Fixed by tracking each in-scope type name's *owning* namespace (target-local vs. the specific commons it came from) and qualifying against that, via a new `ts_type_ref_qualified_multi` (`bynk-emit/src/emitter.rs`) alongside the existing single-namespace `ts_type_ref_qualified`.

Relatedly (a refactor prompted by review, not a bug): `LowerCtx` grew a dedicated `test_scaffold: bool` rather than overloading `assert_loc.is_some()` as the test-scaffold discriminator — `assert_loc`'s real job is `assert`/`expect` diagnostic-location plumbing, an unrelated payload `lower_block_to_async_body` (used by the `stub`/`where`/`requires` lowering above, never real production provider bodies) has no location to give.

**Consequences.** `Locale`/`LocaleTag`/`Message`/`MessageArg`/`render` compile, are usable from an application context on all three platforms (cloudflare/node/browser), each returning a fixed `"en"` (decision A), and — now genuinely verified by running the compiled output — `stub Locale.current() returns "fr"` and the un-stubbed render path both actually pass at runtime, with zero new test-provider surface. All five compiler fixes are general (not Locale-specific) and apply to any future capability/stub combination that reaches a refined type through an `Effect[...]`-wrapped return, a `uses`-sourced commons, or is declared inside an adapter and flattened into a consuming context.
