# Decision records

The design decisions that shaped Bynk's incremental evolution, one record per
language-defining call: `Status / Context / Decision / Consequences`, each
citing the [normative spec](../../site/src/content/docs/book/spec/index.md) section that now
embodies it. Records were harvested from the retired per-increment instalment
documents (see the spec's Appendix B); from v0.19 on, each increment adds its
records here as part of the increment itself.

A record is **immutable once accepted** — a reversal is a new record that
supersedes the old one, the way 0016 supersedes the interim portability tiers.

The table below is **complete by construction**: a CI drift guard (`bynkc`'s
`decisions_index` test, run by the `drift` job) fails if an ADR file has no row
or a row links to no file. Summaries and statuses are curated by hand; the
*presence* of every record is enforced.

| [0195](0195-secrets-at-deploy.md) | **Secrets at deploy: values never at rest, and a derived set that admits what it cannot know** (v0.172, closes #602, deploy track slice 3) — the last of the four v1 resource kinds, and the one that is a **value, not an id**: a KV id is minted by Cloudflare and safe to commit, a queue name comes from the source, but a secret value comes from the user and must reach Cloudflare unwritten. The slice was proposed on a premise that did not survive grounding, and the correction is most of the record: #602 (and the track doc §4.4 before it) claimed the **names** were derivable from the closure walk, hence "Checker: None / Emitter: None". Both false. `bynk.Secrets` is `fn get(name: String)` — an ordinary expression in an ordinary argument position, checked for arity and type alone, so `Secrets.get(someVar)` type-checks and `consumes bynk { Secrets }` says only *that* a context reads secrets, never **which**; nothing collects the names and they reach no emitted artifact. And the draft omitted the class that *is* derivable and matters most: an `actor`'s auth secret (`Bearer`/`Signature`) is a literal fixed at parse time (`SchemeArgValue`), required at compile time (`bynk.actor.bearer_missing_secret`), already resolved into the seams the emitter lowers — and **fail-closed silent**, answering 401 on every request rather than failing the deploy. **A:** no secret value **or name** is ever written to Bynk state — stdin not argv (and not by preference: `wrangler secret put` has *no* value option, so stdin is the only interface), and the ledger records **nothing**, not even presence, since presence is live (D) and a recorded answer could only be stale — a field load-bearing for nothing, which [0194](0194-deploy-queues-and-delegated-do-migrations.md) already called a claim waiting to be believed; `WorkerRecord` stays one field, declining the invitation its own comment extended. Pinned by sentinel against plan and ledger, not described. **B:** the derived set is a **floor, not a census** — exactly the actor-auth secrets (`Oidc` names none, its trust root being the provider's JWKS; a sum's `None` member verifies nothing — both skipped, not defaulted), and **no `Secrets.get(<StrLit>)` scan**: every call site in the repo happens to use a literal, which is precisely what would make such a scan look reliable while being incomplete by construction — a list that is usually right is worse than none on a fail-closed path, because it gets trusted. Each plan line is marked `declared`/`supplied` so the compiler's word is legible from the user's. **C:** names and values are **separate inputs** — names are declared ∪ `--secrets-file` keys ∪ `--secret NAME`; values are file → environment → prompt, per *already-known* name. The environment is never a name source: sweeping `env` into Cloudflare would exfiltrate the user's shell (#602's original "file > env > prompt" precedence was incoherent for exactly this reason). A missing required value is a hard error naming it, never a blank; a malformed file line names its number, never a skip. A supplied name reaches **every** context in the run — nothing says which read it — and the plan lists it per context so the spread is visible. **D:** set-if-absent, `--force` to overwrite, presence asked **live** (`wrangler secret list`) and read structurally; a failed query means "assume none and try" (the `queue_exists` posture) — with the qualification that posture needs once a **value** is required to try with: where presence is unknown *and* the Worker has been live before, an unresolvable secret is **left alone and reported**, not failed, or the commonest CI shape (redeploy, no `--secrets-file`, no TTY, secrets already set) would turn a *read* failure in an advisory check into a blocked deploy misdiagnosed as "no value supplied". A first deploy gets no benefit of the doubt; a known presence is authoritative. The plan **cannot forecast the skip** — presence needs auth, and the plan is derived before authenticating, which is what keeps `--dry-run` offline; that property is worth more than a forecast, so the plan says `set`/`overwrite` and the *run* reports the skip. **E:** the declared names travel in a compiler-emitted `bynk-secrets.json` beside each `wrangler.toml`, **not** an API: `compile_once` has two paths, and under a `bynkc` override the compiler is a child process handing back an exit status — an API answer would work in-process and silently derive an **empty set** under the override, indistinguishable from "declares nothing" under a floor contract. Extends [0193](0193-multi-context-deploy-ordering.md) D3 (read the emitted artifacts) to the one resource with no stanza; derived from the same seams over the same handler enumeration the entry lowers, so it cannot describe a different Worker. Cost: "Emitter: None" was false — a new artifact, nine fixtures of golden churn. **F:** the **put** straddles the push, ledger-conditional — a Worker pushed before has its secrets set first (running code never sees a request without them); a **first** deploy pushes first and sets after. Only the put moves: the presence read, the decision and the value resolution all happen before the push on *both* paths, or the first-deploy path would discover a missing value only after making a live Worker. The straddle exists because `wrangler secret put` against an absent Worker does not fail: it calls `createDraftWorker`, uploading a **stub** (`export default { fetch() {} }`), with its confirm falling back to *yes* in CI (a stub the plan never mentioned) and prompting interactively — where a decline exits **0 having set nothing**, the exact silent-blank failure C exists to prevent. The window is fail-closed (an unset auth secret 401s) on a Worker that did not exist a moment earlier. Deferred and named so it is not re-derived: a `Secrets.get`-literal static rule would make the floor a census — precedent exists (`cors`/`@cache`/`@limit`) and all six call sites comply — but it is a language surface change with a real expressiveness cost and wants its own proposal. Cites ADR 0194, 0193, 0180, 0179, 0018, 0096, 0083. | Accepted (v0.172) |
| [0194](0194-deploy-queues-and-delegated-do-migrations.md) | **The deploy ledger owns the ids it mints, and defers for state another tool owns — queues + DO migrations** (v0.171, closes #600, deploy track slice 1) — slice 0 provisioned only KV, so a context with an `agent` or a queue-protocol service still owed the manual ritual; a `[[queues.consumers]]` binding whose queue does not exist fails the upload. The two kinds sit at opposite ends of [0179](0179-deploy-provisioning-state.md)'s "the format admits the other resource kinds", which is why one ADR lands both. **A:** DO-migration state is delegated to wrangler **entirely** — the ledger records **nothing**, not even 0179's foreshadowed advisory last-applied tag: nothing would read it (reconciliation never consults it; the plan reads the tag from the emitted config), and a field load-bearing for nothing is a claim waiting to be believed. `wrangler deploy` already applies the block slice 0 pushes, so this is a step the driver *declines to duplicate*, not one it adds. Stated cost: `deploy` cannot detect migration drift in v1 — better than a rival ledger that reports drift that isn't there. The plan shows the tag flagged advisory (`migration v1 (advisory — wrangler deploy applies it)`; JSON `{"tag","applied_by"}` names an owner rather than a constant-true `advisory` flag). **B:** queues reconcile **by their user-given name**, create-if-absent, no stored id — `from queue("n")` names it and Cloudflare addresses it so; the ledger gains `environments.<env>.queues`, a name set keyed **environment-wide** (a queue is an account resource — two contexts consuming `"jobs"` mean one queue). Its only reader is the plan's create/reuse wording; the provision step **never consults it**, reconciling each declared queue against **the account** every deploy (`wrangler queues info <name>`, create only where absent) — skipping on the ledger's word would leave an out-of-band deletion unrepaired and the push failing against an empty binding. That is what lets the set be honestly called advisory. The step is not optional politeness: `wrangler deploy` **will not** create a queue its config binds (`ensureQueuesExist` looks the names up and throws "To create it, run: wrangler queues create"), so this is what makes a queue-consuming context deployable at all. Cost: one call per queue per *deploy*, not per consuming context — a shared queue is one queue (and a legal project, the emitter's duplicate-consumer check being context-scoped), so the run tracks names it has attempted, per-run and unpersisted so a fresh deploy still reconciles every queue; a first deploy pays a second call for the create. **The seam is an exit code, not a message.** An earlier draft dismissed a live pre-check as "the same race one call later" and made the create's "already exists" text load-bearing for every re-deploy; that inference was wrong — the race is a *concurrent deploy*, while the common case is a queue that is simply there, which a lookup answers without reading prose. Cloudflare's own deploy path reconciles queues exactly this way (`getQueue`). The message match survives, narrowed to the race-loser's path and reading **both** streams; it stays the slice's one claim about another tool's prose that no test can pin (the wording is Cloudflare's API text, rendered verbatim as `{message} [code: {code}]`, with no queue-specific handling in wrangler and no duplicate-queue code in Cloudflare's published Queues errors, which cover the data plane only) — so the response to an untestable assertion is to make it carry less: being wrong now costs a spurious failure on a rare race a re-run fixes, not every re-deploy. **C:** queue creation runs in `provision` (the upload fails without it), migration application stays inside `push` (the upload *is* what applies it) — no new phase; slice 2's per-context loop ([0193](0193-multi-context-deploy-ordering.md)) absorbs both without touching the ordering, as it anticipated. **D:** `queues` is additive over `version = 1` — a slice-0 or slice-2 ledger loads unchanged, and an empty set is not serialised. **E:** the CI provisioning refusal stays **KV's alone** — a minted id a CI job cannot commit is an orphan; a queue's name comes from the source, so CI creating one loses nothing. The ledger's boundary is now fixed by decision: it owns the ids it mints (KV), records names as a planning aid it never trusts (queues), and records nothing where another tool owns the state (DO). Cites ADR 0193, 0180, 0179, 0017, 0083. | Accepted (v0.171) |
| [0193](0193-multi-context-deploy-ordering.md) | **Multi-context deploy — Service Bindings resolve at *upload*, so the order is a correctness barrier** (v0.170, closes #601, deploy track slice 2) — slice 0 shipped `deploy` for exactly one context, so the flagship several-context architecture could not be shipped at all; [0192](0192-multi-context-local-dev.md) had just made the same topology runnable locally. #601 made one question gating: is Service-Binding resolution by-name-at-request-time (order soft) or at upload (order hard)? **A:** the working assumption was **wrong** — Cloudflare's docs state a Worker whose bound target does not yet exist **fails to deploy**, so the topo-sort is a correctness barrier, not a nicety; recorded per #601's requirement, on documentary evidence (the finding only *hardens* the design, so a wrong call costs an unnecessary guarantee, not a broken deploy). **B:** the two-pass deploy the hard world implies is **not built and should not be** — `bynkc` already rejects a `consumes` cycle (`bynk.context.consumes_cycle`) before emit and `deploy` compiles first, so the checker's acyclicity invariant *is* the precondition Cloudflare's upload rule demands; #601's [DECISION C] **dissolves rather than resolves** (the sort still reports a cycle as defence in depth against a hand-edited tree). **C:** the graph is read from the emitted `[[services]]`, not the checker's `consumes` map — it is the file wrangler uploads and Cloudflare validates; a binding outside the project's build is left to Cloudflare, not invented into the order. **D:** resumable, never transactional, never skips — incremental ledger writes ([0180](0180-deploy-orchestration-idempotency.md)), no rollback, stop at first failure; a re-run **re-pushes** rather than skipping (skipping on a ledger flag would silently fail to ship changed code), and the plan says `redeploy` so the word matches the act. **E:** the ledger gains `environments.<env>.workers.<name>` (additive; a slice-0 ledger still reads) because KV presence cannot answer "does this Worker exist?"; `--context` deploys one and **names + refuses** an absent dependency before any Cloudflare contact. Retires `SelectError::Ambiguous` and the singular `select_context` — with `dev` ([0192](0192-multi-context-local-dev.md)) and `deploy` both acting on every context, nothing is ambiguous about a project having several. Cites ADR 0192, 0180, 0179, 0017, 0084. | Accepted (v0.170) |
| [0192](0192-multi-context-local-dev.md) | **Multi-context local dev — every context served, Service Bindings wired** (v0.167, closes #552, resolves design-review finding Platform #5) — the emitter has generated correct `[[services]]` wiring since v0.57, but nothing ever ran two Workers at once: ADR 0096 D3 served one context and failed a multi-context project as ambiguous, so the flagship architecture was unrunnable locally and a cross-context call had **never been exercised at runtime** (golden-file coverage only). **Supersedes [ADR 0096](0096-bynk-dev.md) D3** (D1/D2/D4/D5 stand). **A:** one `wrangler dev` **per context**, wired by wrangler's **dev registry** — not the first-party `workerd` server the finding recommended and 0096 anticipated; verified on the two-context fixture (`local [connected]`, a live `placeOrder` → `commerce.payment` call, `Declined` mapping through the anti-corruption layer), so the emitter is **untouched** and workerd becomes an optimisation, not the price. | [0191](0191-a-renderer-arm-for-every-resolved-index-kind.md) | **A renderer arm for every resolved index kind; a bare key names a free function** (v0.166, closes [#616](https://github.com/accuser/bynk/issues/616)) — the follow-on ADR 0190 filed against itself (`Actor` "the same gap, found while measuring D1"; `Method`/`CapabilityOp` "an obvious follow-on, not done here"). **Amends [ADR 0190](0190-hover-resolves-references-not-just-declarations.md) D1** — not its rule, which stands, but the *measurement* offered for deferring it. 0190 read the fall-through from the ladder's tail (`qualified_callee_at` bails on a lowercase receiver ⇒ `None`) and concluded the open kinds "already hover as nothing", so arms would buy no user-visible change while risking `CapabilityOp`. Either half alone is defeatable: the sweep goes vacuous for a kind the fixture never declares, the match only proves a name was written down. The pin lives in the tests, not the type system — rung 1 still guards on `Some`, so a missing arm still falls through at *runtime*; it just cannot reach `main`. Where the index does not resolve (builtin statics, refined `of`/`unsafe`) rung 7 answers exactly as before; `bynk-fmt` exports `escape_string` so the actor arm re-escapes a scheme config the way `format_actor` does (the stored value is unescaped — the parser resolves escapes at lex time — so a `"` rendered as invalid Bynk inside the fence), pinned against `format_source`'s own output; D2's ceiling is the unanalysed buffer (a method hovers as nothing until the round lands); `describe_item` and `find_declaration_span` now deliberately disagree on what a bare name means. Cites ADR 0190, 0069, 0091, 0063. | Accepted (v0.166) |
| [0190](0190-hover-resolves-references-not-just-declarations.md) | **Hover resolves *references*, not just declarations; a structural resolution outranks a name match** (v0.165, closes [#611](https://github.com/accuser/bynk/issues/611)) — inside an `agent` handler body hover described the four lines declaring the agent's state but missed the uses. Three gaps, three different causes. **D1:** a resolved index hit is rendered or nothing is — it never falls through to a name match. The checker already recorded record-construction labels as `Field` refs keyed `"Type.field"` (ADR 0069), so the index resolved `title:` correctly, but `describe_item` had no `Field` arm and returned `None`; hover then fell through to the locals path, which matches by *name in scope*, and rendered the same-named handler param — a confidently **wrong** hover, not a missing one. Where the renderer is the missing piece, the fix belongs in the renderer. **D2:** a reference to agent state renders exactly what its declaration renders (extends ADR 0161 D2 from the keyword/field-name pair out to the uses). **D3:** the reference scope is where state is *referenceable* (handler bodies, invariant/transition predicates), not the agent's span — matching is by name, so over-reach is the live risk, and the declaration region is excluded: `@indexed(by: id)` names a field of the *stored value*, which must not masquerade as a same-named `key id` (ADR 0161's test pinned this). **D4:** name-based paths defer to the checker's own dispatch rule — a store op binds a bare-ident receiver *not in the value scope*, so a shadowing local wins, in the editor as in the compiler. **D5:** storage operations get an enumerable `store_ops` registry beside `kernel_methods` (ADR 0063) — the op names live in `match` arms, authoritative for typing and invisible to tooling — pinned by a drift test that drives every entry through the **real checker** so the table cannot list a phantom. **D6:** the fixtures sit at *reference* offsets in the file the issue reproduces in; the gaps existed because every prior hover test used a declaration offset. **D7:** gap B being a *fall-through* means the **rung order** is the behaviour, and behaviour a test replicates is behaviour no test pins — so the ladder moves out of `Backend::hover` into a pure `hover::hover_content` the handler and its tests share (the handler becomes transport; the snapshot and live buffer stay distinct). D1 is stated as the direction of travel, not enforced: measurement showed `Method`/`Actor` already hover as nothing via the fall-through, so enforcing it would buy nothing while risking `CapabilityOp` — the honest fix is a renderer arm per kind. Store/key fields remain un-indexed (so `references`/`rename` over them stay unsupported — indexing would retire D3 and D4); `Actor` is filed as the same gap found while measuring; the drift test pins op names one way only (phantoms, not omissions) and not signature strings. Cites ADR 0161, 0069, 0063, 0110, 0113, 0121, 0125. | Accepted (v0.165) |
