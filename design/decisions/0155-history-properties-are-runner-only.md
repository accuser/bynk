# 0155 — history/protocol properties are test-runner-only, over a generated call-history driven through the real handlers: no commit-time temporal enforcement, and the temporal reach is bounded `List` quantification, not a new logic

- **Status:** Accepted (v0.119; 2026-07-03)
- **Provenance:** the v0.119 increment — the testing track's eighth slice and the top rung of the subject ladder. It is the load-bearing record for how a Bynk test judges an agent's *behaviour over time*, and it closes the invariant↔test duality the track has built rung by rung.
- **Realises:** the spine's history rung — `value → domain → call → snapshot → step → history`. A `for all run: History[Agent]` binds `run` to a generated, driven history of the agent: the runner generates a bounded random sequence of the agent's handler calls, drives them through the **real** handlers from the agent's initial state, and binds the observed history — an ordered `List[Step]`, each step carrying its call, its outcome, and the committed old→new state pair — for the predicate to judge. It also fixes the **sound route for agent behavioural generation**: an agent's behaviour is generated by driving handler sequences and observing the states they *reach*, never by fabricating a valid-but-unreachable state.
- **Relates:** the one predicate surface (ADR 0144 — a history predicate is the ordinary predicate/`List` surface, so there is no LTL vocabulary and no history-matcher library); generation is valid inhabitants (ADR 0149 — a step's generated arguments are drawn from each handler parameter's refinement domain, reusing the value generator); the invariant subject widens to the step (ADR 0151 — a history *is* the ordered sequence of those steps, now the generated subject of a runner-only property); observation records at the capability seam (ADR 0152 — a driven handler's capability calls are still recordable, and its seams are still `provides`-stubbable, so observation and test doubles compose inside a driven run); the tier dial (ADR 0153 — a `property` is `case`-only for tiers, so a history property carries **no** `as`: it is in-process and generative by construction, the flake-free tier).

## Context

Slices 4 and 5 put behavioural checks at the commit boundary: `invariant` judges a reached snapshot and `transition` a reached step, both enforced when a handler commits. What was missing was a way to judge a whole *run* — a claim spanning several handler calls, like "no spend is ever accepted without a prior accepted top-up". Such cross-step claims are the territory of temporal logic (LTL) and model checking, and the open question the track carried (DECISION — open H) was how far to let that reach go without dragging an unbounded temporal-logic surface, and its cost and flake, into the language.

Two sub-questions had to be answered together. First, *where* is a cross-step claim evaluated — at the commit boundary (an always-on runtime enforcement) or only in the test runner (a sample over generated runs)? Second, *how* is an agent's behaviour generated at all — by fabricating agent states that satisfy the invariants, or by driving real handler sequences and observing the states they reach? A fabricated state can satisfy every invariant yet be unreachable by any real sequence of handlers (DECISION P), so a property over fabricated states would report counterexamples that can never occur.

## Decisions

**D1 — history properties are test-runner-only; there is no commit-time temporal enforcement (DECISION H, the gate).** Cross-step "eventually / always" claims are evaluated **only** by the runner, over a **bounded, generated** call-history — never at the commit boundary. Commit-time stays where slices 4–5 put it: `invariant` over a reached snapshot and `transition` over a reached step. This keeps the runtime cheap, keeps the flake-free guarantee scoped to the lower tiers, and — the load-bearing reason — keeps a *bounded* history out of the LTL / model-checking surface an unbounded commit-time temporal logic would drag in. The whole slice sits behind this decision.

**D2 — the temporal reach is `List` quantification, not new operators (DECISION A).** "Always P" is `run.all((s) => P)`; "P before Q" is a quantified prefix check via `run.upTo(...)`; "eventually P" is `run.any((s) => P)`. There are **no** `always` / `eventually` / `before` keywords and **no** history-matcher library — the reach is exactly what `List` plus the predicate surface already give over a finite run (ADR 0144). Consequence, stated honestly: **unbounded** "eventually" (liveness over an infinite run) is *not expressible*, by design — a bounded run can only witness bounded reach. This is the deliberate ceiling D1 buys; the docs say so plainly so no reader mistakes it for model checking. An always-on-every-path universal, if it is ever wanted, belongs to a policy/`system`-tier guarantee, not to a history property (a runner sample, never a proof).

**D3 — generation is by driving, not fabrication (DECISION P, made concrete).** The runner generates a bounded random sequence of the agent's `on call` handlers — each handler chosen uniformly, each argument drawn from its parameter's refinement domain (the value generator, so the same seed and shrinker apply) — and drives the sequence against a **fresh** agent instance from its initial state, invoking the **real** handlers and their real invariants. Every state in `run` is therefore one a handler actually reached: reachability by construction. An agent is never `for all`-generated as a value (`bynk.val.agent_not_generable` points here).

**D4 — the step is the typed unit a history sequences.** Each `Step` record carries `.call` (a synthesised sum over the agent's handlers, each variant carrying the handler's generated arguments, matched with `is` / `match`), `.accepted` (whether the handler committed a new state — an invariant/`transition` refusal throws and leaves the state uncommitted), and `.old` / `.new` (the committed old→new pair, the same `old`/`new` a `transition` sees). All fields are pure values, so a history predicate is a pure `Bool` over them, held to the same predicate-purity discipline as every other rung, and `is` / `implies` / field access read over them unchanged. This resolves the track's open question about the typed shape of `old` / `new` / a bound history.

**D5 — `History[Agent]` is agent-only and generative-only (DECISION B).** Only an agent has handlers to sequence and reachable states to observe; a value or a service does not (`bynk.history.not_an_agent`). `History[...]` is legal **only** in `for all` position inside a `property` — it *is* a generator; anywhere a type is otherwise declared it is `bynk.history.outside_property`, and it never resolves as a value type. The agent must be **drivable**: every handler parameter must be refinement-generable, else `bynk.history.not_generable`. Because a `property` is `case`-only for tiers (ADR 0153), a history property carries no `as` and needs no new tier rule — it runs in-process against the real handlers on the generative, flake-free tier.

**D6 — bounding and shrinking is seeded, length-bounded, and shrinks the sequence, re-driving after each reduction (DECISION E).** The run reuses the value-property infrastructure: a fixed root seed by default, a bounded run length and case count, boundary-biased argument sampling, and a printed repro command. On failure the shrinker delta-debugs the step list (drop steps, **re-driving** after each reduction so the reduced sequence stays *reachable*) and then shrinks the surviving arguments with the value shrinker. The counterexample it prints is therefore guaranteed reproducible under `--seed`.

**D7 — a history property that merely restates a snapshot/step invariant is flagged (DECISION D).** A history predicate that re-checks a guarantee a declared `invariant` / `transition` already enforces on every reached state (the driver only commits admissible states) is `bynk.history.restates_invariant`. The check is **conservative and syntactic** — an α-equivalent predicate over `.new` (or `.old`/`.new`) versus a declared invariant/transition, mirroring `bynk.property.restates_refinement`. Never full semantic equivalence: a false rejection would block a valid test, so near-duplicates slip through by design.

**D8 — single-agent histories only; multi-agent / cross-context protocols are a named follow-on (DECISION F).** A `History[Agent]` sequences **one** agent's handlers. A protocol *across* agents, or across the `system`-tier wire (an order in context A eventually settles a ledger in context B), is the LTL-flavoured surface D1 warns about; it is out of scope for v1 and named here so the boundary is deliberate. The history rung earns its place in v1 as *single-agent* behavioural generation; the multi-agent-protocol tail ships later (or never) without blocking the rest of the language.

## Consequences

- The subject ladder is whole: `value → domain → call → snapshot → step → history`, every rung realised. `invariant` / `transition` are the commit-boundary half of the duality; a history property is the runner-adversarial half over the same reached states.
- No new predicate or temporal vocabulary enters the language. A history is a `List[Step]` asserted with the surface a reader already knows, plus one prefix accessor `run.upTo(step)` (the history strictly before `step`) — history-scoped for now, to collapse into a general `List` slice if slicing ever lands independently.
- The emitted driver is a test-only export on the agent module (gated on the agent being a history target), so a production build and every non-targeted agent are byte-for-byte unchanged; the whole surface is stripped from deploy builds with the rest of the test scaffolding.
- The reproducibility guarantee holds because a subject has exactly three honest sources (a pinned value, a generated value, an injected capability) — the generative tier provides every capability seam deterministically (real or `provides`-stubbed), so the whole run is a function of the seed.
- **Re-openable:** multi-agent / cross-context protocol properties (the named follow-on above); an unbounded liveness or always-on-every-path universal (which would live in a policy/`system`-tier guarantee, not a history property); and the exact shrink-aggressiveness / re-drive budget (an implementation-tuning knob, not a language decision).
