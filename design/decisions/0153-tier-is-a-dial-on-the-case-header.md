# 0153 — tier is an `as <tier>` clause on the `case`/`suite` header, not a kind of test: one body promoted across `unit | integration | system`, participants inferred

- **Status:** Accepted (v0.118; 2026-07-03)
- **Provenance:** the v0.118 increment — the testing track's seventh slice, the one that unifies the test *tiers*. It is the load-bearing record for how a Bynk test declares *how much of the real world is present when it runs*, and for retiring the standalone `suite integration "…" { wires … }` construct.
- **Realises:** the testing track's "tier is a dial, not a kind" pillar. The three tiers borrow the testing pyramid verbatim — `unit` (collaborators doubled), `integration` (real collaborators within one context, no wire), `system` (contexts wired across the real serialise → JSON → deserialise boundary) — so a newcomer arrives already holding the model and the pyramid's "many unit, some integration, few system" proportion advice rides along for free.
- **Relates:** the one predicate surface (ADR 0144 — a promoted case's `expect`s are unchanged; a tier changes substitution, not assertion); the test-time provider override `provides` (ADR 0154 — `as <tier>` sets the default provision of every seam, a `provides` clause overrides one); structural test-ness and the build strip (ADR 0147 — a `suite` is a test-only declaration, so the whole tier machinery is test-build-only for free); the auto-recorded seam (ADR 0152 — observation records at whatever seam the tier establishes, unchanged).

## Context

Before this slice, Bynk had two test artefacts: a unit `suite` (collaborators replaced by a `mocks` block) and a separate `suite integration "name" { wires C1, C2 }` construct that stood participants up as Workers and exercised a flow across the real boundary. Two artefacts, two grammars, a `wires` list to maintain, and no *middle* — no way to run a unit's real collaborators in-process without crossing the wire.

The testing pyramid says these are not three kinds of test but one body run with more or less of the real world behind it. The question was how to *spell* that. A tier is metadata about a case, not an executable statement — so it belongs in the header, not the body. But the natural spelling, `as`, is already the `consumes … as Alias` keyword, so it had to be shown unambiguous first (settled: `as` is consumed only in `consumes`-declaration position; a case/suite header is a distinct production). Two further questions: do the higher tiers still need an explicit participant list, and where do the old integration constraints (`≥ 2 contexts, no mocks`) now attach?

## Decisions

**D1 — tier is an `as <tier>` clause on the `case` header; `unit` is the default and elided.** A `case` runs at one of `unit | integration | system`, written `case "…" as integration { … }`. `unit` is the default, so it is never written. The clause sits in the header where declaration metadata belongs, leaves a clean default, and — the load-bearing reason — frames a tier as **one body promoted, not a distinct kind of test**. `as integration` reads as "run this *as* an integration test", not "this is a different artefact". Rejected, recorded because they were close: a body-statement keyword (`realism`/`scope`/`tier <level>` — reads executable but isn't, and `scope` re-collides with lexical scope), and a tier-leading case keyword (`integration "…"` — re-instates "three kinds" in the grammar and reserves the common nouns `unit`/`system`).

**D2 — `as` also sits on the `suite` header, as an inherited default; the case wins.** `suite checkout as integration { … }` sets a default every `case` member inherits and may override (`case "an edge" as unit`). This is the "set once" ergonomics of the old `suite integration` block without a separate construct. A case's effective tier is `case.tier ?? suite.tier ?? unit`.

**D3 — tiers are a `case`-only affordance; a `property` ignores a suite tier.** A `property` *generates*, and does not promote (promoting it would multiply generation by real-collaborator cost and re-admit the ambient nondeterminism the tier removes). So a suite-level `as` binds its `case` members only; a `property` carries no tier. A tier attached to a property is `bynk.tier.property_has_tier`. To check a generated input end-to-end, promote *that witness* as a concrete `case as integration`.

**D4 — the tier names are contextual, not keywords.** `unit` / `integration` / `system` are parsed only in the `as`-clause position after a case/suite header; everywhere else they stay ordinary identifiers, so nothing that spells `system` today breaks. This mirrors the slice-5 observation words and `for`/`all`. It also lets the old hard keywords `integration`/`wires` be retired outright.

**D5 — participants are inferred from the `consumes` graph; `wires` is retired (DECISION K).** `as integration` / `as system` derive their real/wired participant set from the unit under test's transitive `consumes` closure — the compiler already builds that graph, and an explicit list can only drift from it. The standalone `suite integration "…" { wires … }` construct and the `wires` clause are removed. A `system` test whose inferred set is fewer than two contexts (the target and at least one consumed context) is `bynk.tier.system_needs_wire` (replacing the old `bynk.integration.too_few_participants`). Re-openable only against a concrete case where the participant set is genuinely ambiguous; none is known.

**D6 — constraints split by tier (DECISION L).** The old integration rules were a property of the one integration construct. With per-case `as` they attach to the tier they describe: `system` is the cross-context, wired tier (it stands up ≥ 2 contexts across the real serialisation edge); `integration` is real collaborators *within one context, no wire* — the missing middle — and carries **no** ≥ 2-context rule, which deliberately *relaxes* today's integration constraints. `provides` is legal at every tier (it overrides one seam), so the old "no mocks in an integration test" prohibition disappears with `mocks` itself.

**D7 — the agent-state lifecycle is fixed across tiers.** The unit under test is always a real in-memory instance, keyed normally, fresh per case; only the realness of its *collaborators* and whether sends cross a serialisation boundary change with the tier. Snapshot and step invariants are checked at the commit boundary and therefore fire at **every** tier for free — which is why promoting a `unit` case can surface a collaborator's invariant a stub was hiding, with no new test code.

**D8 — at `unit`, an un-overridden seam keeps its real provider; full auto-stubbing is a named follow-on.** The idealised pyramid has `unit` *double every collaborator* by default, with a return synthesised from each operation's result type. That synthesis is a second generator (a "one canonical value" cousin of `Val[T]`) with its own failure modes (which `Result` outcome is the default? an opaque return has no derivable value), and it must surface the synthesised return in the trace so a green-because-convenient test is never silent — a materially larger build. This slice ships the tractable, migratable semantics: **`unit` and `integration` emit in-process with real providers; a `provides` clause overrides a specific seam** (ADR 0154). The distinction between them is the *default provision discipline* an author follows, not a compiler-enforced auto-stub. Full unit auto-stubbing with stub-return fidelity is recorded here as a **named follow-on** ("unit auto-stubbing"), not part of v0.118. `system` is the one tier that differs mechanically: it wires participants across the real boundary.

## Consequences

- One test grammar spans all tiers: `suite`/`case` with an optional `as <tier>`. The standalone `suite integration` construct, the `wires` clause, and the `bynk.integration.*` diagnostic family are gone.
- Promotion is a header edit — the body is byte-for-byte identical across tiers, so "did I stub this faithfully?" becomes a checkable question by changing one word.
- A `system` test needs no participant list: the wired set is the target's transitive `consumes`. This cannot drift from the real dependency graph.
- The `integration` tier fills the missing middle (real collaborators, one process, no wire) and relaxes the old integration constraints by design.
- **Re-openable:** full `unit` auto-stubbing with type-derived default returns and trace-visible stub fidelity (the named follow-on above); a genuine per-case-in-a-mixed-suite `system` wiring (this slice wires a suite whose effective tier is `system`); and observing/asserting across the `system` wire.
